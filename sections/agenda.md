# 研究议程

> 文献综述核心交付物 · 5 条证据驱动的可执行研究议程

**总注**：以下 5 条议程不是凭空提出——每条均反向引用 §9 双轴矩阵/就绪度表中的具体空白。优先级排序依据：① 空白严重程度 ② 技术可行性 ③ 预期影响范围。

---

## 议程 1（最高优先）：带健全性保证的 SCA 可达性验证

**反向引用**：§9.3 矩阵空白 (A,4) — 调用图断裂补偿 + 零漏报保证 完全空白；§9.1 观察 3 — D1●+D3● 从未同篇。

### 问题定义

现有 SCA 工具只做版本比对或函数定位，不判断"已知 CVE sink 在应用中是否调用图可达"。SemTaint（D1●）证明了 LLM 可以恢复静态不可解析的调用边，但它面向 JavaScript taint 检测，不处理 SCA 场景、不保证健全性。保守分析（D3●）证明了零漏报约束是可行的，但它面向 SAST 报告过滤而非可达性判定。

**核心空白**：没有人将"调用图可达性判定"和"零漏报约束"组合到一起——而这恰恰是 SCA 误报消减从"版本维度"推进到"可达性维度"所需的关键技术。

### 技术路线

```
静态调用图（CodeQL/Joern）
    │
    ├─ 静态可解析边 ──→ 确定性答案（保守：不处理 = 告警）
    │
    └─ 动态分发断裂边 ──→ LLM 选择性补判
          │                  │
          │                  ├─ LLM 高置信 → 补边
          │                  └─ LLM 低置信 → fail-safe 保守策略（保留告警）
          │
          └─ 最终判定：若 sink 调用可达路径完整 → 真漏洞；否则 → 不可达（可安全消减）
```

核心设计原则：
1. **静态调用图做主判**——不依赖 LLM 做完整路径分析
2. **LLM 仅在动态分发断裂边做选择性补判**——反射/Spring DI/SPI/反序列化
3. **保守 fail-safe**——LLM 判定不稳定时，宁可保留告警也不误删
4. **仅剪"不可达"告警**——不改动静态分析的 vulnerability detection 部分

### 关键挑战

- **Java 框架动态分发的系统性断裂**：反射、Spring DI、SPI（ServiceLoader）、反序列化（ObjectInputStream）、动态代理——这些让完整编译下的静态调用图依然断裂
- **LLM 对调用链语义判定的可靠性**：需要研究上下文表示（CPG 切片 vs SARIF 轨迹 vs 自然语言摘要）对判定准确率的影响
- **保守策略的"安全 vs 消减率"权衡**：过于保守 → 消减率低（没用）；过于激进 → 漏报风险（不安全）

### 评测方案

- **基准**：构建含动态执行真值的 Java 漏洞可达性数据集（现有基准全为 C/C++）
- **指标**：必须同时报告误报消减率和漏报率，按断图机制分层（反射/DI/SPI/反序列化）
- **真值**：用动态执行（插桩或测试用例触发）确认 sink 是否实际可达

### 预期影响

将 SCA 误报消减从"版本维度"推进到"可达性维度"，且首次引入健全性约束。解决的是工业界 SCA 工具最痛的误报问题——数千条告警中 90%+ 是不可达的。

## 议程 2（高优先）：面向真实部署的离线/小模型可行性研究

**反向引用**：§9.2 发现 1 — 48 篇中仅 5 篇使用本地模型；§9.3 矩阵空白 (all, D2) 稀疏。

### 问题定义

现有工作几乎全部依赖 GPT-4/5 级云端大模型（48 篇中 43 篇）。离线/气隙/等保（MLPS 2.0）合规场景无法使用。虽然已有存在性证明（Bridging <4B 达 93.57%、SecureFixAgent <8B Fix Acc 87.83%、Boosting-Pointer Qwen3-14B），但缺乏系统的权衡分析。

### 技术路线

1. **多任务能力基准测试**：在同一组 Java SCA 任务（版本定位/函数定位/可达性判定/误报分类）上，量化 7B/14B/32B 本地模型 vs GPT-5/Claude-4 的能力差距
2. **上下文压缩效果研究**：CPG 切片/SARIF 轨迹/语义摘要对小模型的增益——小模型受限于上下文窗口，精确切片可能比大模型无切片更关键
3. **"模型规模-任务精度-部署约束"权衡曲线**：给出不同部署场景（云端/私有云/单机/气隙）的模型选型建议

### 预期影响

为企业/政府/等保等受限环境给出可落地的模型选型指南。证明"离线小模型 + 智能预处理"可以在关键任务上缩小与云端大模型的差距。

## 议程 3（高优先）：统一的健全性评测框架

**反向引用**：§9.1 观察 1 — 论文集中于"只剪误报"，无人系统报告漏报率；§9.1 观察 2 — 纵轴顶部近乎空白。

### 问题定义

48 篇论文中，46 篇只报告 Precision/FPR/F1 或误报消减率，几乎无人系统报告"引入了多少漏报"。这是本综述双轴分析暴露的最大盲区——安全性不可审计，不同论文的方法无法在安全性维度上比较。

### 技术路线

1. **标准化指标体系**：在传统 P/R/F1 之外，强制要求：(a) FN rate / 漏报率 (b) 按失败模式分层（断图机制/LLM 误判类型/CWE 类别）(c) 误消减 vs 漏报的消融分析
2. **建立含动态执行真值的 Java 基准**：现有基准几乎全是 C/C++ 的函数/片段级漏洞检测，缺乏 Java SCA 可达性场景
3. **要求论文同时报告"消减了什么"和"代价是什么"**：FP 消减率必须伴随 FN 引入率

### 预期影响

让该领域实验结果可比较、安全性可审计。短期目标：一篇论文报告了"FP 消减 80%"时，审稿人能追问"FN 引入了多少"——这是该领域成熟化的必要条件。

## 议程 4（中优先）：跨语言/跨框架的动态分发语义建模

**反向引用**：§9.3 横轴 A 缺 Java；观察 3 — SemTaint (D1●) 仅限 JS。

### 问题定义

反射/DI/SPI 是 Java 特有表现，但"运行时动态分发让静态分析断裂"是跨语言共性：Python monkey-patching、JS 动态 require/eval、Go interface 动态分发。SemTaint 证明了 LLM 修复调用图是可行的（JS 场景，恢复 65.4% 漏报），但其 TICR 算法是 JS 特化的。

### 技术路线

1. 抽象"动态分发边"的统一表示——跨语言通用
2. 研究 LLM 跨语言迁移：在 JS 上训练的边修复模型能否迁移到 Java/Python？
3. 针对 Java 特有机制（Spring DI、SPI、反序列化、动态代理）构建专项评测

### 预期影响

把议程 1 从 Java 推广到多语言。长期目标：一个通用的"LLM + 静态调用图"可达性判定框架，语言相关部分仅在于动态分发边的识别模式。

## 议程 5（中优先）：可解释证据链与合规对接

**反向引用**：§9.2 发现 2 — 可解释性整体薄弱；D5 统计 — 0 篇 ● 覆盖。

### 问题定义

LLM 判定是黑箱。等保/SOC2 等合规框架要求可追溯审计证据——安全团队需要的不只是"LLM 说这是误报"，而是"为什么这是误报"以及"证据是什么"。48 篇论文中，无一提供可自动验证的结构化证据链。VulAgent（D5◐）和 Sifting（D5◐）输出文本解释但无法自动审计。

### 技术路线

1. 让 LLM 输出结构化证据链：(a) 完整调用路径（从 source 到 sink 的每个 hop）(b) 每个断边的 LLM 判定依据（为什么认为这条边不可达）(c) 可验证性标签（哪些 hop 是静态确定的，哪些是 LLM 推断的）
2. 研究证据链的可验证性：(a) 与动态执行结果的一致率 (b) 人工复核成本 (c) 能否自动化部分验证

### 预期影响

打通"LLM 辅助安全分析 → 合规交付物"的最后一公里。当安全审计员问"你凭什么说这个 CVE 不可达"时，能拿出一条可追溯的证据链而非一句"GPT 说的"。
