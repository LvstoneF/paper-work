# 方向②: SCA 可达性与漏洞函数/版本定位

## 2.1 本章定位

软件成分分析（SCA）的核心任务是：当应用依赖的开源组件曝出 CVE 时，判断该漏洞是否真正影响当前系统。传统 SCA 工具仅做版本比对——如果应用的依赖版本落在 NVD 报告的"受影响版本区间"内，即告警。这一策略导致极高的误报率（商业 SCA 产品报告 70--80% 的依赖从未在应用代码中被引用 [Veracode Dynamics, 2019]），因为版本匹配既不验证函数是否被调用，也不验证调用路径是否连通。

本章追踪 SCA 从**版本比对到函数定位再到语义可达性判定**的三阶演进，通过 VFArche、VERCATION、SAVANT、Veracode Dynamics、Patch-Localization、Similar-but-Patched、DGA 和 SLICEMATE 八篇工作，展示这一领域的进展与共同局限。

## 2.2 第一阶：从版本到函数/补丁

### 2.2.1 VFArche: 定位"漏洞函数"在哪

VFArche（arXiv: 2506.18050）解决的问题是：NVD 只告诉你哪个组件有漏洞，但不告诉你漏洞具体在哪个函数里。没有"漏洞函数（Vulnerable Function, VF）"，就不可能做调用图可达性分析。

VFArche 的核心发现具有奠基性意义：在 600 个 Java CVE 的系统标注中，**26.67% 的漏洞函数根本不在补丁修改的函数范围内**。这意味着"补丁改了哪些函数 → 那些函数是漏洞函数"这一直觉有超过 1/4 的错误率。四种 VF 不在补丁中的模式分别为：配置变更（61%）、方法替换（17.83%）、追加参数（13.33%）和类替换（7.83%）。此外，CVE 描述的语义信息也严重不足——仅 4% 明确指出了 VF，14% 包含的函数名甚至是误导性的。

VFArche 采用三阶段架构：Phase 1 用 CWE 知识库扩充 CVE 描述；Phase 2 有补丁时用 Tracker（调用图+数据流追踪规则引擎）筛选候选函数（全仓库约 15,000 个方法缩减到约 100 个），无补丁时用 CodeBERT 分类器做知识迁移；Phase 3 用 Swiss-system 竞赛排序（仅需约 400 次 LLM 查询而非全量 4,950 次）将候选排序后输出 top-10。有补丁时 Recall@10 达 96.3%，无补丁时达 84.0%。应用到真实 SCA 工具（Snyk、OWASP DC、Dependabot）上，误报率从 88--96% 降至 10--12%。

**局限**: VFArche 定位"漏洞函数"但**不验证它在应用调用图中是否可达**；Phase 3 依赖 GPT-4.1 在线 API，不可离线；Tracker 规则是 Java 专用的，不处理反射/Spring/SPI 造成的调用图断裂。

### 2.2.2 VERCATION: 定位"漏洞版本"区间

VERCATION（arXiv: 2408.07321）解决的则是在时间轴上定位漏洞的引入点：从修复 commit（vfc）回溯到漏洞引入 commit（vic），确定哪些版本真正受影响。其核心发现同样具有冲击性：在扫描 NVD 的 342 个 CVE 后发现，**59.1% 的 NVD 受影响版本信息有误**（134 个漏报，68 个多报），NVD 自身的 F1 仅 51.4%。

VERCATION 的方法设计是三篇中"静态分析 + LLM 协同"的范本：Phase 1 用 Joern 做程序切片（平均切出 22.64 条/CVE），再用 DeepSeek-V3（Few-shot + CoT）精筛到 5.66 条；Phase 2 用 AST 展开与归一化技术处理重构提交（32.2% 的中间提交是重构提交）对回溯的干扰，使用敏感函数加权提升高危险语句的匹配权重。

消融实验清晰展示了协同效应：纯静态（Joern）F1 48.3%，纯 LLM（DeepSeek-V3）F1 76.8%，**两者结合达 93.1%**。每个 CVE 平均分析时间 28.61 秒，API 成本仅 $0.0034/CVE（DeepSeek-V3），比动态执行约 3 小时/CVE 的效率高约 742 倍。知识截断时间分析证明 LLM 依赖的是代码理解能力而非记忆——post-cutoff CVE 上 F1 甚至略高（94.3% vs 92.7%）。

**与 VFArche 的关系**: VFArche 回答"漏洞函数是哪个"，VERCATION 回答"漏洞版本是哪个区间"——两者互补但都止步于可达性判定之前。两者均未验证"已知漏洞函数/版本在当前系统的调用图中是否可达"。

### 2.2.3 Veracode Dynamics: 静态+动态组合的先例

Veracode Dynamics（arXiv: 1909.00973）是工业界在 SCA 可达性上的早期探索。它采用**静态调用图与动态调用图（测试插桩获取）模块化组合**：预计算第三方库的调用链，应用端只需判断这些链是否与应用调用图相接。

三个关键数据至今仍有参考价值：(1) 70--80% 的依赖在应用代码中从未被引用——SCA 误报的主要来源；(2) 纯静态调用链中 25.5% 在运行时不可行（Rountev 2004）；(3) 动态调用图比静态平均多 824% 顶点、361% 边——说明静态调用图在 Java 动态分发场景下存在严重断裂。

**局限**: 动态分析依赖测试覆盖率（覆盖率不足则引入漏报），且需手动配置项目编译运行；不做控制流分析，方法级 sink 检查弱于完整的可利用性验证。尽管如此，其"预计算+模块化组合"的设计思想对后续工作（包括 VFArche 的 Tracker）有直接启发。

### 2.2.4 Patch-Localization: 补丁定位作为相邻任务

Patch-Localization / LLM-SPL（arXiv: 2409.06816）处理的是"给定 CVE 编号，在开源项目中找到修复它的 commit"这一相邻任务。其核心洞察是 LLM 不能直接做全量判定（$62 万美元+百年工期），而是先让传统推荐模型（VCMatch）排初榜，**只让 LLM 对 top-100 候选做相关性判定**（F1 特征）和 commit 协作关系构建（F2 图），成本降至 $880/3 天。

实验结果：全量数据集 R@10 = 92.74%，1-N 场景（21.04% 的 CVE 需多个 patch 协同修复）R@10 从 60.30% 提升至 83.13%。

**对本综述的意义**: LLM-SPL 的"LLM 作为特征增强器而非决策器"架构是双轴框架的一个典型案例——LLM 在成本可控的前提下做"补判"而非"主判"，这一思路对"静态分析做主判 + LLM 辅助补判"的议程 1 有直接借鉴意义。

### 2.2.5 Similar-but-Patched: 误报消减的零假阳策略

Similar-but-Patched / FVF（arXiv: 2412.20740）从代码克隆检测的视角进入 SCA 误报消减。核心问题是 SBP（Similar-but-Patched）——某段代码看起来和已知漏洞几乎一样，但实际已被修复，这是克隆检测工具误报的主要来源（四套工具平均误报率 76.2%）。

FVF 的解决方案不依赖 LLM，而是利用 git 变更历史匹配：如果检测器报警说"这段代码和漏洞 A 很像"，就去查该函数的 git 历史是否包含漏洞 A 的修复操作；有则判为假警。实验达到**零假阳**（无真漏洞被误过滤），将整体误报率从 76.2% 降至 26.6%，在 1,081 个真实项目中验证了 SBP 的普遍性（40% 的项目包含至少一个 SBP）。

**对本综述的意义**: FVF 是方向②中少有的明确以"不牺牲健全性"为设计约束的工作——虽然它不做调用图可达性，但提供了一个零漏报误报消减的参考模板，与议程 1 的"fail-safe"要求形成对照。此外，SBP 现象本身也说明了"代码相似度不等于漏洞传播"——版本级 SCA 需要更精细的语义判断。

## 2.3 第二阶：LLM 驱动的语义可达性判定

### 2.3.1 SAVANT: 用 LLM 语义替代调用图

SAVANT（arXiv: 2506.17798v2）是目前最接近"LLM 做 SCA 可达性判定"的工作，且是知识库中**唯一直接针对 Java 项目做 LLM + SCA 可达性判定**的研究。它的思路是完全跳过调用图：先用 AST 增强的代码切块做向量嵌入，再以漏洞补丁中的 PoV Test（漏洞证明测试用例）为检索种子，让 LLM 通过反射式迭代检索（iteratively 问"当前上下文够不够判断？还缺什么？"）来扩展上下文，最终由 LLM 判定该 API 调用在应用上下文中是否可触发。

在 55 个真实 Java 项目（25 个 CVE）上，SAVANT 的 F1 为 78.5%（精确率 83.8%、召回率 73.8%），远超 Eclipse Steady（F1 35.9%）和 VAScanner（F1 38.6%）。最佳配置（GPT-4o + 2500 tokens）F1 达 0.87。开源模型 Llama-3.1-405B 最佳 F1 仅 0.74，召回到 0.60——说明**当下开源模型在可达性判定上与 GPT-4o 仍有显著差距**。

**方法演进谱系中的位置**: VFArche（函数定位）→ VERCATION（版本定位）→ **SAVANT（语义级可达性判定）**。SAVANT 是三重中唯一直接回答"漏洞在应用中是否可触发"的工作，但它选择的路线是"**用 LLM 语义理解替代结构化调用图**"，而非"用 LLM 辅助补全调用图断裂边"。

**关键局限**: (1) 不构建调用图意味着完全无法处理 Java 动态分发（反射、Spring DI、SPI、反序列化）导致的图断裂；(2) 无零漏报约束，召回率 73.8% 意味着 26.2% 的漏报率；(3) 纯 LLM 语义判定的结果不可审计，输出文本推理过程而非结构化证据链——这与等保/SOC2 合规要求存在根本矛盾。

## 2.4 第三阶：范式转换与系统性尝试

### 2.4.1 DGA / LiveSCA: "无数据库"SCA

DGA / LiveSCA（arXiv: 2503.22576, FSE'25）提出了一个大胆的范式转换：**不做 SCA 特征数据库**。传统 SCA 依赖预建数据库维护第三方库特征，但 C/C++ 没有统一包管理器，数据库永远不全。LiveSCA 的替代方案是：提取代码语义特征 → 去 Google 搜索 → 多 LLM 智能体协作分析搜索结果 → 判断这是什么开源库。

实验结果：Android SO 库定位准确率 59.20%，C/C++ 克隆库定位 57.50%。准确率尚不敌传统有数据库方案，但证明了"不依赖数据库"的可行性。平均 29.36 秒/例，成本约 $8/200 例。

**对本综述的意义**: LiveSCA 间接暴露了 SCA 的"数据库依赖"问题——离线/内网场景下，即使数据库方案可用，数据库的更新滞后和覆盖不全仍是瓶颈。这对议程 2（离线小模型）和议程 5（合规对接）提供了实证支持。但它也暴露了"依赖互联网搜索 + GPT-4o"的部署约束——气隙场景完全不可用。

### 2.4.2 SLICEMATE: LLM 智能体做程序切片

SLICEMATE（arXiv: 2507.18957, ISSTA'25）将 LLM 智能体应用于静态程序切片——找出所有影响某条语句的代码行，本质上是**反向可达性分析**。其三个智能体（合成→逐步扩范围、验证→检查完整性、精修→修补错误）通过 ACI 接口（四种代码检索工具）协作，合成智能体从切片标准出发逐步扩范围（函数内→跨函数→跨文件），验证-精修循环至多 5 轮。

在自建基准 SliceBench（2,200 例，5--8,577 行，Java+Python）上，大代码场景下 Python F1 从 0.247（NS-Slicer Pro）提升至 0.761（+51.4%），Java F1 从 0.447 提升至 0.736（+28.9%）。传统工具（JavaSlicer/Joern）在平均 2,106 行代码的样本上精确率低于 0.2。下游 bug 定位 Top-10 准确率最多提升 34%。

**对本综述的意义**: SLICEMATE 证明了 LLM 可以不依赖依赖图做精确的依赖追踪——这直接启发议程 1：如果 LLM 能通过多智能体协作重建调用依赖关系，就有可能在不牺牲健全性的前提下修复调用图断裂边。但 SLICEMATE 本身不做健全性保证（验证-精修循环只能减少错误，不能消除错误），且未在含 Spring/反射的工业代码上验证。

## 2.5 共同局限

方向②的全部八篇工作呈现出以下共同局限，直接映射到本综述的研究议程：

1. **都不做调用图可达性验证**：VFArche 和 VERCATION 定位漏洞函数/版本但不验证调用路径；SAVANT 用 LLM 语义替代调用图但不处理图断裂；Veracode Dynamics 做到了但依赖测试覆盖率；Patch-Localization 和 Similar-but-Patched 处理的是相邻问题而非可达性本身；DGA 和 SLICEMATE 在更上游/下游的技术空间。**在"找到漏洞函数"与"确认漏洞可达"之间存在一条清晰的方法论鸿沟**——这正是议程 1 的核心空白。

2. **都不处理 Java 动态分发**：反射、Spring DI、SPI、反序列化等 Java 企业框架特性导致的调用图断裂，是方向②中无一工作系统解决的问题。VFArche 的 Tracker 是规则引擎不处理动态分发；SAVANT 完全依赖 LLM 语义不建调用图；SLICEMATE 未在工业 Java 框架上验证。这是议程 1 和议程 4 的直接驱动力。

3. **无一具备零漏报保证**：全部工作的评估指标都是 P/R/F1 或误报消减率，无人以"引入多少新增漏报"作为评价维度。Similar-but-Patched 是唯一以"零假阳"为设计约束的工作，但它处理的是克隆检测而非可达性。这一系统性缺失直接对应本综述双轴框架的纵轴——健全性代价。

4. **离线部署不可行**：全部方法在最佳配置下依赖 GPT-4 级云端大模型。Beyond ChatGPT（方向①）的系统多模型评估和 DGA/LiveSCA（方向②）的"无数据库"探索虽触及了部署约束，但均未以"离线小模型可行性"为研究目标。
