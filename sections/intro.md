# 引言

> 文献综述章节 · 动机与综述目标

---

## 1.1 背景

开源组件（Open-Source Software, OSS）已成为现代软件工程的基石。据 Synopsys 2024 年报告，97% 的代码库包含开源组件，其中 84% 至少包含一个已知漏洞。软件成分分析（Software Composition Analysis, SCA）和安全静态分析（SAST）工具应运而生，但面临一个共同的核心困境：**误报太多，而真实漏洞又被淹没在告警洪水中**。

大语言模型（Large Language Models, LLMs）在 2022-2026 年间展现出强大的代码语义理解能力，研究者开始探索用 LLM 弥补传统程序分析工具的固有短板。然而，**这条路线面临一个被系统性忽视的问题**：LLM 消减误报的同时，是否会引入新的漏报（False Negatives）？如果 LLM 错误地将真实漏洞标记为误报并屏蔽，其安全后果远比静态工具的误报更严重——误报消耗人力，漏报直接导致安全事故。

---

## 1.2 动机：现有综述的盲区

截至 2026 年，已有 3 篇综述覆盖"LLM + 软件安全"领域 [sheng2025, basic2024, wang2025]，但存在三个共同盲区：

1. **未将"可达性验证"作为独立分析维度**。现有综述按"检测→修复→评估"流水线分类，未区分"版本级 SCA"和"调用图可达性级 SCA"——这是两个完全不同的技术问题。
2. **未讨论健全性（Soundness）代价**。几乎全部已有工作只报告误报消减率（Precision/FPR），几乎无人系统报告引入了多少漏报。这个盲区对安全领域不可接受——一个自动屏蔽系统如果存在不可控的漏报风险，其部署价值极为有限。
3. **未覆盖离线/内网部署约束**。现有工作默认依赖 GPT-4 级云端大模型，未分析等保（MLPS 2.0）合规、气隙网络等场景下的可行性。

---

## 1.3 本综述的目标与核心贡献

**本综述的核心交付物是一份具体的、可执行的研究议程（Research Agenda）**——不是泛泛的"未来方向"，而是基于双轴矩阵分析和部署就绪度评估推导出的 5 条有优先级、有技术路线、有评测方案的立项指南。

为此，本文提出三重贡献：

1. **新分类法（双轴框架）**：
   - **横轴** = LLM 补偿静态分析的哪个短板（调用图不完整 / 模式不精确 / 污点规范缺失 / 版本-可达性鸿沟 / 数据与标注质量）
   - **纵轴** = 补偿对健全性的代价（只剪误报可能引入漏报 / 只补漏报 / 声称两者兼顾 / 零漏报保证）
   - **叠加**部署就绪度评分（可编译性、云端依赖、可解释性、真实代码库验证）

2. **健全性代价系统化**：首次将"有无漏报分析""有无零漏报保证"作为评价维度，暴露该领域被系统性忽视的安全风险。

3. **证据驱动的 5 条研究议程**：双轴矩阵定位 → 发现空白 → 就绪度评估 → 推导议程。每一条议程都有具体的矩阵坐标反向引用，不是凭空提出。
