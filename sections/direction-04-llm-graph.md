# 方向 4：LLM + 调用图 / 代码属性图（CPG）

## 7.1 解决的问题

静态程序分析中的图结构（调用图 Call Graph、控制流图 CFG、程序依赖图 PDG、代码属性图 CPG）是理解代码结构和数据流的基础工具。然而，传统图分析有三个固有缺陷：（1）**图规范缺失**——污点分析的 source/sink 规范需人工标注，工作量大且易遗漏；（2）**图精度不足**——静态构建的图因缺失运行时信息而过度近似，产生大量误报；（3）**图本身断裂**——动态分发（反射、依赖注入、SPI、多态回调）导致调用图中存在大量无法静态解析的边，切断数据流路径，直接造成漏报。

LLM 凭借从大规模代码语料中习得的语义知识，为弥补上述三个缺陷提供了新可能。本章将方向 4 的 11 篇论文归纳为三种范式进行讨论。

## 7.2 范式一：LLM 推断图规范

此类工作的核心洞察：静态分析之所以精度不足，部分原因是缺少准确的"什么是什么"的语义规范——哪些 API 是数据入口（source）、哪些是危险操作（sink）、数据如何在调用间传播。传统做法由安全工程师逐条手动编写，LLM 可用于替代或辅助这一过程。

**IRIS（Li 2024, ICLR 2025）** 是该范式的里程碑工作，也是本综述的必引锚点。它将 LLM 插入传统 CodeQL 污点分析流程的两个位置：

- **阶段 1：污点规范推断**。LLM 对项目中使用的外部 API 和内部函数参数进行分类，自动标注 source/sink。外部 API 用 few-shot 提示，内部 API 用 zero-shot 提示附上 JavaDoc/README。
- **阶段 2：上下文分析过滤**。对 CodeQL 检测出的每条数据流路径，LLM 根据 CWE 描述和路径上下文判断是否在真实代码中可利用。

IRIS 在自建数据集 CWE-Bench-Java（120 个可编译的真实 Java 项目，每个含手动验证的漏洞）上，将 CodeQL 的检测数从 27 提升到 55（+103.7%），平均 FDR（误报率）降低 5.21 个百分点，还发现了 4 个前人工具未检出的 0-day 漏洞。消融实验证明两个组件缺一不可：单独 LLM source + CodeQL sink 仅检测出 36 个，LLM sink + CodeQL source 仅 24 个，而两者结合能检出 55 个。

值得注意的是，开源小模型 DeepSeekCoder 7B 检测出 52 个，仅比 GPT-4（55 个）少 3 个，说明框架设计本身比模型大小更重要。但小模型 FDR 高达 95.40%，几乎不可用——"检测数接近但误报率高企"是开源小模型在该方向上的典型特征。

**IRIS 的局限**直接对应本综述的多项研究空白：（1）依赖完整 Java 编译环境（Maven build）；（2）小模型 FDR 过高，最佳效果需要 GPT-4 级云端大模型；（3）没有零漏报保证，明确承认 LLM 漏标 source/sink 会直接导致漏洞被漏检；（4）不处理 Java 动态分发——CodeQL 的调用图基于静态解析，反射/Spring DI/SPI 导致的动态边不在 IRIS 范围内。

**SemTaint（Ghebremichael 2026）** 也包含图规范推断组件（Source/Sink Agent 按 CWE 提取包特定的 source 和 sink），但它的核心贡献在范式三（修复断裂图），将在 7.4 节详述。

## 7.3 范式二：图引导 LLM 上下文

此类工作的核心洞察：LLM 做漏洞检测或代码定位时，如果直接将整个函数或文件送入，大量无关代码会稀释注意力；如果借助图结构做精确的上下文裁剪，LLM 的效率和质量都可以显著提升。这是"静态图做筛选 + LLM 做语义理解"的经典混合范式。

**LLMxCPG（Lekssays 2025）** 采用双模型流水线。LLMxCPG-Q 接受过专门微调，能写出 Joern 的 CPGQL 查询语句，自动从代码中提取与漏洞相关的执行路径切片——将代码量减少 67--91%，只保留关键路径。LLMxCPG-D 对切片做二分类（有漏洞/安全），F1 比 SOTA（JOIN 等纯图方法）提升 15--40 个百分点。在 PrimeVul 数据集上，Precision 达 1.0，但 Recall 仅 0.45——表明模型极度保守，宁可漏报也不误报。消融实验证实，去掉 CPG 切片后所有数据集上的准确率下降 16%--33%。

LLMxCPG 的独特贡献在于"LLM 控制静态分析工具"的思路——LLM 不是被动接受上下文，而是主动生成图查询来驱动静态分析。但局限同样突出：仅限 C/C++ 的内存类漏洞（CWE-119 系），依赖 Joern 的完整编译环境，未涉及任何 Java 动态分发场景。

**LocAgent（Chen 2025）** 将代码仓库转化为有向异构图（节点=目录/文件/类/函数，边=包含/导入/调用/继承），然后让 LLM Agent 用三个工具（SearchEntity、TraverseGraph、RetrieveEntity）在图结构上做多跳推理，完成代码定位任务。在 SWE-Bench-Lite 上文件级 Acc@5 达 94.16%，函数级 Acc@10 达 77.37%。微调后的 Qwen2.5-32B 效果接近 Claude-3.5，但成本降低 86%（从 $0.66 降至 $0.09 每次）。

虽然 LocAgent 的定位任务并非直接的安全分析，但其"图索引+Agent 搜索"范式可直接复用在安全审计中：给定 CVE 描述，定位受影响的代码位置。LocAgent 为议程 2（离线/小模型可行性）和议程 5（可解释证据链——Agent 轨迹天然可追溯）提供了正向实证。

**Vul-LMGNNs（2024）** 走了另一条技术路线：不是让图引导 LLM，而是让 LLM（CodeBERT/CodeT5 等代码语言模型）提供语义嵌入来初始化 GNN 的节点表示，再通过在线知识蒸馏让结构信息在 GNN 层间传播。融合后的模型在 VDISC 上 F1=83.87%，在 DiverseVul（极度不平衡数据集）上 F1=32.59%，比最佳基线 CodeT5-Base（26.40%）高出 6.19 个百分点。不过它实质上是函数级二分类模型，不涉及调用图分析或可达性验证。

**Bridging（Gajjar 2025）** 是另一种融合范式，也是本方向中 D2（离线小模型）最突出的代表。它用双向门控融合机制将控制流图（CFG）的结构嵌入和轻量级本地 LLM（<4B：Qwen2.5 Coder 3B、DeepSeek Coder 1.3B、Phi 3.5 Mini 等）的语义嵌入结合，做 Java 漏洞检测的二分类。在 35,610 个 Java 样本上准确率达 93.57%，比纯 GAT 图模型高 8.36 个百分点，比纯 LLM 零样本高 17.81 个百分点。门控权重同时提供可解释性——对每个样本，模型自动决定"偏重结构还是偏重语义"。

Bridging 的最大价值是为议程 2 提供了**直接存在性证据**：<4B 的本地模型 + 图结构融合即可在 Java 漏洞检测上取得竞争性结果，完全无需云端大模型。但需注意，这是函数级二分类而非组件级可达性分析，且安全样本为合成生成（从漏洞代码剥离产生），其真实代码逼真度存疑。

## 7.4 范式三：LLM 修复断裂图

这是方向 4 最前沿的范式——LLM 不满足于"读图做判断"或"用图做裁剪"，而是直接修复静态分析无法解析的图断裂。该范式的研究工作数量最少，但与本综述研究议程 1（带健全性保证的 SCA 可达性验证）直接对应。

**SemTaint（Ghebremichael 2026）** 是本方向中**唯一处理动态分发断边**的工作，也是本综述的必引锚点。SemTaint 是一个三智能体系统（Source/Sink Agent + CallGraph Agent + Flow Summary Agent），核心创新是 **TICR（Taint-Informed Callee Resolution）**——一种双向算法，只对"处于潜在漏洞路径上的不可解析调用点"进行 LLM 修复，而不是穷举修复全部不可解析调用。对于 JavaScript，SemTaint 首次大规模量化了调用图断裂的程度：CodeQL 在 2,849,543 个调用点中只能解析 40.3%，剩余 59.7%（1,702,150 个）不可解析。

在 162 个 CodeQL 之前完全检测不到的 JavaScript 漏洞上，SemTaint 恢复了 106 个（65.43% 召回率）。TICR 将 LLM 调用量减少了 94.5%（从每包 585.9 降至 32.1 次），需求驱动的第三方边分析减少 93.2%。消融实验显示，73% 的检测提升来自 Source/Sink Agent 提供的包特定 source/sink，而 27% 必须依赖 CallGraph Agent 修复断裂调用边——这 27% 是硬检测天花板：无论 source/sink 多精确，不修复调用图断裂就检测不到。

SemTaint 的纵轴位置属于"声称两者兼顾"（既补漏报也消误报），但无零漏报保证。论文承认 TICR 的扩展语义规则是启发式的而非完备的，intra-procedural 断裂也被排除在外。它目前仅限 JavaScript/TypeScript，依赖 GPT-5 级云端大模型（离线小模型方案未探索）。

**Boosting Pointer Analysis（CAFD, 2025）** 是另一条修复路径：LLM 帮助指针分析识别自定义分配函数（Custom Allocation Function, CAF），让指针分析区分来自不同函数调用点的堆对象，从而提升别名分析的精度。它先用轻量值流追踪找候选 CAF，再让 LLM（phi-4、Qwen3-14B 等 3B--14B 本地模型）判断候选函数中的副作用语句是否仅存在于错误处理路径——如果是，可安全地当作标准分配函数建模。

在 17 个真实 C 项目上，CAFD 找到了 700+ 个此前未被识别的 CAF，别名校减少了 41.5%，下游检测器发现了 29 个之前漏掉的内存漏洞。LLM 精度 100%（不引入错误标签），召回率 77.1%。全部使用本地开源小模型（不需要云端 API），为议程 2 提供了 D2● 级别的证据。但工作聚焦 C 语言的指针分析，不涉及 Java 动态分发。

**LLMDFA（Wang 2024, NeurIPS 2024）** 的范式更为激进：不是修复传统数据流分析工具，而是用 LLM 完全替代传统编译型数据流分析的基础设施。它将数据流分析拆解为三个子任务：（1）LLM 合成 Python 脚本来提取 source/sink（确定性、可审查）；（2）CoT 推理做函数内数据流汇总；（3）SMT 求解器做路径可行性验证。在 Juliet 数据集上，平均 F1 超过 CodeFuseQuery 0.23、超过端到端 LLM 方案 0.36。真实 Android 恶意软件（TaintBench）上 F1=0.67，显著优于 CodeFuseQuery（0.55）和端到端方案（0.32）。

LLMDFA 的设计哲学是"LLM 合成确定性的工具+在 LLM 不可靠的地方用形式化方法兜底"——与 LLM4PFA（方向 3）的思想一脉相承。它声称可多语言扩展（仅改 <100 行代码就支持 C/C++ 和 JavaScript），但仅在简单语言构造上验证，未处理 Java 框架动态分发。路径编码仍可能出错（附录报告了多条误编码路径条件导致漏报的案例），无零漏报保证。

**VulAgent（2025）** 使用 Joern 提取的 CFG、DFG 和调用图作为"假设验证"的程序上下文——每个可疑的"漏洞假设"需要用图结构中的路径来验证是否可行。它不是修复图断裂，而是用图来约束 LLM 的推理不偏离事实。在 PrimeVul 和 SVEN 上，VulAgent 的 FPR 比直接 CoT 评估低约 29 个百分点（20.48% vs 49.40%），且假设验证比文本式"自我反思"更有效。但 VulAgent 依赖 GPT-4o/DeepSeek v3.1 等云端大模型，每样本平均 8.83 次 LLM 调用，离线部署成本高。

**AndroByte（Khatun 2025）** 用于 Android 隐私分析，用 LLM 读 Smali 字节码，自然语言总结每个方法的行为，递归生成动态数据流感知调用图（D2CFG）——只追踪从敏感数据源出发的调用链，而不是全量调用图。在 300 个真实 App 和 DroidBench/UBCBench 基准上，使用 Gemma3（8--15B 参数本地模型），泄露检测 F1=83.42%，超过需 82 个预定义 sink 的 FlowDroid（F1=78.13%）。其核心卖点是"不用预定义 sink 列表"和"全本地运行"，每个 App 平均 2.5 分钟。

**GRACE（Lu 2024）** 也属于图修复范式，但未获得详细分析数据，在此仅作收录提及。

## 7.5 三种范式的对比分析

| 维度 | 范式一：LLM 推断图规范 | 范式二：图引导 LLM 上下文 | 范式三：LLM 修复断裂图 |
|------|----------------------|----------------------|----------------------|
| 代表论文 | IRIS, AdaTaint, SemTaint (部分) | LLMxCPG, LocAgent, Vul-LMGNNs, Bridging | SemTaint (核心), Boosting-Pointer, LLMDFA, AndroByte, VulAgent |
| 核心设计 | LLM 替代人工写 source/sink 规则 | 图结构做上下文选择/导航, LLM 做语义推理 | LLM 修复静态分析无法解析的断边或缺失信息 |

| 目标语言 | Java | C/C++, Python | JS, C, Android, 多语言 |

## 7.8 共同局限

1. **D1●+D2●+D3● 三重空白**。方向 4 的 11 篇论文中，没有一篇同时满足三个条件：处理动态分发断边（D1●）、支持离线小模型（D2●）、提供零漏报保证（D3●）。SemTaint 覆盖 D1● 但无 D2● 和 D3●；Boosting-Pointer 和 Bridging 覆盖 D2● 但无关 D1●；IRIS、LLMDFA 等 D1 和 D3 均为空白。这直接对应研究议程 1 的联合攻关目标。

2. **Java 动态分发仍是无人区**。SemTaint 处理了 JavaScript 的动态调用解析，但 Java 企业系统的动态分发（反射、Spring DI、SPI、反序列化、AOP 切面）比 JavaScript 更复杂——涉及框架级容器管理、XML/注解配置、运行时字节码增强等。本方向上无一篇论文处理 Java 场景的动态分发断裂。IRIS 虽然使用 Java，但它依赖 CodeQL 已有的静态调用图，不做任何断边修复。

3. **评估基准差异大，无法直接对比**。11 篇论文使用了 8 种不同的数据集（CWE-Bench-Java、DroidBench、PrimeVul、SVEN、VDISC、DiverseVul、OWASP Benchmark、自建数据集等），报告指标各异（F1、Acc、P-C、FDR、TPR、召回率、漏报识别率等），彼此之间难以横向比较。这验证了议程 3（统一健全性评测框架）的必要性。

4. **与 SCA 场景脱节**。除 SemTaint 的 TICR 间接涉及版本可达性外，所有工作都是"给定一份代码，判断是否存在漏洞"——这是程序分析视角，而非 SCA 视角。SCA 场景中的核心问题"已知 CVE 在应用中的调用子图中是否可达"，未被方向 4 的任何工作专门处理。这直接对应研究空白①（少有把 LLM 用于开源组件调用图可达性）。

5. **大部分依赖云端大模型**。11 篇中仅 Bridging（<4B）、Boosting-Pointer（3B--14B）、LocAgent（7B/32B 微调后可离线）和 AndroByte（8--15B）证明了本地小模型的可行性。其余 7 篇的最佳结果均需要 GPT-4 或 GPT-5 级别的云端模型。

6. **可解释性不足**。大多数论文仅输出"有漏洞/无漏洞"或"TP/FP"的二分类标签，未提供结构化的审计证据链（调用路径、判定依据、置信度分层）。VulAgent 的输出结构化为"触发路径+假设验证结果"是最接近合规要求的，但仍未达到可追溯审计的程度。这对应议程 5（可解释证据链与合规对接）。

## 7.9 与双轴框架的关系（本章小结）

方向 4 在双轴矩阵中的分布比方向 3 更为分散。横轴覆盖了调用图不完整（A）、模式不精确（B）和污点规范缺失（C）三个坐标，但纵轴整体偏低——仅 SemTaint 声称"两者兼顾"但无硬保证，其余均处于"只剪误报"区间。**最显著的空缺在双轴矩阵的右上角**：横轴 A（调用图不完整/可达性）× 纵轴 4（零漏报保证）的区域为空，且该区域同时叠加 D2（离线小模型）的需求时更是完全空白。这正是研究议程 1（带健全性保证的 SCA 可达性验证）的精确坐标定位。
