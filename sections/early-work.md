# 早期工作：LLM 漏洞检测/修复能力的基础探索 (2022–2023)

## 章节概述

2022–2023 年是 LLM 在代码安全领域应用的第一波探索期。这一时期的研究普遍尝试让 LLM（主要是 GPT-3.5/ChatGPT/GPT-4 系列）直接充当独立的漏洞检测器或修复器——将代码片段输入提示词（prompt），询问"这里有漏洞吗"或"请修复这段代码"。本章覆盖 8 篇代表性早期工作，核心论点是：**早期实证一致表明纯 LLM 路线有明确上限，LLM 无法可靠替代专业静态分析工具**。这一反面参照为 2024–2026 年"静态分析提供结构、LLM 提供语义"的互补范式（脉络一）提供了关键的方法论转向动力。

---

## 2.1 代码生成与安全：Copilot 时代的警钟

### 2.1.1 Pearce et al. (2021): Copilot 的 40% 不安全代码率

Pearce et al. (2021, arXiv:2112.02125) 是 LLM 代码安全评估的早期奠基性工作。他们在 89 个场景中测试了 GitHub Copilot（基于 Codex/GPT-3 系列的代码补全模型）在安全敏感上下文中的代码生成行为，发现在约 **40% 的场景中 Copilot 会生成包含已知漏洞模式的代码**。这一数字在后续研究中被广泛引用，成为"LLM 生成不安全代码"问题的最早量化证据。

该研究的贡献在于：(i) 定义了 LLM 安全评估的命题式测评范式（给定一个有漏洞风险的编程场景，看 LLM 是否触发已知 CWE 模式）；(ii) 将安全评估从"LLM 能修复漏洞吗"扩展到了"LLM 会制造漏洞吗"。Copilot 的闭源属性和模型迭代速度使得 Pearce 的具体数字已经历次更新，但其揭示的"LLM 默认不安全"现象作为领域共识已被后续大量工作（包括 RealSec-bench 的 SecurePass@1 < 6% 结果）所验证。

## 2.2 漏洞检测：LLM 直接推理的尝试与局限

### 2.2.1 Xu et al. (2022): Code LLM 能力的系统评测

Xu et al. (2022, arXiv:2202.13169) 对当时的代码 LLM（主要是 Codex/GPT-3 系列）在多种代码理解任务上的能力做了系统评测，覆盖了代码补全、缺陷检测、测试生成等维度。其评测框架为此后 LLM 代码能力评估的方法学提供了参考。在漏洞检测维度上，该研究揭示了 LLM 的早期局限性：**LLM 在不经微调或专门提示设计的情况下，对漏洞的检测能力与随机基线差异不显著**。这一发现奠定了后续若干工作的基调——LLM 需要专门的提示工程（prompt engineering）或微调才能发挥安全领域的作用。

### 2.2.2 Fu et al. (2023): ChatGPT 漏洞检测能力边界

Fu et al. (2023, arXiv:2310.09810) 是首批系统评估 ChatGPT（GPT-3.5/GPT-4）在漏洞检测任务上能力边界的实证研究之一。通过与传统的深度学习模型（如 CodeBERT、GraphCodeBERT）和静态分析工具对比，该研究发现 ChatGPT 在检测已知漏洞类型上展现出一定的语义理解能力，但在以下场景中表现不稳定：

- **未见漏洞类型泛化**：在训练数据 cutoff 日期后披露的新漏洞类型上，ChatGPT 的检测精度显著下降
- **上下文长度敏感**：当输入代码片段超过 prompt 窗口大小时，关键漏洞信息被截断导致检测失败
- **检测不一致性**：对同一漏洞片段在不同推理轮次中的判定结果可能不一致

该研究的两大贡献是：(i) 首次对比了 ChatGPT 与当时最佳专用模型的能力差距；(ii) 揭示了 LLM 直接检测范式的固有局限——没有静态分析的结构化支撑，纯语义判断不可靠。

### 2.2.3 Gao et al. (2023): LLM 漏洞检测能力评估

Gao et al. (2023, arXiv:2311.12420) 从更系统的角度评估了 LLM 在漏洞检测上的能力，覆盖了更多模型（GPT-3.5、GPT-4、Llama 系列）和更多漏洞类型。其核心结论与 Fu et al. 一致：**LLM 能够检测简单的、有明确模式的漏洞（如 SQL 注入、命令注入），但在复杂的过程间漏洞和逻辑漏洞上表现差**。该研究还首次系统化分析了 LLM 对提示模板的敏感性——同一漏洞在不同提示措辞下的检测结果可能相差 20% 以上。

### 2.2.4 Zhang et al. (2023): Prompt 增强漏洞检测

Zhang et al. (2023, arXiv:2308.12697) 沿着"如何设计更好的提示"这一方向推进，探索了 prompt engineering 对 LLM 漏洞检测能力的提升效果。该研究尝试了 Chain-of-Thought（CoT）推理和结构化提示模板，发现：

- CoT（让 LLM 先分析代码功能、再分析安全风险）比直接"有无漏洞"提问的检测率提升 10–20%
- 但 CoT 同时也增加了 FP（误报）率——LLM 在"过度分析"下倾向于将安全代码也判定为有风险
- 提供相关代码上下文比不提供的检测率提升显著，但上下文质量（是对应漏洞路径的还是无关的）影响巨大

这一发现为此后方向④（LLM + 图/代码属性图）的工作奠定了研究基础——方向④ 的核心命题正是在"如何提供高质量的结构化代码上下文"。

### 2.2.5 Chan et al. (2023): ChatGPT vs CodeBERT 的对比

Chan et al. (2023, arXiv:2306.01754) 提供了一个有趣的对比视角：ChatGPT（通用对话模型）vs CodeBERT（代码专用预训练编码器）在缺陷检测任务上的能力差异。其核心发现是：**CodeBERT 在精准率上优于 ChatGPT，但在召回率上不如**——CodeBERT 能更准确地锁定有问题的代码行（"疑似有问题的区域"），但 ChatGPT 在覆盖更多漏洞类型时的语义理解更灵活。这暗示了两种技术路线的互补潜质：预训练编码器的结构理解 + 生成模型的全景语义。

### 2.2.6 Ullah et al. (2023): "LLMs 不可靠"的里程碑级结论

Ullah et al. (2023, arXiv:2312.12575) 是早期工作中最常被引用的"反面结论"之一。该研究通过大规模实验（覆盖多个 LLM、多个漏洞类型、多个评测维度）提出：**在当前状态下，LLMs cannot reliably detect vulnerabilities（LLM 无法可靠检测漏洞）**。其核心论据包括：

- 对简单漏洞类型（如硬编码密码、XSS）LLM 表现尚可，但跨 CWE 类型的一致性极差
- LLM 的检测决策缺乏可解释性——无法提供可靠的结构化证据支撑其判断
- 不同 LLM 之间的检测结果一致性低，同一模型在不同提示下的稳定性也低

这一结论——虽然宽泛但被后续多项实证研究（如 ProjectScale 2026 的 Java 召回约 34%、TOSSS 2026 的最差模型 0.48 接近随机）所验证——从根本上动摇了"LLM 作为独立漏洞检测器"的范式，为领域转向"静态+LLM 互补"范式提供了关键的理论和经验动量。

## 2.3 自动程序修复：LLM 在修复任务上的初步尝试

### 2.3.1 Sobania et al. (2023): ChatGPT 自动程序修复

Sobania et al. (2023, arXiv:2301.08653) 将 LLM 的应用场景从"检测"扩展到"修复"——评估 ChatGPT 在自动程序修复（Automated Program Repair, APR）任务上的能力。在 QuixBugs 基准上，ChatGPT 在接近零样本条件下成功修复了约 30% 的 bug——显著低于当时最优的专用修复工具（如 TBar、CURE），但展示了 LLM 在无需任务特定微调的情况下即可理解并修改代码的潜力。

该研究还揭示了一个重要模式：**LLM 擅长修复"局部替换"类的 bug（如修改一个运算符、添加一个空检查），但对"结构性修复"（如重构一段逻辑、添加缺失的控制流分支）能力非常有限**。这一发现与后续 Verify-Before-Fix (2026) 的执行验证思路有方法论上的继承关系——LLM 的修复功能需要结构化验证来保证正确性。

## 2.5 核心发现与历史定位

### 发现一：纯 LLM 作为独立漏洞检测器存在明确上限

从 Xu (2022) 的"接近随机"到 Ullah (2023) 的"cannot reliably detect"，早期工作在近两年时间内用一致的研究结论回答了该范式的前提假设："LLM 直接检测漏洞"这一路线——将一段代码扔给 GPT-4 并问"有没有漏洞"——在方法学上是不可靠的。这不是模型迭代可以解决的问题（GPT-4 比 GPT-3.5 有所提升，但定性结论不变），而是范式本身的结构性局限——缺少静态分析的结构化约束，LLM 的语义判断缺乏可靠的上下文锚点。

### 发现二：为"静态+LLM 互补"范式提供了反面参照

正是早期工作揭示的纯 LLM 上限，推动了领域的方法论转向。这一转向的证据包括：
- IRIS (Li et al., 2024) 明确将 LLM 定位为"辅助静态分析推理"而非独立检测器
- LLMDFA (Wang et al., 2024) 将 LLM 定位为"代码合成器委托外部工具"而非直接推理器
- 保守分析 (Wagner et al., 2025) 将 LLM 定位为"SAST 的互补"而非替代

早期工作的结论为这一范式转向提供了实证动量——没有早期工作的"此路不通"结论，后续的"绕道而行"范式就缺乏立论基础。

### 发现三：修复能力的基础奠定了后续 Agent 方向

Sobania (2023) 的"局部替换有效、结构性修复困难"的发现，与 2025–2026 年方向⑤（Agent）多个工作的设计选择一致——Agent 论文普遍将执行验证作为修复正确性的保障（Verify-Before-Fix 的 strict invariant、CodeCureAgent 的三层验证）。早期工作的经验证明：**LLM 的修复输出不可信任，必须经过外部验证**。

### 发现四：数据质量问题已在评测中暴露隐患

Pearce (2021) 和 Fu (2023) 等早期工作使用的数据集（如基于 SARD/Juliet 测试套件的合成数据）存在与真实项目分布显著不同的偏差。这一问题直到 Croft (2023) 的数据质量审查才被系统化揭示（20-71% 标签不准），但早期工作受到该问题影响的定量评估至今仍是一个开放问题。本综述方向⑥（评测与数据集）进一步深化了这一批判。

## 2.6 与后续方向的关系

本节的 8 篇早期工作构成了后续所有方向的基础参照：

- **方向①（LLM 漏洞检测综述）**：3 篇综述均以早期工作为历史起点，分析"检测→修复→缓解"的演进（basic 2024）或"LLM 替代 vs 辅助"的范式转向（sheng 2025, wang 2025）
- **方向②（SCA 可达性）**：V-SZZ (2022) 虽然属于传统方法（非 LLM），但与早期 LLM 工作在时间上有交集，共同构成了 2022–2023 年的技术图景
- **方向③（LLM 消减误报）**：早期工作的"LLM 直接检测不可靠"结论是方向③ 的 motivation 来源——既然不能替代，那就作为辅助过滤工具
- **方向④（LLM + 图）**：早期工作没有利用结构化代码表示（AST/CFG/调用图），方向④ 的"将图注入 LLM"的思路正是对这一空白的有意填补
- **方向⑤（LLM Agent）**：早期工作的纯文本 prompt 范式被 Agent 替代为"工具调用+执行验证"的多步骤流水线，实现了从"在对话框里问"到"Agent 自主操作"的方法论跃迁
- **方向⑥（评测与数据集）**：早期工作使用的数据集（SARD/Juliet/Devign/BigVul）的数据质量在方向⑥ 中被系统性质疑和清洗
