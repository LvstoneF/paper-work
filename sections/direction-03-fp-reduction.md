# 方向 3：LLM 消减 SAST 误报

> 文献综述章节 · 覆盖 11 篇论文 | 叙事线：静态提示 → 上下文工程 → 智能体框架 → 路径可行性 → 保守策略 → 离线小模型突破

---

## 6.1 解决的问题

静态分析工具（SAST）在生产中面临一个公认的困境：告警中 30%--100% 是误报（False Positive, FP）[保守分析]。安全工程师的大量时间被消耗在逐一排查这些假警报上，导致真实漏洞被淹没在告警洪水中，最终被忽略。LLM 因其代码语义理解能力，被引入作为 SAST 告警的"二审法官"——接收 SAST 输出的告警及相关代码，判断其是否确为可利用的漏洞。

**本章的核心问题是：** LLM 消减 SAST 误报的效果如何？以什么样的健全性代价实现？能否满足离线/小模型部署的约束？

## 6.2 方法演进全景

本节按方法演进逻辑组织，从最早的静态提示、到上下文工程、到智能体框架、再到路径可行性与保守策略，展示该子方向从"让 LLM 读告警做判断"到"让系统保证不漏判"的渐进深化。

### 6.2.1 静态提示：首次尝试

**LLM4SA（Wen 2024）** 是该子方向的开创性工作。它将 SAST 工具产生的告警及所在函数整体送入 LLM，让 LLM 一次性给出"是否为误报"的二分类判断。在内存相关漏洞（buffer overflow、use-after-free 等）的告警过滤上，LLM4SA 报告了 81% 的精确率和 94% 的召回率[LLM4SA]。这一结果表明 LLM 具备辨别 SAST 误报的初步能力。

然而，LLM4SA 的局限性同样明显：（1）它对所有 CWE 类型使用同一套提示策略，未区分不同漏洞类的判断逻辑；（2）送入的上下文是未经裁剪的完整函数体，既包含干扰信息，又缺失必要的跨文件上下文；（3）仅覆盖内存相关漏洞，未验证注入类、密码学类等其他 CWE 类型。LLM4SA 为后续工作确立了基线，同时也暴露了"全函数喂入+统一提示"模式的天花板。

### 6.2.2 上下文工程：让 LLM 看到该看的东西

为解决 LLM4SA 的上下文质量问题，两条改进路线几乎同时出现。

**LLM4FPM（Chen 2025）** 走的是"精确切片"路线。它在 Joern 的 CPG（代码属性图）基础上扩展了三种依赖边（调用依赖 F、结构依赖 S、变量依赖 V），构建 eCPG，然后对 SAST 告警所在位置做行级精确切片，只保留与告警直接相关的代码行。同时提出 FARF（Find All Referenced Files）算法，用文件引用图+强连通分量（Tarjan 算法）快速定位告警依赖的所有跨文件代码。在 Juliet 数据集上，LLM4FPM 的 F1 超过 99%（远超标线 LLM4SA 的 88--93%），在 D2A 数据集上将标签准确率从 53% 提升到 86%，真实项目中消除超过 85% 的误报。每告警仅平均消耗 912 tokens、耗时 4.7 秒，使用开源的 Qwen2.5-32B 本地部署——这是该方向最早证明"低成本+本地部署"可行性的工作之一。

**ZeroFalse（Iranmanesh 2025）** 走的是"结构化上下文+CWE 专属提示"路线。它从 CodeQL 输出的 SARIF 中提取完整的数据流路径、CWE 元数据和代码切片，然后为每个 CWE 类别构建专属的微判据（micro-rubric）——一套 10--20 条声明式规则，列举高风险模式、安全惯用法和常被误认为消毒的操作。最佳模型 gpt-5 在自建真实项目集 OpenVuln 上达到 F1=0.955（Precision=1.0, Recall=0.914），Grok-4 在 OWASP Benchmark 上达到 F1=0.912。核心发现是：**CWE 专属提示远好于通用提示**，而推理型模型（gpt-5、grok-4）在实际代码上的泛化性远优于纯大参数模型（Gemini 2.5 Pro 在 OpenVuln 上 F1 从 OWASP 的 0.910 暴跌至 0.372）。

**AdaTaint（Lin 2025）** 走的是"神经符号融合"路线。它不仅消减误报，还让 LLM 自动推断 SAST 所需的 source/sink 规范。LLM 提建议，符号约束检查做裁决——LLM 只做局部语义推理，最终判定由确定性规则把关。在 Juliet、SV-COMP 和真实项目（Apache HTTPD、Node.js）上，AdaTaint 的精确率 84.3%、F1 79.6%、FPR 17.5%，相比基线静态分析减误报 43.7%、召回提升 11.2%。消融实验显示，去掉自适应 source/sink 推断后召回下降 14 个百分点——说明规范推断是整个流程的基石。

### 6.2.3 智能体框架：从一次性判官到主动调查者

静态提示的局限在于 LLM 只能看到传入的固定上下文，无法主动探索代码库。智能体（Agent）框架让 LLM 拥有了反复读代码、查文档、搜索相关文件的工具，能够像人类审计员一样主动收集证据。

**Sifting the Noise（Xiong & Zhang 2026）** 是首个系统比较多种 LLM 智能体框架在 SAST 误报消减上效果的工作。它对比了三种代表性框架（Aider、OpenHands、SWE-agent）和三种骨干模型（Claude Sonnet 4、GPT-5、DeepSeek Chat）。最佳配置（SWE-agent + Claude Sonnet 4）将 OWASP Benchmark 的初始误报率从 98.3% 降至 6.3%，消减了 92.1% 的误报。但论文最重要的贡献不是先进的结果，而是首次系统量化了**智能体消减误报的健全性代价**：即使是最佳配置，也有 **22.25% 的真实漏洞被智能体误判为误报而屏蔽**，其中密码学类（CWE-327：77.17% miss rate）和策略类（CWE-501：77.11%）尤为严重。论文总结道：智能体不适合无条件自动屏蔽，应作为决策辅助工具。这一发现直接暴露了无约束智能体方案在安全性上的根本缺陷。

**QASecClaw（Ameen 2026）** 采用了多智能体架构：SAST 过滤智能体、测试计划智能体、证据关联智能体、报告生成智能体，由 Mission Orchestrator 统一调度。它有一个务实的工程设计——**fail-open 策略**：如果 LLM 调用失败/超时/返回格式错误，不做任何过滤，保留原始告警。在 OWASP Benchmark 上，F1 从 Semgrep 的 78.39% 提升到 90.93%，误报从 560 降至 64（减少 88.6%），召回仅下降 3.1%。fail-open 策略让 LLM 异常不会造成漏报，但正常推理中 3.1% 的 TP→FN 率仍然存在。

**CodeCureAgent（Joos 2026）** 更进一步，实现了"分类→修复→验证"的端到端闭环。它用 Classification Sub-Agent 判断告警是 TP 还是 FP（FP 直接压制，TP 送入修复），Repair Sub-Agent 生成修复代码，最后经过三层验证（编译通过 → 原告警消失且不新增告警 → 测试套件通过）。在 106 个 Java 项目的 1000 条 SonarQube 告警上，整体分类准确率 91.8%（TP 97.4%，FP 81.0%），正确修复率 86.3%，平均每条告警 2.9 美分、4.4 分钟。但 FP 分类准确率仅 81.0%，意味着 19% 的 FP 被误判为 TP 导致不必要的修复尝试。且 SonarQube 告警偏向代码规范（code smell）而非安全漏洞——安全类仅占 3.1%。

### 6.2.4 路径可行性分析：用形式化工具兜底 LLM 的不确定性

智能体框架本质上仍然信任 LLM 的语义判断。与之不同，**LLM4PFA（Du 2025）** 将 LLM 的角色严格限定在局部推理，最终判决交由确定性 SMT 求解器（Z3）完成。

LLM4PFA 的工作流程是：给定 SAST 报告的 source→sink function call trace，逐函数做两阶段分析——(1) 提取关键路径条件分支（哪些 if/while 分支影响 sink 可达性），(2) 用 LLM Agent 做变量符号区间推理，然后将约束转化为 Z3 能读的 Python 脚本，由 Z3 判断路径是否可满足。如果任何函数返回 UNSAT（不可行），整条告警即为误报。

结果在 1800 万行 Linux Kernel、OpenSSL、Libav 上验证：过滤掉 72%--96% 的误报（按 bug 类型分），且 45 个真实 bug 中仅漏掉 3 个（Recall=0.93）。相比 LLM4SA 好 41.1%，相比 LLMDFA 好 105.7%（FPR_R 指标）。

**LLM4PFA 在健全性维度上是该子方向中位置最高的论文之一**，因为 LLM 不做最终判定——Z3 的确定性求解为"零漏报"提供了硬约束，LLM Agent 的错误会被 SMT 求解器捕获。但代价是适用面窄：仅限可以用 SMT 表达的条件约束，目前仅覆盖 C/C++ 的 NPD/UAF/BOF 三类内存漏洞。

### 6.2.5 保守策略：零漏报作为硬约束

与 LLM4PFA 的"Z3 兜底"思路不同，**保守分析（Wagner 2025）** 从问题定义层面将"零新增漏报"（TPR=100%）设为硬约束——它宁可放过误报不删，也绝不以漏掉真漏洞为代价换取高消减率。

论文系统评估了约 20 个 LLM 在 OWASP Benchmark 上带零漏报约束的误报消减能力。核心机制是 Chain-of-Thought 分步推理 + Self-Consistency（5 遍取共识）+ 阈值决策：LLM 对每条告警打 0--10 分（10=肯定真漏洞），分数低于阈值才标为误报。阈值在有标注数据上反推得到，目标是"找到零漏报前提下能删最多误报的最高阈值"。

**6 个模型（GPT-4o、Phi-4、Qwen2.5-32B、Qwen2.5-72B、Phi-3 Medium 128k、Qwen2.5-Coder-7B）在 OWASP 上实现了零漏报前提下的误报消减。** 最佳单模型（GPT-4o 或 Qwen2.5-32B）消减 62.5% 误报（TPR=100%, F2=96.8%, Precision=87.8%）；3 模型集成将消减率拉至 78.9%（F2=98.1%, Precision=91.3%）。

**最反直觉的发现是个头最大的模型（Llama 3.1 70B）反而做不到保守**——它误报消减最狠（86.7%），但误删了真漏洞，直接出局。这揭示了一个关键洞见：**"零漏报"是一个独立于"检出能力"的评估维度，不能从模型规模或通用能力推断。**

但保守分析也有致命局限：阈值依赖标注数据回推，在无标注的真实生产环境中无法保证保守。从 OWASP 到真实项目（Mnestix）时，最佳单模型 Phi-4 的消减率从 62.5% 大幅缩水至 33.85%。GPT-4o 在真实数据上甚至漏掉了一个真漏洞，失去保守资格。

### 6.2.6 离线与小模型突破

前述方法大多依赖 GPT-4 或 Claude Sonnet 级别的云端大模型。**SecureFixAgent（2025）** 和 **SAST-Genius（Agrawal 2025）** 分别从不同角度证明了本地小模型的可行性。

**SecureFixAgent** 全部使用 <8B 的本地模型（DeepSeek Coder 1.3B/6.7B、Qwen2.5-Coder 3B/7B、CodeLlama 7B、CodeGemma 7B），通过 LoRA 微调 + 迭代检测-修复-验证循环处理 SAST（Bandit）告警。最佳配置（微调 + 迭代）的修复准确率 87.83%，假阳性率从 Bandit 的 18.91% 降至 8.11%，开发者对解释质量评分为 4.5/5。迭代验证使准确率从单次的 74.32% 提升至 87.83%，证明"自验证"机制对小模型尤为关键。局限是仅限 Python，且未报告 LLM 判断是否引入了漏报。

**SAST-Genius** 构建了 Semgrep + 微调 Llama 3 8B 的混合管线，精确率从 Semgrep 的 35.7% 提升到 89.5%，F1=94.5%，召回在 170 个 ground truth 上声称 100%。它额外做了 PoC 生成（~70% 成功率）和修复建议输出。但论文质量偏低（无 arXiv ID、方法细节不透明），召回 100% 的声称有误导性——验证集仅包含"已确认"的漏洞，不包含未知漏洞。尽管有这些局限，SAST-Genius 依然是方向 3 中为数不多同时验证了"小模型+真实项目+多语言"的工作。

## 6.3 健全性代价的横向对比（纵轴分析）

这是本综述双轴框架在方向 3 上的核心应用。以下按健全性代价从高到低排列：

| 论文 | 纵轴坐标 | 漏报量化 | 核心机制 |
|------|---------|---------|---------|
| **保守分析** | D3● 零漏报 | TPR=100% 硬约束 | CoT+SC+阈值, 阈值依赖标注反推 |
| **LLM4PFA** | D2● 极少量漏报 | Recall=0.93（3/45） | LLM 局部推理 + Z3 确定性求解 |
| **QASecClaw** | ◐ 部分保守 | Recall 降 3.1%（40/1273） | Fail-open 兜底 LLM 异常, 正常 TP→FN 仍存在 |
| **LLM4FPM** | ◐ 未保证但 F1>99% | 未单独报告 FN 率 | 上下文精度提升降低概率 |
| **ZeroFalse** | ◐ 设计偏保守 | 未单独报告 FN 率 | Precision 极高 (≈1.0), Confidence 字段留复审 |
| **CodeCureAgent** | ◐ 部分保证 | FP 分类准确率仅 81.0% | 三层验证: 编译→SA 重扫→测试 |
| **AdaTaint** | 只剪误报 | 未报告 FN 率 | 符号约束检查默认, 无 fail-safe |
| **SAST-Genius** | 只剪误报 | 未报告 FN 率, 声称 Recall 100% | 无零漏报设计 |
| **SecureFixAgent** | 只剪误报 | 未报告 FN 率 | Bandit 重扫验证, 但仅验证修复 |
| **Sifting** | 只剪误报 | TP retention 77.7%（FN 22.25%） | 无约束智能体, 激进消减 |
| **LLM4SA** | 只剪误报 | 未报告 FN 率 | 静态提示, 无任何安全兜底 |

**关键观察**：11 篇论文中仅 2 篇（保守分析、LLM4PFA）提供了接近零漏报的设计，其余 9 篇均未系统量化或控制健全性代价。这是本综述认为最值得关注的系统性空白。

## 6.5 共同局限

尽管方向 3 在 2024-2026 年间取得了显著进展，所有 11 篇论文共享以下局限：

1. **均不处理 Java 动态分发导致的调用图断裂**。无论使用什么 LLM 策略（静态提示、智能体还是路径可行性），所有工作都默认静态分析传过来的告警和上下文是"完整的"。但如果告警路径涉及反射、Spring DI、SPI、反序列化等动态分发机制——这些在 Java 企业系统中极为普遍——SAST 自身的告警就建立在断裂的调用图上，LLM 的二审再精确也无法弥补这个根本缺陷。这是议程 1（带健全性保证的 SCA 可达性验证）和议程 4（跨语言/跨框架动态分发语义建模）的核心动机。

2. **大部分依赖云端大模型**。11 篇中只有 LLM4FPM、SecureFixAgent、SAST-Genius 实现了完全本地小模型部署，其余 8 篇的最佳结果依赖 GPT-4/Claude/Grok 等云端 API。这直接验证了议程 2（离线/小模型可行性研究）的必要性。

3. **几乎无人系统报告 FN 率**。11 篇中仅保守分析（TPR=100% 硬约束）、LLM4PFA（Recall=0.93）、Sifting（TP retention=77.7%）和 QASecClaw（召回下降 3.1%）报告了消减误报的同时引入了多少漏报。其余 7 篇只报 Precision/F1——这正好是双轴纵轴盲区的实证。议程 3（统一健全性评测框架）正是为解决此空白而设。

4. **缺乏动态验证**。所有方法均为纯静态分析，依赖源代码/字节码层面的推理，没有运行时插桩、fuzzing 或 PoC 执行来确认漏洞是否真实可利用。这限制了误报消减结论的可靠性——LLM 判定"不可利用"的告警，是否真在运行时不可利用，无人验证。

5. **CWE 覆盖不平衡**。注入类（SQL/XSS/命令注入）被几乎所有方法处理得很好（miss rate < 3%），但密码学类（CWE-327/328）和策略类（CWE-501/614）在所有方法上表现一致差。Sifting 报告的 CWE-327 miss rate 高达 77.17%，QASecClaw 中 CWE-501 的 F1 比 Semgrep 基线反而下降 22.8%。这暗示某些 CWE 类型可能不适合纯静态的 LLM 过滤，需要结合动态验证。

## 6.6 与双轴框架的关系（本章小结）

方向 3 在双轴矩阵中的分布高度集中于**横轴 B（模式不精确/误报）**，且集中在**纵轴下半部分（只剪误报，无零漏报保证）**。仅保守分析和 LLM4PFA 触及纵轴顶部（D3● 或接近零漏报），但两者都牺牲了适用面（保守分析缺乏真实数据保守性保证、LLM4PFA 仅限 C/C++ 内存漏洞）。D1（动态分发）和 D2（离线小模型）在方向 3 中几乎完全空白——11 篇中只有 SecureFixAgent 和 SAST-Genius 聚焦 D2，D1 零篇。

这暴露了双轴矩阵右下象限（D1● + D2● + D3● 三重空白）——即"同时处理 Java 动态分发、支持离线小模型、且保证零漏报"的方向 3 工作完全不存在。这直接导入研究议程 1 和议程 2 的联合攻关需求。
