# 方向⑥ — 评测与数据集

## 章节概述

方向⑥ 涵盖评测基准与数据集的 7 篇代表性工作，它们共同构成对前五个方向方法论有效性的检验基础设施。本章的核心叙事线是：数据集质量危机（Croft 揭示 20-71% 标签不准、17-99% 数据重复）→ 项目级实证揭露 LLM 不可靠（ProjectScale 发现 Java 场景平均召回率仅约 34%、误发现率约 85%）→ 新基准试图填补空白（TOSSS 从代码选择角度、RealSec-bench 从仓库级生成角度）。然而所有数据集均不标注可达性信息，且缺乏面向 Java SCA 可达性验证的专用评测基准——这是本综述六方向中共同的基础设施空白。

## 9.1 数据集质量危机

### 9.1.1 Croft：数据质量的系统性审查

Croft et al. (2023, DOI:10.1109/ICSE48619.2023.00022) 是这一子方向的奠基性实证研究。它对四个主流漏洞数据集（Devign、ReVeal、BigVul、CVEfixes）进行了系统性的数据质量审查，发现了一系列严重问题：

- **标签不准确率**：20%–71% 的漏洞标签不准确——自动标注（基于 VFC/Vulnerability-Fixing Commit 的启发式规则）引入了大量噪音
- **数据重复率**：17%–99% 的数据存在重复——同一漏洞的不同提交在不同版本间重复出现
- **不一致标签**：最高 47% 的数据集内标签不一致
- **去重后性能下降**：下游模型的 F1 分数最高下降 82%，说明此前报告的"高指标"可能来自数据泄露而非真正的学习能力

Croft 定义了基于 ISO/IEC 25012 的五维数据质量框架（准确性、唯一性、一致性、完整性、时效性），并给出了可重复的测量方法。其核心结论是：**系统性数据质量评估应成为漏洞检测研究的标配环节**——这一建议在 2023–2026 年间正逐渐成为领域共识。

Croft 的发现对理解方向③（LLM 消减误报）和方向①（LLM 漏洞检测综述）中报告的高指标具有重要警示意义：如果训练集和测试集都存在标签噪声，那么报告的 P/R/F1 可能系统性地偏离真实性能。

### 9.1.2 DiverseVul：规模和多样性的里程碑

DiverseVul (Chen et al., 2023, arXiv:2304.00409) 在当时发布了最大、最多样化的 C/C++ 漏洞数据集：18,945 个漏洞函数、覆盖 150 种 CWE 类型、来源项目数超过此前所有数据集的总和。作者系统测试了 11 种模型架构（含 GNN、RoBERTa、GPT-2、T5 四类），得出两个关键结论：

第一，**代码预训练比模型大小更重要**——CodeT5 等模型的代码专项预训练（预测变量名/函数名）比通用的更大规模模型对漏洞检测更有效。第二，**即便使用最优配置，最佳模型 F1 仅为 47.2%，误报率 3.5%**——"每 100 个函数就有约 3.5 个误报"在千万级项目的实践中仍然过高。更严峻的是**未见项目泛化测试**——当测试集与训练集按项目划分（而非随机划分）时，F1 从 49% 暴跌至 9.4%。这一数据直接支撑了"深度学习在漏洞检测上尚未准备好"的论断。

### 9.1.3 CleanVul：LLM 辅助数据清洗

CleanVul (Li et al., 2024, arXiv:2411.17274) 将 LLM 应用于数据清洗本身，回应了 Croft 提出的数据质量问题。其核心组件 VulSifter 组合 LLM 语义理解和启发式规则，从 VFC 中自动筛选真正的漏洞修复改动——排除测试改进、重构、文档更新等无关改动。

VulSifter（GPT-4）在清洗任务上的 F1 达 0.82，最终产出的 CleanVul 数据集包含 8,198 个函数，正确率 90.6%（达到人工校验水平）。一个有趣的交叉验证结果是：**在 CleanVul 上训练的模型在 PrimeVul 上测试的 F1（58.09%）甚至高于在 PrimeVul 自身训练集上训练的模型在 PrimeVul 测试集上的 F1（56.61%）**——数据质量的提升可以弥合模型与训练数据之间的分布偏移。CleanVul 覆盖 Java、Python、JavaScript、C#、C、C++ 六种语言，且不依赖 NVD 条目，适用性更广。

## 9.2 LLM 漏洞检测能力的实证评估

### 9.2.1 Eval-Line：行级定位的系统评测

Zhang et al. (2024, arXiv:2404.00287) 进行了首个对 LLM 行级漏洞定位（Automated Vulnerability Localization, AVL）的大规模系统评估，覆盖 19 个 LLM（参数范围从 60M 到 70B），涵盖三种架构和三种范式（few-shot 提示、判别式微调、生成式微调）。

关键发现包括：(i) 判别式微调效果最佳，F1 达 63.8%，大幅超越此前所有 AVL 方法；(ii) 但"未见项目"泛化仍然很差——所有模型在经过时间切分的 BV-LOC-LF 基准（377 个 cutoff 后披露的漏洞）上精度骤降，表明 LLM 对不熟悉的词法和结构模式脆弱；(iii) 滑动窗口策略可缓解输入长度限制，最多带来 F1 提升 29.7%。

值得注意的是，该评测中的最佳方法（判别式微调）本质上是将 LLM 作为序列标注器使用，而非利用其推理能力。这提示我们：LLM 在漏洞定位上的优势可能更多来自大规模代码预训练学到的语义表示，而非其对话或推理能力。

### 9.2.2 ProjectScale：项目级的警告

ProjectScale (Li et al., 2026, arXiv:2601.19239) 是本综述涵盖范围内最直接的"LLM 不可靠"实证研究。它将 5 个最新的 LLM 漏洞检测工具（RepoAudit、Claude-3.5-Sonnet、IRIS、LLMDFA、INFERROI）与 2 个传统静态分析工具（CodeQL、Semgrep）放在 24 个真实开源项目上做了 head-to-head 对比，耗时 150+ 人工小时验证 385 个抽样报告。

核心数字如下：
- **召回率**：C/C++ 平均 21.09%（27/128），Java 平均 33.82%（69/204）
- **误发现率（SFDR）**：表现最优的工具平均仍高达 85.3%（RepoAudit 在 C/C++ 上甚至达到 97.0%）
- **token 消耗**：最高 225M 输入 token + 38M 输出 token（单个项目），运行时间最长超过 3 天

ProjectScale 还对 363 个误报样本做了根因分类，前三位是：(A1) 浅层过程间推理（37.47%，如 RepoAudit 默认只探索 3 层调用链）、(B1) 不精确的 Source/Sink 识别（19.00%）、(D1) 遗漏关键程序点（12.67%）。

该研究对本综述的特殊价值在于：(i) **嵌入 Java 场景数据**——Java 召回率虽高于 C/C++（33.82% vs 21.09%），但距实用水平仍远；(ii) **佐证了"静态+LLM 互补"的必要性**——纯 LLM 方法（LLMDFA）在 source/sink 推断上的不可靠性（42/44 漏报源于 source-sink 不匹配）强烈支持了"静态分析提供结构、LLM 提供语义"的范式；(iii) 间接暴露了 Java 动态分发问题——A2（不正确的控制流）的示例涉及多实现函数的分发问题。

## 9.3 新一代基准

### 9.3.1 TOSSS：从"生成"到"选择"的范式转换

TOSSS (Damie et al., 2026, arXiv:2603.10969) 在基准构建思路上做出了独创性贡献。与所有现有基准不同，TOSSS 不是让 LLM 生成代码然后检测是否安全，而是**给 LLM 两个实现相同功能的代码片段（一个有漏洞、一个安全），让 LLM 选择哪个更安全**。

这一设计有两大优势：首先，基准构建可以完全自动化——从 CVE 数据库挖掘漏洞修复前后的代码对（通过 MegaVul 流水线），新漏洞可随时加入，不依赖静态分析工具的规则更新。其次，二选一设计消除了生成质量、编译成功率等混杂因素，直接测量 LLM 的安全辨识能力。

在 14 个模型（含 GPT-5.4、Claude Opus 4.6、GLM-5、DeepSeek V3.2 等）上，TOSSS 分数从 0.48 到 0.89 不等（0.5 = 随机）。关键发现：
- **最优模型**：GLM-5（0.878）> GPT-5.4（0.866）> Claude Opus 4.6（0.864）
- **代码模型不占优**：Codestral 2508（0.680）和 Qwen3 Coder Next（0.644）排名靠后——"专精代码生成≠安全能力强"
- **安全提示增益有限**：加入"请选择最安全的实现"后平均仅提升 0.021；Codestral 反而下降 0.072
- 最差模型 Mistral Large 2512（0.480）与随机猜测无异

TOSSS 对本综述的独特价值在于：它为"安全是独立能力维度"提供了量化证据，直接支持议程 2（离线/小模型可行性）中"能力差距"论据的建立。同时，其代码选择范式为评测基准设计提供了新思路——但 TOSSS 的粒度限于函数片段，不涉及仓库级上下文或可达性信息。

### 9.3.2 RealSec-bench：Java 仓库级安全代码生成基准

RealSec-bench (Wang et al., 2026, arXiv:2601.22706) 是本综述覆盖范围内最接近 Java SCA 评测需求的基准。它从 GitHub 最热门的 4,000 个 Java 项目中筛选出 105 个精心构建的任务（覆盖 19 种 CWE），要求 LLM 重写有漏洞的函数，且必须同时通过功能测试和安全检测（SecurePass@1）才算合格。

结果极具警示性：**5 个最先进模型的 SecurePass@1 全部低于 6%**（最优 GPT-4.1 和 Qwen3-235B 均为 5.71%）。密码学类任务 SecurePass@1 = 0.00%。模型能写出功能正确的代码，但几乎一定会引入安全漏洞。

RealSec-bench 还报告了几个重要的消融发现：(i) RAG（BM25/Dense/Dataflow）显著提升功能正确性（Pass@1 从 16.19% 到 22.86%），但对安全（SecurePass@1）几乎无增益——精确的静态分析路径不仅不提升安全性，还可能因为"过于聚焦漏洞路径而丢失功能上下文"（Dataflow 检索器表现最差）；(ii) 加入安全规范后 Claude 的功能正确率从 16.19% 暴跌至 9.52%——安全和功能在当前模型上存在明确的 trade-off。

RealSec-bench 的 34-hop 跨过程依赖标注（跨过程跳数分布：0-hop 35.2%、1-hop 23.8%、2-hop 13.3%、3-hop 6.7%、>3-hop 21.0%）为 Java 项目的过程间复杂性提供了重要的量化标杆。但**该基准标注的是"是否存在漏洞"而非"是否可达"**——这是其与本综述目标之间的关键差距。

## 9.4 对比分析与核心发现

### 9.4.1 对比表

| 论文 | 类型 | 语言 | 粒度 | 规模 | 可达性标注 | 核心结论 |
|------|------|------|------|------|:---------:|---------|
| Croft (2023) | 数据质量审查 | C/C++ | 函数级 | 4 数据集 | 否 | 20-71% 标签不准, 17-99% 重复 |
| DiverseVul (2023) | 数据集 | C/C++ | 函数级 | 18,945 vuln fn | 否 | 最佳 F1 47.2%, 未见项目 F1 仅 9.4% |
| CleanVul (2024) | 数据集+清洗 | 6 语言 | 函数级 | 8,198 fn | 否 | 正确率 90.6%, LLM 清洗可行 |
| Eval-Line (2024) | 评测 | C/C++ | 行级 | 19 模型 | 否 | 判别微调最佳 F1 63.8% |
| ProjectScale (2026) | 实证研究 | C/C++/Java | 项目级 | 5 LLM + 2 SAST | 否 | Java 召回 ~34%, FDR ~85% |
| TOSSS (2026) | 基准 | C/C++/Java | 函数片段 | 14 模型 | 否 | 分数范围 0.48-0.89 |
| RealSec-bench (2026) | 基准 | Java | 函数级 | 105 实例/19 CWE | 否 | SecurePass@1 < 6% |

### 9.4.2 核心发现

**发现一：数据质量是贯穿始终的基础性挑战。** 从 Croft (2023) 揭示的 20-71% 标签不准，到 ProjectScale (2026) 发现的 150+ 人工小时验证中 LLM 工具平均 FDR ~85%，数据质量问题不是孤立的"数据清洗"子问题，而是直接影响方法论评估可靠性的基础性障碍。CleanVul 的工作证明了 LLM 可以辅助数据清洗（正确率从约 80% 提升到 90.6%），但改进空间仍然巨大——90.6% 的标签正确率意味着在真实项目中仍有约 10% 的误标注。

**发现二：LLM 的 Java 能力显著低于 C/C++。** ProjectScale 的实证跨语言对比（C/C++ 召回 21% vs Java 召回 34%）表面上看 Java 更高，但考虑到 Java 项目的规模更大、框架更复杂，这一对比反而暗示了 Java 场景下 LLM 检测的独特挑战。TOSSS 在 C/C++ 和 Java 上的得分接近（0.756 vs 0.740），但 RealSec-bench 对 Java 安全代码生成能力的评估（SecurePass@1 < 6%）是最低的——生成比识别更难。综合三项研究，**LLM 对 Java 安全任务的适用性远低于文献中常见的 C/C++ 评测数字所暗示的水平**。

**发现三：所有数据集均不标注可达性信息。** 这是方向⑥ 最突出的结构性缺陷。无论是函数级的 CleanVul/DiverseVul/TOSSS，还是项目级的 ProjectScale/RealSec-bench，标注的都是"该函数/代码片段是否存在漏洞"，而非"该漏洞在给定的调用上下文中是否可达"。对于 SCA 场景而言，这两个问题是正交的——一个函数在孤立状态下存在漏洞，在实际系统的调用图中可能永远不被触发。**缺乏可达性标注意味着无法区分"漏洞存在"和"漏洞可达"**——这是本综述议程 1（SCA 可达性验证）和议程 3（统一评测框架）的核心动机。

**发现四：缺乏 Java SCA 可达性专用评测基准。** RealSec-bench 是最接近的——使用真实 Java 仓库、最多 34-hop 跨过程依赖——但仍标注"是否存在漏洞"而非"是否可达"。构建一个含动态分发真值标注（按反射/Spring/SPI/反序列化四种机制分层）、同时标注调用图可达性的 Java SCA 评测数据集，是推动方向②（SCA 可达性）研究的必要基础设施条件。

**发现五：TOSSS 的"安全是独立能力维度"发现具有跨方向意义。** 代码生成最强的模型（Codestral、Qwen3 Coder）在安全选择任务上表现倒数——这一发现超出 TOSSS 自身范围，对议程 2（离线/小模型可行性）具有直接含义：小模型在编程基准上的表现不能直接外推为安全任务上的能力。

## 9.5 与议程的映射

方向⑥ 的基础设施角色使其与所有五个研究议程直接相关：

**议程 1（SCA 可达性 + 健全性）**: 需要同时满足两个标注维度——CVE 漏洞存在性 + 调用图可达性。目前无数据集提供后者。RealSec-bench 的 34-hop 跨过程依赖标注可作为路线参考。

**议程 2（离线/小模型）**: TOSSS 的 14 模型跨模型评测（0.48-0.89）和 ProjectScale 的计算成本数据（最高 225M token + 4,638 分钟/项目）为模型选型和成本建模提供了定量参考。

**议程 3（统一评测框架）**: Croft 的数据质量框架和 ProjectScale 的误报根因分类法（A1/B1/D1/A2/C1）可直接作为规范化评测框架的方法论基础。

**议程 4（跨语言动态分发）**: TOSSS 的 C/C++/Java 双语评测和 RealSec-bench 的 Java 专用评测为跨语言能力对比提供了数据基础。

**议程 5（证据链合规）**: 本节工作为"可验证的 benchmark"提供方法论——CVE-Genie（方向⑤）的 CTF Verifier 和 FORGE 的 Oracle 裁决可视为证据链验证的技术参考。
