# 分类体系：双轴框架

> 文献综述章节 · 定义双轴 + 子方向分类 + 矩阵框架

---

## 2.1 现有分类法的局限

现有综述 [sheng2025, basic2024, wang2025] 采用传统的任务分类法（检测 vs 修复 vs 评估）或方法分类法（静态 vs 动态 vs 混合）。这些分类虽然便于梳理文献，但无法回答以下三个关键问题：

1. **LLM 到底补了静态分析的哪个短板？** 同样是"LLM + 静态分析"，IRIS 补的是污点规范缺失，SemTaint 补的是调用图断裂，Sifting 补的是误报模式不精确——它们解决的问题完全不同。
2. **补了这个短板后，安全性是否有保障？** LLM 消减了误报——但有没有引入新漏报？这个代价是否可量化？是否可控？
3. **这个方案离真实部署有多远？** 是否需要完整编译？是否依赖云端大模型？输出能否审计？

本章提出双轴框架（§2.2–2.3）和部署就绪度评分（§2.4），系统地回答这三个问题。

## 2.2 横轴：LLM 补偿静态分析的哪个短板

| 横轴坐标 | 说明 | 代表论文 |
|---------|------|---------|
| **A. 调用图不完整（可达性）** | 静态分析无法构建完整调用图（反射/DI/动态加载），导致数据流路径断裂 | SemTaint (D1●), IRIS, Boosting-Pointer |
| **B. 模式不精确（误报）** | 静态规则过于宽泛，大量告警实际上是安全的（FP），需要语义理解区分 | 保守分析 (D3●), Sifting, LLM4PFA, ZeroFalse |
| **C. 污点规范缺失（source-sink）** | 静态工具需要人工标注哪些 API 是入口/出口，规范不全导致漏检 | IRIS, AdaTaint |
| **D. 版本与可达性的鸿沟** | SCA 只做版本比对，不判断 CVE sink 在应用中是否调用图可达 | VFArche, VERCATION, SAVANT |
| **E. 数据与标注质量** | 训练/评测数据集存在标签错误、重复、不标注可达性等问题 | Croft, CleanVul, ProjectScale |

**关键观察**：一篇论文可能覆盖多个横轴坐标（如 IRIS 同时补 B 和 C）。横轴不是互斥分类，而是能力维度——论文可以在多个维度上贡献。

## 2.3 纵轴：补偿对健全性（Soundness）的代价

**此维度是本综述首次系统化**。现有综述未将健全性作为评价维度，而本综述认为：在安全领域，漏报（FN）的代价通常远高于误报（FP）。一个自动屏蔽系统如果存在不可控的漏报风险，其部署价值极为有限。

| 纵轴坐标 | 说明 | 代表论文 |
|---------|------|---------|
| **1. 只剪误报（可能引入漏报）** | 论文只报告 FP 消减率，未量化 FN 率。LLM 可能错误地将真实漏洞判为误报并屏蔽 | Sifting (FN 22.25%), IRIS, 大部分方向③论文 |
| **2. 只补漏报** | 论文用 LLM 扩展静态分析的检测范围（如推断新规范），但可能引入更多 FP | Vul-RAG (知识级 RAG 扩充检测) |
| **3. 声称两者兼顾** | 论文声称同时改善 Precision 和 Recall，但无硬约束保证 | SemTaint, Bridging |
| **4. 零漏报保证（fail-safe）** | 论文设计了明确的保守策略保证不引入新漏报，或使用形式化工具兜底 | 保守分析 (D3●), LLM4PFA (Z3 兜底) |

**关键发现**：在 48 篇精读论文中，仅 2 篇（保守分析、LLM4PFA）提供接近"零漏报保证"的设计，其余 46 篇均未系统量化健全性代价。

## 2.4 部署就绪度评分维度

| 维度 | 评分标准 |
|------|---------|
| **可编译性要求** | ● = 不需编译 / ◐ = 部分需要 / ○ = 必须完整编译 |
| **云端大模型依赖** | ● = 纯离线可运行 / ◐ = 小模型可替代 / ○ = 必须云端大模型 |
| **可解释性** | ● = 输出结构化证据链 / ◐ = 有文本解释但不可验证 / ○ = 黑箱输出 |
| **真实代码库验证** | ● = 在真实项目上验证 / ◐ = 混合 / ○ = 仅合成数据 |
