# 方向①: LLM 漏洞检测综述

## 1.1 本章定位

LLM 漏洞检测是近三年来增长最快、论文产出最密集的研究方向。本章覆盖三篇系统性综述（2024--2026）和七篇检测增强方法，核心论点是：已有综述虽然系统梳理了大量工作，但均未将"可达性验证"作为独立分析维度、未讨论健全性代价（soundness cost），也未覆盖离线部署约束——这三个盲区正是本综述双轴框架的直接驱动力。

## 1.2 三篇综述的覆盖范围与盲区

### 1.2.1 Basic et al. (2024): 规模最大的 SLR

Basic 等发表的《From Vulnerabilities to Remediation: A Systematic Literature Review of LLMs in Code Security》是目前纳入论文最多的综述，从 7008 条记录中筛选出 102 篇（2021--2026 年初），覆盖 LLM 产生漏洞、检测漏洞、修复漏洞及数据投毒影响四个维度 [arXiv: 2502.07049]。

**关键发现**: (1) LLM 检测能力常超越传统 SAST，但误报率偏高（34--97%），而 SAST 的误报率在 25--82% 之间；(2) LLM 与 SAST 结合效果显著，如 IRIS 使检测率提升 35% 同时误报消减 80%；(3) 最佳检测 F1 在微调后可达到 84--99%，CodeLlama-7B 和 GPT-4o-mini 均可达到 97%；(4) LLM 生成的代码最常出现注入类漏洞，Copilot 生成的不安全代码率约 40%。

**本综述的关注点**: 该综述虽系统性强，但未将"可达性"作为独立维度——文中讨论的漏洞检测大多停留在函数级或片段级判定，未区分"代码本身有漏洞"与"漏洞在应用上下文中是否可达"这两个问题。

### 1.2.2 Sheng et al. (2025): 聚焦漏洞检测技术

Sheng 等的《LLMs in Software Security: A Survey of Vulnerability Detection Techniques and Insights》聚焦 58 篇高质量论文，从 LLM 选型、数据集、评估指标、技术方法与挑战五个角度切入 [arXiv: 2412.15004]。

**关键发现**: (1) GPT-4 系列使用最多（29 次），C/C++ 占研究的一半以上（50%），Java 仅占 21.1%；(2) 约 83% 的研究只分析孤立代码片段（函数级或文件级），离仓库级检测尚有距离；(3) 最佳 F1 在 GPT-4 或 CodeLlama 经 PEFT 微调后可达约 0.9；(4) 数据质量差、复杂代码处理难、可解释性不足是三大主要障碍。

**本综述的关注点**: 该综述揭示了"83% 研究只做孤立片段分析"这一关键现象，但未进一步追问：即使检测出有漏洞的片段，在真实仓库的调用图中是否可达？这个"检测→可达性"之间的断裂带恰好是本综述核心议程的起点。

### 1.2.3 Wang et al. (2025): LLM 辅助程序分析

Wang 等的《A Contemporary Survey of Large Language Model Assisted Program Analysis》从静态分析、动态分析和混合方法三个维度分类，核心论点是 LLM 补充了传统程序分析的短板（误报高、路径爆炸、知识门槛高），但引入了幻觉、非确定性输出和 token 限制等新问题 [arXiv: 2502.18474]。

**关键发现**: (1) IRIS 用 LLM 推断污点规范 + CodeQL 做符号分析，检测增 35%、误报减 80%；(2) 渗透测试智能体 PenHeal 比 PentestGPT 检测覆盖增 31%、修复效果增 32%、成本降 46%；(3) 全自动渗透成功率仅 21%，半自动为 64%，说明人类介入仍然重要；(4) 提出"动态→静态"转换的新兴方向——用 LLM 在不运行代码时分析运行时行为。

**本综述的关注点**: 该综述对"LLM 增强静态分析"的技术分类最为详尽，但同样未系统讨论健全性代价——LLM 的"非确定性输出"对安全关键场景的接受度构成了根本性威胁，现有综述未将此作为独立评价维度。

### 1.2.4 三篇综述的共同盲区

将三篇综述的覆盖范围与本综述的双轴框架对比，可发现三个系统性盲区：

1. **可达性作为独立维度缺失**：三篇综述均在"检测是否准确"的框架下讨论，未区分"函数级漏洞检测"与"仓库级调用图可达性判定"是本质不同的两个问题。后者需要前者的输出（漏洞在哪个函数），但增加了"判断调用路径是否连通"的维度。
2. **健全性代价未被系统化**：无一篇综述将"LLM 检测引入多少漏报"作为独立评价指标。大量论文报告 P/R/F1，但几乎无人追问：消除误报的同时，是否以牺牲漏报为代价？
3. **离线部署约束未涉及**：三篇综述均未讨论 LLM 的部署方式（云端 API vs 本地小模型）对安全审计场景的适用性。对于等保合规、气隙网络等受控环境，这一约束直接决定了方法的可行性。

## 1.3 检测增强方法：从提示工程到知识增强

### 1.3.1 LLM 基线能力的"揭底"

Vul-RAG（arXiv: 2406.11147）率先做了一个"揭底"实验：在 PairVul 基准（586 对漏洞/修复代码对，覆盖 420 个 CVE 和 Top-10 CWE）上测试 GPT-4o、Claude 等前沿模型区分漏洞代码与修复后代码的能力。结果令人警醒——pair accuracy 仅 0.06--0.14，即 86%--94% 的案例无法区分。即使引入 CoT 和 CWE 增强提示词，最佳效果（CoT-1 + GPT-4o）也仅达到 0.20 [待核实]。

Vul-RAG 还发现了一个关键现象：**不稳定偏向（unstable bias）** — GPT-4o 在不同中性提示词下表现出系统性的判定偏差，basic prompt 下判 77% 为有漏洞，CoT-1 下仅判 48%。这一发现直接触及 LLM 作为安全判定工具的根本可靠性问题。

Beyond ChatGPT（arXiv: 2409.01001）从另一个角度验证了类似结论。作者测试了 6 个 LLM（GPT-3.5、GPT-4o、LLaMA-3-70B/8B、Gemma-7B、Mixtral-8x7B），发现在漏洞检测二分类任务上，最小的模型 Gemma-7B 反而超越了 GPT-3.5（准确率高 7.8%）。不同模型有各自的"独特正确预测"——投票组合多个 LLM 结果可提升 11%+。更值得关注的是"交叉验证"机制：用一个 LLM 验证另一个 LLM 的回答，效果甚至优于 6 个模型投票（GPT-4o 由 LLaMA-3-70B 验证，故障定位 Top-1 提升 16.2%）。

**启示**: LLM 在漏洞检测基线上的表现远不如预期。前沿模型区分漏洞/修复代码近乎随机，小模型反而在特定任务上超越大模型，模型间的多样性可被组合利用——这些发现共同指向一个结论：LLM 的漏洞检测能力被早期论文高估，需要更严格的评估框架。

### 1.3.2 知识增强路线：RAG 与多任务微调

Vul-RAG 提出的解决方案是**知识级 RAG（Knowledge-level RAG）**。不同于传统代码级 RAG（直接检索代码片段），Vul-RAG 从历史 CVE 中提取多维度漏洞知识（功能语义、漏洞原因、修复方案），检测时检索高层语义知识辅助 LLM 判断。pair accuracy 提升 16%--24%，平衡精确率提升 9%--14%，平衡召回率提升 7%--11%。更令人信服的是其实践验证：在 Linux 内核 v6.9.6 中发现了 10 个此前未知的 bug，其中 6 个已分配 CVE。Vul-RAG 还发现了一个附加价值：人类审计员使用 Vul-RAG 知识后，检测准确率从 60% 提升至 77%。

MSIVD（arXiv: 2406.05892）走的是**多任务自指导微调**路线。它在 CodeLlama-13B 上同时学习"有无漏洞 + 漏洞类型 + 原因 + 修复位置"四个任务，辅以 GNN 编码控制流图，在 BigVul 上 F1 达 0.92（超越 LineVul 的 0.81）。然而，关键发现是**数据泄露的严重程度**：CodeLlama 在 BigVul 上预训练就达 F1 0.74，但用截止日期后的新数据集 PreciseBugs 评估时骤降至 0.22——降幅达 70%。MSIVD 在 PreciseBugs 上 F1 为 0.48，LineVul 为 0.31。这组数据说明：现有基准测试可能严重高估 LLM 的泛化能力，**评估 LLM 必须用 cutoff 日期后的新数据**。

### 1.3.3 多智能体协作：从通用检测到定向审计

LLM-SmartAudit（arXiv: 2410.09381）将多智能体对话引入智能合约审计。BA（广泛分析）模式让 LLM 用通用安全知识扫描代码，GPT-3.5 的召回率就达 74%，超越传统工具 Mythril（54%）和 Slither（46%）。TA（定向分析）模式针对特定 CWE 类型构建约束条件，GPT-4o + TA 在 10 类漏洞中 7 类达 F1 95%+，特别是传统工具难以检测的 TOD（交易顺序依赖）从 0% 跃升至 94.7%。关键洞察是多智能体对话能缓解单一 LLM 的"退化推理（Degeneration-of-Thought）"问题——当 LLM 自反思时推理链条逐渐退化，多智能体协作相当于引入了外部校验点。

LLMVD.js（arXiv: 2604.20179）代表了更激进的"纯 LLM 智能体"路线：完全抛弃传统静态/动态分析引擎，让 LLM 自己完成污点分析、可利用性判定和 PoC 生成。在 517 个 npm 包的基准测试中，确认率 83.75%，远超传统工具（<22%）；在 260 个新包中挖出 36 个有效 PoC（传统工具最多 2 个）。平均每包 API 成本仅 $0.089。但 LLMVD.js 的成功有其特定边界——Node.js 生态 90% 的包代码量小于 32K tokens，这一结论能否推广到大型 Java 企业项目（动辄数十万行）尚待验证。其失效模式也值得关注：混淆（变量重命名+去注释）后，lodash 检测结果从 8 个 finding 归零。

## 1.4 共同局限

方向①的全部论文呈现出以下共同局限：

1. **无健全性保证**：所有方法均以精确率/召回率/F1 为评估指标，无人承诺"不产生新增漏报"或提供 fail-safe 兜底机制。对于安全审计等需要"宁可错杀不可放过"的场景，这一缺失是根本性的。

2. **仓库级可达性被回避**：绝大多数工作的评估样本是函数级或片段级的孤立代码，不涉及跨函数/跨文件的调用图可达性判定。83% 的研究（Sheng 等综述数据）停留在片段级，意味着大部分的"检测准确率"数字在仓库级场景下可能大幅缩水。

3. **离线部署缺乏关注**：全部方法在最佳配置下依赖 GPT-4 级云端大模型。Beyond ChatGPT 是唯一系统评估多模型（含开源小模型）的工作，但其结论却是"需要多模型投票/交叉验证"——这反而增加了部署复杂度。

4. **Java/企业框架未覆盖**：即使在 Java 相关的检测研究中，也极少涉及 Spring、反射、SPI、反序列化等企业级框架特有的动态分发问题。LLM-SmartAudit 的智能合约场景和 LLMVD.js 的 npm 场景距离企业 Java 系统有显著差距。
