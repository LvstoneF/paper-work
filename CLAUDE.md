## 项目是什么

写**一篇文献综述**(纯综述,**没有**自建方法,也**没有**自己的实验)。
主题:大语言模型(LLM)用于**开源组件 / 软件成分分析(SCA)安全、代码审计、漏洞可达性验证、误报消减**。
时间范围:2022–2026,主体 2024–2026。
作者视角(综述切入与侧重的镜头):Java/JSP 企业系统、离线内网、等保(MLPS 2.0)合规。

## 本综述自身的新意(核心贡献,务必贯穿全文)

**本综述的最终目标是推导出一份具体的、可执行的研究议程(Research Agenda)**。双轴框架和部署就绪度评分是推导工具,不是目的本身。审稿人看完应该能直接照着议程去立项。

叙事线:**现状盘点 → 双轴定位 → 发现矩阵空白 → 评估部署就绪度 → 推导研究议程**。前面所有章节都在为最后那份议程"攒证据"——不是我们凭空提的,而是数据/矩阵驱动的。

### 双轴框架(分析工具)

- **横轴 = LLM 在补静态分析的哪个短板**:调用图不完整(可达性)/ 模式不精确(误报)/ 污点规范缺失(source-sink)/ 版本与可达性的鸿沟 / 数据与标注质量。
- **纵轴 = 这种补偿对健全性(soundness)的代价**:只剪误报(可能引入漏报)/ 只补漏报 / 声称两者兼顾 / 有无"零漏报"保证。**此维度是现有综述的盲区,本综述首次系统化它。**
- **叠加部署就绪度评分**:可编译性要求、是否依赖云端大模型、可解释性、是否在真实代码库(而非合成数据)上验证。

### 研究议程(最终交付物,5 条,按优先级排序)

**议程 1(最高优先):带健全性保证的 SCA 可达性验证。**
- 问题:现有 SCA 只做版本比对或函数定位(横轴左半),无人做"已知 CVE sink 在应用中是否调用图可达"且同时保证零新增漏报(纵轴顶部)。
- 技术路线:静态调用图做主判 + LLM 只在动态分发断裂边做选择性补判 + 保守 fail-safe。
- 关键挑战:Java 框架(反射/Spring/SPI/反序列化)的动态分发让完整编译下的静态图依然断裂;LLM 判定不稳定需保守策略兜底。
- 评测:需同时报误报消减率和漏报率,用动态执行做真值。
- 预期影响:把 SCA 误报消减从"版本维度"推进到"可达性维度",且首次引入健全性约束。

**议程 2:面向真实部署的离线/小模型可行性研究。**
- 问题:现有工作几乎全部依赖 GPT-4 级云端大模型,离线/气隙/合规场景无法使用。
- 技术路线:量化"本地 7B 级小模型 vs 云端前沿模型"的能力差距;研究上下文压缩(CPG 切片/SARIF 轨迹)对小模型的增益;建立"模型规模-任务精度-部署约束"权衡曲线。
- 预期影响:为企业/政府/等保等受限环境给出可落地的模型选型指南。

**议程 3:统一的健全性评测框架。**
- 问题:现有论文只报 P/R/F1 或误报消减率,**几乎无人系统报告"引入了多少漏报"**——这是本综述双轴分析暴露的最大盲区。
- 技术路线:定义标准化指标体系(必须含漏报率/FN、按断图机制分层);建立含动态执行真值的 Java 基准(现有基准几乎全是 C/C++);要求论文同时报"消减了什么"和"代价是什么"。
- 预期影响:让该领域实验结果可比较、安全性可审计。

**议程 4:跨语言/跨框架的动态分发语义建模。**
- 问题:反射/DI/SPI 是 Java 特有表现,但"运行时动态分发让静态分析断裂"是跨语言共性(Python monkey-patching、JS 动态 require、Go interface)。
- 技术路线:抽象"动态分发边"统一表示,研究 LLM 跨语言迁移。
- 预期影响:把议程 1 从 Java 推广到多语言。

**议程 5:可解释证据链与合规对接。**
- 问题:LLM 判定是黑箱,等保/SOC2 等合规框架要求可追溯审计证据。
- 技术路线:让 LLM 输出结构化证据链(调用路径+判定依据),研究证据链的可验证性(人工复核成本、与动态执行结果的一致率)。
- 预期影响:打通"LLM 辅助安全分析 → 合规交付物"的最后一公里。

### 贡献总结

三重贡献 = 新分类法(双轴)+ 新评价维度(健全性代价,现有综述盲区)+ **证据驱动的 5 条研究议程**(核心交付物)。

## 综述结构(目标章节)

摘要 → 引言(动机+综述目标:推导研究议程)→ 分类体系(双轴定义+矩阵图)→ 各子方向逐节(每节:解决什么/主流方法/共同局限)→ 方法-效果对比表 → **综合分析(双轴矩阵定位+部署就绪度评分,暴露空白)** → 研究空白与挑战 → **研究议程(5 条,证据驱动,本文核心交付物)** → 结论。

注意:综合分析节必须有双轴矩阵图和就绪度表,让读者自己看到空白,议程从图表自然导出——"不是我们想做什么,而是数据告诉你该做什么"。

## 研究空白(gap 段)

现有工作多为 C/C++、函数/片段或版本级。共同空白:① 少有把 LLM 用于开源组件调用图可达性;② 不处理 Java 框架动态分发(反射/Spring/SPI/反序列化);③ 误报消减缺零漏报硬约束;④ 不针对离线/小模型部署;⑤ 少结合动态执行验证。

## 注意

- 这是**文献综述**,不做实验、不跑代码、不声称验证了什么方法。
- **但综述可以且必须提出研究议程**——议程是综述识别出的开放问题与路线图,是本文核心交付物。写法是"证据(双轴矩阵+就绪度)→ 空白 → 议程",不是凭空提出。
- 不要出现"我们的方法/我们的实验/我们的靶场"这类原创论文叙述。
- 早期生成的《科研执行手册》是为原创论文写的,综述忽略;《相关文献清单》《强相关论文深读》可取其"论文分析"部分复用。
