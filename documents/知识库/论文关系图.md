# 论文关系图

> 论文间的显式关系网络。每新增一篇精读，追加该论文的关系条目。
>
> 关系类型：
> - 🔗 **互补** — 解决同一问题的不同子问题（合在一起才是完整方案）
> - ⚔️ **竞争** — 同一问题的不同解法（方法不同，目标相同）
> - 🔧 **同方法论** — 共享核心技术栈（可以互相借鉴实现细节）
> - ➡️ **前后继** — 后者基于前者改进或受前者启发（技术演进脉络）

## 按脉络组织的关系链

### 脉络一：静态分析与 LLM 的关系从「替代」转向「互补」

```
早期 (2022-2023): 纯 LLM 检测器
  ├── fu:2023 (ChatGPT 直接判漏洞)
  ├── gao:2023 (LLM 漏洞检测能力评估)
  └── zhang:2023 (Prompt 增强检测)
        │
        ▼ 转向互补
中期 (2024-2025): 静态 + LLM 联合
  ├── VERCATION (切片 + LLM 精筛) ─── 🔧 同方法论 ─── LLM4SA (依赖遍历 + LLM)
  ├── IRIS (LLM 辅助静态分析推理)
  └── SAVANT (语义预处理 + LLM 上下文感知)
        │
        ▼ 实证否定纯 LLM
晚期 (2025-2026): 混合管线成为共识
  ├── SAST-Genius (Semgrep + LLM, FP ↓91%)
  ├── ProjectScale (实证: Java 召回仅 34%, FDR 85% → 纯 LLM 不可行)
  └── SemTaint (CodeQL + LLM 补全断边)
```

### 脉络二：调用图/CPG 成为 LLM 与程序分析的桥梁

```
传统程序分析
  ├── CodeQL (静态调用图)
  └── Joern (CPG: AST+CFG+DFG)
        │
        ▼ LLM 融合
图结构注入 LLM
  ├── GRACE (AST+CFG+PDG → prompt, F1 ↑28.65%)
  ├── LLMxCPG (CPG 引导 LLM 上下文感知)
  └── Vul-LMGNNs (LM 嵌入 + GNN 传播)
        │
        ▼ 关键空白
断边修复（D1 核心）
  └── SemTaint (LLM 解析 CodeQL 不可解析调用边) ← **唯一 D1●**
        └── ⚠️ 面向 JavaScript taint，非 Java SCA
```

### 脉络三：Agent 范式的「执行验证」共识

```
早期 Agent
  ├── T2L-Agent (崩溃 trace → 代码行)
  └── CVE-Genie (CVE → exploit 闭环)
        │
        ▼ 执行验证成标配
2026 Agent (D4● 聚集)
  ├── VIPER-MCP (反馈驱动提示进化 + PoC)
  ├── FORGE (5 Agent 分级利用 L0-L3)
  ├── Verify Before Fix (strict invariant: 无确认不修复)
  └── LLMVD.js (扫描→PoC→执行验证)
        │
        ▼ 关键偏离
验证目标差异
  └── 所有 Agent 验证「可利用性」(exploitability)
        ≠ SCA 需要「可达性」(reachability)
```

## 更新规则

每次完成一篇新论文的精读后：
1. 识别该论文与已有论文的所有关系（逐篇过 4 种类型）
2. 追加到上表
3. 如果该论文形成新的脉络节点，更新对应的脉络图
4. 如果该论文与「正文论文」有直接技术关联（方法论可复用），标注 ➡️ 正文论文
