## §1 Introduction（引言）

**篇幅**：约 2 页

**内容要点**：
1. **Motivation（1-2 段）**
   - 开源组件漏洞数量爆发，SCA 工具产生大量误报
   - 静态分析可达性判定受限于 Java 动态分发（反射/Spring/SPI/反序列化）
   - LLM 提供了语义级判定的新可能，但离线/内网环境需要本地小模型

2. **已有综述的局限（1 段）**
   - 列举已有综述（引用方向①的 3 篇），指出现有综述要么偏检测、要么偏程序分析、要么未覆盖 SCA+可达性+LLM 的交集
   - 强调本文综述的独特视角：不只看「LLM 能不能检测漏洞」，而是看「LLM 能不能帮助判定开源组件漏洞在特定系统中是否真的可达」

3. **Contributions（项目符号，4-5 条）**
   - 首次从 SCA 可达性验证视角系统综述 LLM 的应用

## §2 Background（背景）

**篇幅**：约 3–4 页

**内容要点**：

### 2.1 Software Composition Analysis (SCA) and the Reachability Problem
- SCA 工作流：版本匹配 → 漏洞数据库查询 → 可达性分析
- 版本匹配的挑战（引用 bao:2022 VSZZ, tan:2025 Similar-but-Patched）
- 可达性分析的挑战：调用图完整性问题（引用 foo:2019 Dynamics）

### 2.2 Static Analysis and the False Positive Crisis
- SAST 工具（CodeQL, Semgrep 等）的原理与局限
- 误报来源：不可行路径、过近似、调用图断裂
- 数据质量对漏洞检测的影响（引用 croft:2023 Data Quality — 发现 20-71% 标签不准、17-99% 数据重复）

### 2.3 Java Framework Dynamic Dispatch: The Root of Call Graph Breakage
- 反射、Spring DI/IoC、SPI、反序列化——四种动态分发机制
- 为什么传统静态调用图在这些场景下必然断裂
- 这对 SCA 可达性意味着什么

### 2.4 Large Language Models for Code Analysis
- Code LLM 的能力边界：能理解语义、不能替代精确指针分析
- 关键概念：few-shot、CoT、RAG、agent
- 离线小模型（7B）的能力与限制（引用 xu:2022 code LLM eval, ullah:2023 llms-cannot-reliably）

### 2.5 Key Concepts and Terminology
- 调用图 (CG) / 代码属性图 (CPG) / 程序依赖图 (PDG) / 数据流图
- 可达性 (reachability) vs 可利用性 (exploitability)
- 误报 (FP) vs 漏报 (FN) vs 零漏报保守策略
- 证据链 (evidence chain)

### 10.1 Synthesis Across Six Directions

对 49 篇论文按 6 个方向进行系统梳理后，我们识别出三条贯穿全文的技术脉络。

**脉络一：静态分析与 LLM 的关系正在从「替代」转向「互补」。** 早期工作（2022–2023）普遍将 LLM 视为独立的漏洞检测器——将代码片段直接输入 GPT-4 或 ChatGPT，询问「这里有漏洞吗？」（fu:2023, gao:2023, zhang:2023）。这一范式在 2024–2026 年被逐步放弃。取而代之的是「静态分析提供结构、LLM 提供语义」的互补范式：IRIS（li:2024）让 LLM 辅助静态分析推理；LLMDFA（wang:2024）将 LLM 定位为代码合成器而非直接推理器，委托其生成调用外部工具（Tree-sitter、Z3）的脚本；SAST-Genius（agrawal:2025）将 Semgrep 的输出交由 LLM 做上下文分析，FP 从 225 降至 20（降幅 91%）。SAVANT（lingxiang:2025）则将这一范式直接应用于 Java SCA 场景：语义预处理 + LLM 上下文感知可达性分析，在 55 个真实 Java 项目上达到 83.8% 精确率和 73.8% 召回率。ProjectScale 的实证研究（li:2026）从反面印证了纯 LLM 路线的不可行：在 Java 场景下，5 种 LLM 检测器的召回率仅约 34%，误发现率高达约 85%。**结论是清晰的：LLM 不能替代静态分析，但可以显著增强它。**

**脉络二：调用图/代码属性图正在成为 LLM 与程序分析之间的关键桥梁。** 方向④的 11 篇论文共同勾勒出一条技术路线：将程序的结构化表示（CPG、CFG、DFG、调用图）注入 LLM，使其获得超越纯文本的语义理解能力。GRACE（lu:2024）的实验结果最具说服力——将 AST+CFG+PDG 注入 LLM prompt 后，F1 提升 ≥28.65%（vs 6 个基线）。LLMxCPG（lekssays:2025）进一步将 CPG 作为 LLM 上下文感知的引导框架。然而，这条脉络存在一个关键盲区：**所有图结构均来自静态分析，而静态分析在 Java 动态分发场景下本就不可靠。** 将不可靠的图结构注入 LLM，并不能自动修复图的断裂——这需要专门的动态分发处理机制。

**脉络三：Agent 范式的「执行验证」共识已经形成，但目标偏离了可达性判定。** 方向⑤的 6 篇论文中，后 4 篇（VIPER-MCP、FORGE、Verify Before Fix、LLMVD.js）全部将动态执行验证作为核心组件。Verify Before Fix（gajjar:2026）甚至将「无执行确认不修复」作为 strict invariant——禁用验证导致不必要修复增加 131.7%。这一共识标志着领域的方法论成熟。但所有 Agent 的验证目标都是「可利用性」（exploitability）而非「可达性」（reachability）：它们回答的是「这个漏洞能否被利用」，而非「这个漏洞在当前系统中是否可达」。**从 SCA 审计的视角看，可达性是比可利用性更前置、更实用的问题**——等保审计需要判定的是「该组件漏洞在系统中是否构成真实风险」，而非「该漏洞在实验室环境中能否被利用」。

**三条脉络的交叉点——也是它们共同的空白——是：还没有工作将「互补范式 × 图桥接 × Agent 执行验证」三者贯通，应用于 Java 动态分发可达性判定场景。**

### 10.2 The Five-Dimension Differentiation Matrix

为系统量化这一空白，我们定义了一个五维分析框架，将综述覆盖范围内每一篇技术论文按以下维度评分：

| 维度 | 定义 | 判定标准 |
|------|------|---------|
| **D1: Java 动态分发可达性** | 是否处理反射/Spring 依赖注入/SPI/反序列化导致的调用图断裂？ | ●=明确设计了对动态分发断边的检测或修复机制; ◐=面向 Java 但未专门处理动态分发; ○=未涉及 |
| **D2: 本地小模型离线** | 是否使用 ≤7B 参数的模型在离线/内网环境运行？ | ●=明确使用 ≤7B 本地模型并报告了离线性能; ◐=提及小模型但主要实验仍用 API; ○=仅使用在线大模型 API |

此外，所有 Agent 的验证目标都是「可利用性」而非「可达性」。可利用性回答「能否被利用」，可达性回答「是否被调用」。前者需要构造 exploit，后者需要分析调用链。两者在方法论上有交集（都需要调用图分析），但在目标、方法深度、和审计交付物上存在本质差异。

从表 10.1 可以得出以下核心发现：

**发现一：没有任何一篇论文同时覆盖 3 个及以上维度。** 在 49 篇论文中，单个维度获得 ● 的论文最多 4 篇（D4），获得 ◐ 及以上的最多 7 篇（D1）。但跨维度组合极为匮乏——没有一篇论文在 D1 和 D2 上同时非 ○，没有一篇论文在 D2 和 D3 上同时非 ○。维度覆盖的最高组合仅为「D4 ● + D5 ◐」（如 VIPER-MCP 和 FORGE，但两者在 D1–D3 上均为 ○）。

**发现二：D2（离线小模型）虽已突破零覆盖，但存在严重的场景错配。** SecureFixAgent 证明了 <8B 模型在 Python 漏洞修复上可行，Bridging 证明了 <4B 模型在 Java 漏洞检测上可行。然而，两者均面向「漏洞检测/修复」，均非面向「SCA 可达性判定」。**「离线小模型 × SCA 可达性」组合在 49 篇文献中为零。**

**发现三：D1 的 ● 论文（SemTaint）和 D2 的 ● 论文（Bridging）之间没有任何技术交叉。** SemTaint 使用在线 LLM（GPT-4），Bridging 不做动态分发处理。如果有一篇论文在 Java SCA 场景下使用本地小模型解析反射/Spring/SPI 调用断边，它将同时获得 D1 ● 和 D2 ●——而这样的论文不存在。

**发现四：D1+D2+D3 组合完全未被探索。** 具体而言：
- D1（动态分发可达性）仅 SemTaint ●，面向 JavaScript，不涉及 SCA
- D2（离线小模型）仅 SecureFixAgent 和 Bridging ●，均不涉及可达性
- D3（零漏报保守）仅保守分析 ●，不涉及动态分发和离线部署
- **将三者组合——在 Java SCA 场景下，使用离线小模型对动态分发断边做零漏报保守的可达性补判——在 49 篇文献中没有任何工作涉及。**

**发现五：D4 的繁荣和 D5 的贫瘠形成鲜明对比。** 「动态执行验证」已成为 Agent 方向标配（4 ● + 3 ◐），但「证据链交付」仍无一篇 ●。这意味着即使有论文做到了动态验证，其输出仍是「是/否」标签或 exploit 脚本，而非面向审计的结构化证据。对于等保场景而言，这是一条从方法到交付的完整空白链。

### 10.4 Gaps by Direction

在汇总统计之外，每个方向还存在特定的结构性空白：

**方向② — SCA 可达性。** 最核心的空白是 LLM 语义补判的缺失。VFArche（zhang:2025）在 C/C++ 上做了版本+函数双维度定位，但未使用 LLM；VERCATION（cheng:2024）在 Java 上使用了 LLM，但仅用于版本识别而非可达性补判；SAVANT（lingxiang:2025）最接近目标——Java SCA + LLM 语义分析——但其核心仍是「漏洞存在性」而非「调用路径可达性」。**将 LLM 专门用于修复 Java 动态分发导致的调用图断边并判定可达性，在方向②中尚未出现。**

**方向③ — LLM 消减误报。** 空白在于保守策略与离线部署的结合验证。保守分析（wagner:2025）提供了零漏报的方法论框架，SecureFixAgent（gajjar:2025）证明了小模型可行，但两者从未在同一场景下被同时评估。**「零漏报保守 × 离线小模型 × 真实 SAST 警告」的三元组合在方向③中尚未被验证。**

**方向④ — LLM + 图。** SemTaint（ghebremichael:2026）在 D1 上的突破同时揭示了空白的具体位置：其「LLM 解析不可解析调用边」的范式可以被迁移到 Java SCA 场景，但需要 (i) 将 CodeQL 查询从 JavaScript taint 改写为 Java 调用图可达性；(ii) 面向反射/Spring/SPI/反序列化四种机制设计专门的提示策略；(iii) 评估本地小模型替代 GPT-4 的可行性。**这三个步骤在方向④中均未被执行。**

**方向⑤ — LLM Agent。** Agent 社区已经解决了「如何验证可利用性」的方法论问题（D4 ●），但尚未提出「如何验证可达性」的问题。将 Agent 的「执行验证」能力从可利用性转向可达性——通过运行时插桩采集真实调用图与静态调用图做差异对比——是一个尚未被探索但方法论上可行的方向。

**方向⑥ — 评测与数据集。** ProjectScale（li:2026）揭示的 Java 场景下 LLM 召回 ~34%、误发现率 ~85% 的数据表明，纯 LLM 路线在 Java 漏洞检测上的性能远低于文献中常见的 C/C++ 评测数字。这暗示了 Java 语言的特性（类型系统、框架抽象、动态分发）对 LLM 的语义理解构成了独特挑战——而现有数据集均未将这些特性纳入标注维度。RealSec-bench（wang:2026）的 34-hop 跨过程依赖标注是向正确方向迈出的一步，但仍未区分「静态可达」和「动态可达」。

### 10.5 Threats to Validity

本综述存在以下效度威胁：

**检索覆盖范围。** 我们的检索以 arXiv、dblp、Google Scholar 为主，辅以顶会 proceedings 的手工检索。这一策略对预印本和会议论文覆盖较好，但可能遗漏 (i) 纯期刊论文（如 TSE、TOSEM 中未经 arXiv 预印的论文）；(ii) 工业界白皮书和技术报告（通常不出现在学术数据库中）；(iii) 非英文文献。我们通过从已核实论文的引用链做前向/后向追踪来缓解这一问题，但无法完全消除。

**纳入/排除标准的主观性。** 「方法可迁移到 Java」的判断边界存在主观性——部分 C/C++ 工作的方法论可能通过工程适配迁移到 Java，但我们在标题/摘要筛选阶段可能已将其排除。我们采取保守策略：仅当论文明确提出其方法可跨语言应用或已经包含 Java 实验时，才判定为「可迁移」。这可能导致少数潜在相关工作被遗漏。

**五维评分的推断性质。** D1–D5 的评分基于论文的原文表述。对于部分维度（尤其是 D3 零漏报保守策略和 D5 证据链交付），论文可能在实际实现中包含相关设计但未在文本中明确描述。我们对此采取「文本可见即得分」的保守原则——仅当论文中明确提及相关设计时才给予 ◐ 或 ●。这意味着部分维度的得分可能偏低（假阴性），但不会偏高（假阳性）。最终的汇总统计因此是「下界估计」——实际的维度覆盖可能比表中显示的略多，但不会改变「无 3 维组合」的核心结论。

**领域时效性。** LLM + 代码安全是快速发展的领域。本综述的检索截止于 2026 年 6 月，纳入论文以 2024–2026 年为主体。未来 12 个月内可能出现填补本文所述空白的新工作，综述的结论需要在持续更新中校准。

尽管存在上述威胁，我们认为本综述的核心贡献——**通过五维矩阵系统识别「Java 动态分发可达性 + 离线小模型 + 零漏报保守 + 动态验证 + 证据链」的交叉空白**——对领域具有稳健的指导价值。该空白的存在性不依赖于单篇论文的评分细调，而是由 49 篇论文的跨维度分布模式（无任何论文覆盖 ≥3 维、D1–D2–D3 完全无交叉）所结构性决定的。

## §11 Future Research Directions（未来研究方向）

**篇幅**：约 2–3 页

**内容要点**：
1. **Unified Reachability Framework (D1+)**：静态调用图构建 + LLM 对动态分发断边做选择性语义补判的混合架构。VERCATION 的「静态+LLM 联合」范式和 LLMDFA 的「LLM 作代码合成器」范式的融合方向
