# DGA (LiveSCA) 精读笔记 (v2)

> arXiv:2503.22576 | Zhang et al. | NTU + Tianjin Univ. + Imperial Global + Nankai Univ. | 2025 | FSE'25

---

## A. 大白话总结（3-5句）

LiveSCA 提出了一个大胆的设想：**不做 SCA 数据库**。传统 SCA 工具需要维护一个包含所有第三方库特征的大数据库来比对，但 C/C++ 没有统一的包管理器，数据库永远不全。作者另辟蹊径——当检测到一个可能是第三方库的代码时，不去数据库里找匹配，而是像安全分析师一样：提取代码特征 → 去 Google 搜 → 用多 LLM 智能体协作分析搜索结果 → 判断这究竟是哪个开源库。这是个"无数据库 SCA"的初探，在 Android SO 库定位上成功率 59.20%，C/C++ 克隆库定位上 57.50%。虽然准确率还远不如传统有数据库方案，但证明了在不依赖预建数据库的情况下做 SCA 是可能的，尤其适合那些在主流数据库里找不到的长尾库。

## B. 术语卡片（3-5个）

- **DB-Less SCA / 无数据库 SCA**: 不依赖预建特征数据库，实时通过互联网搜索 + LLM 分析来识别第三方库来源。
- **Multi-Agent Framework**: 5 个各司其职的 LLM 智能体（证据提取/搜索/关联分析/验证/汇总），模拟人工分析流程。
- **Evidence Extraction / 证据提取**: 从目标代码中提取功能描述性语义（API 注释、依赖配置、文件结构等），而非传统 SCA 的代码结构特征。
- **Validation Feedback Loop / 验证反馈循环**: Agent4 独立核对搜索结果与原始证据是否吻合，不吻合则反馈给证据摘要模块优化关键词，最多迭代 3 轮。
- **"Hints Found" / 线索发现**: 即使没成功定位到库的来源，LiveSCA 也能提供厂商名/版本信息等线索（102/108 例），辅助人工跟进。

## C. 核心知识点（3-5个，只留硬数字和结论）

1. **DB-Less SCA 成功率**: Android SO 定位 59.20%（64/108），C/C++ 克隆库定位 57.50%（115/200），**准确率尚不能与有数据库方案竞争**。
2. **成本效率**: 平均 27.99-29.36 秒/例，Token 消耗 10K-15K，预估总成本 < 8 美元/200 例。
3. **失败的三大原因**: ① 提取的特征不具区分度（关键词模糊）；② 搜索结果页面 404（库已下线）；③ 库名太常见导致搜索被无关结果淹没。
4. **局限**: 只做粗粒度库来源定位，不支持版本级细粒度识别；依赖互联网公开信息，私库/内网库无法覆盖；用 GPT-4o 有 API 调用成本和不稳定性。
5. **启示**: 把 SCA 从"数据库匹配"范式转换到"智能体搜索+验证"范式，对 C/C++ 无统一包管理器的场景尤其有价值。

## D. 综述用途

### ① 双轴坐标

- **横轴（补静态分析的短板）**: 数据与标注质量 —— 传统 SCA 数据库的"永远不全"问题。
- **纵轴（健全性代价）**: 声称两者兼顾 —— 设计上不引入额外漏报（因为不依赖有限数据库），但实际准确率低导致大量漏识别。

### ② 可引硬数字

- 59.20% Android SO 库定位准确率。
- 57.50% C/C++ 克隆库定位准确率。
- 平均 29.36s/例，< 8 USD/200 例。
- 102/108 例至少提供有效线索。

### ③ 方法演进谱系

```
传统 SCA: LibScout(2016) → LibD(2017) → CENTRIS(2021) → OSSFP(2023)
                                             ↓
LiveSCA: 无数据库方案，用 LLM Agent 替代特征数据库
         └── 受 LLM 应用进化启发: Fine-tune → RAG → Agent
```

这是 SCA 技术路线的一次"范式翻转"——从"大数据 > 大能力"到"小数据 + 智能推理 > 大能力"。

### ④ Gap 贡献

- **暴露 gap**: 数据库的不完全性是 SCA 的瓶颈，但现有研究几乎全部忽略此问题；C/C++ 无统一包管理器的问题长期被回避。
- **对双轴框架的意义**: 间接证明"数据库依赖"是 SCA 部署就绪度的隐藏维度——离线/内网场景下 LiveSCA 完全不可用（需联网搜索），进一步强化"离线部署"的紧迫性。
- **局限即 gap**: 不支持版本级识别、不支持私库，这些是议程 2（离线小模型）和议程 4（跨语言）的直接证据来源。

| 维度 | 评分 | 说明 |

| D1: 方法新颖性 | ● | "无数据库 SCA"想法具有颠覆性 |
