# SLICEMATE 精读笔记 (v2)

## A. 大白话总结（3-5句）

SliceMate 是用 LLM 智能体来做**静态程序切片**——给一段代码和一个"切片标准"（比如某个变量在第几行），找出所有影响了这个变量的代码行。传统方法需要构建全量依赖图然后做可达性分析，大项目上跑不动；学习的方案受限于训练数据质量且逐行分类效率低。SliceMate 设计三个 LLM 智能体：合成（逐步扩范围生成候选切片）、验证（检查缺不缺/多不多）、精修（根据验证反馈修补），由控制模块协调迭代。在自建的 2200 例切片基准 SliceBench（含 5-8577 行代码）上，比最好的基线（NS-Slicer Pro）F1 高最多 28%，大代码上表现更突出。单次切片成本约 0.15 美元（GPT-4o）。下游调试/bug 定位 Top-10 准确率最高提升 34%。

## B. 术语卡片（3-5个）

- **Static Program Slicing / 静态程序切片**: 不执行程序，找出所有可能影响某位置（切片标准）的代码语句。
- **ACI (Agent-Computer Interface)**: SliceMate 为智能体提供的四种代码检索接口（作用域搜索/调用提取/函数检索/模块信息提取），类似工具调用。
- **Synthesis Agent / 合成智能体**: 从切片标准出发，逐步扩范围（函数内→跨函数→跨文件），每次让 LLM 判断依赖。
- **Verification-Refinement Loop / 验证-精修循环**: 验证智能体检查切片是否"缺"（完整）和"多"（精简），精修智能体根据反馈修补，至多迭代 5 轮。
- **SliceBench**: 2200 例人工标注的 Java/Python 切片基准（含 2000 CodeNet 小样本 + 200 GitHub 大样本），双人交叉验证（Kappa 0.893-0.927）。

## C. 核心知识点（3-5个，只留硬数字和结论）

1. **性能碾压基线**: 在 SliceBench 大代码（GitHub 数据集）上，Python F1 从 0.247（NS-Slicer Pro）→ 0.761（SliceMate，+51.4%），Java F1 从 0.447 → 0.736（+28.9%）。
2. **传统工具在大代码上失败**: 传统工具（JavaSlicer/Joern）在平均 2106 行代码的 GitHub 样本上精确率低于 0.2（0.199/0.136/0.193），基本不可用。
3. **成本可控**: 大代码单次切片约 0.15 美元（GPT-4o）。
4. **下游效果显著**: IncBL + SliceMate 的 Top-10 bug 定位准确率提升 34%（BugsInPy）和 21%（Defects4J）。
5. **消融验证有效**: 去除验证+精修模块后 F1 下降 5.5%-24.9%，5 轮迭代后收益递减。

## D. 综述用途

### ① 双轴坐标

- **横轴（补静态分析的短板）**: 调用图不完整（可达性） + 数据与标注质量 —— 切片本质上就是反向可达性分析；LLM 替代依赖图构建绕开了调用图断裂问题。
- **纵轴（健全性代价）**: 两者兼顾但无保证 —— 设计上同时追求"不缺"和"不多"，但没有健全性证明；验证-精修循环只能减少不能消除错误。

### ② 可引硬数字

- 大代码上 Python F1: 0.761（+51.4% vs NS-Slicer Pro）。
- 大代码上 Java F1: 0.736（+28.9% vs NS-Slicer Pro）。
- 单次 GPT-4o 成本约 0.15 美元。
- Bug 定位 Top-10 提升 34%。
- SliceBench: 2200 例 / 5-8577 行 / Java+Python / Kappa 0.893-0.927。

### ③ 方法演进谱系

```
传统切片: JavaSlicer → Joern → TypeSlicer (依赖图+可达性)
                    ↓
学习式切片: NS-Slicer (CodeBERT, 逐行二分类, 2019)
                    ↓
LLM Agent切片: SliceMate (合成+验证+精修, 2025)
```

SliceMate 代表"LLM Agent 替代传统静态分析管道"的新范式，与 IRIS（LLM+CodeQL）和 LLMxCPG（LLM+CPG）形成三类对比：增强 vs 替代。

### ④ Gap 贡献

- **暴露 gap**: 传统切片工具在大代码/跨文件场景下基本不可用；学习式切片受限于训练数据质量和逐行分类的低效。
- **对双轴框架的意义**: 证明 LLM 可以不依赖依赖图做精确的依赖追踪——这对 SCA 可达性验证的"调用图断裂"问题是直接启发。但 SliceMate 不做健全性保证，正是议程 1（健全性保证的可达性验证）的反面证据。
- **局限**: 2000 CodeNet 样本仅 30 行单函数，不具备代表性；完全依赖 GPT-4o，未评估小模型；基准是学术代码而非企业级 Java 框架代码（无 Spring/反射）。
