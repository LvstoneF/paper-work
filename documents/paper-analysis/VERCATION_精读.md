# VERCATION 精读笔记

> arXiv: 2408.07321 | 方向 ② SCA 可达性 | ★ 必引必区分 | 2025

---

## A. 大白话总结

这篇论文解决的是 NVD 漏洞数据库的一个老问题：**官方报告里写的"受影响版本"经常是错的**——要么漏报（只写了发现漏洞的那个版本，实际更早的版本也有问题），要么多报（一刀切把所有旧版本都标为漏洞版本）。作者做了一个叫 VERCATION 的工具，分三步：先用程序切片从补丁代码里把和漏洞相关的语句揪出来，再让 LLM（用 DeepSeek-V3）像安全研究员一样分析漏洞逻辑、筛掉无关语句，最后用 AST 语法树比对在代码提交历史里回溯，找到漏洞是哪个 commit 引入的（vic）。在 12 个开源项目、122 个 CVE 上测试，F1 达到 93.1%，比现有最强方法（V-SZZ 80.1%）高出 13%。最有意思的发现是：**他们用 VERCATION 扫了 NVD 里 342 个 CVE，发现 202 个（59.1%）的受影响版本信息是错的**——其中近一半是高危或严重级别。

## B. 术语卡片

### 术语 1：漏洞引入提交 (vulnerability-introducing commit, vic)
- **大白话**：第一次把漏洞写进代码的那次 git 提交。从这次提交开始，所有后续版本都带着这个漏洞，直到修复提交（vfc）把它修掉。
- **在本文中的作用**：整篇论文的核心输出——找到了 vic 就能确定「哪些版本受漏洞影响」。vic 和 vfc 之间的版本 = 漏洞版本。
- **打个比方**：vic 是「病源感染时刻」，vfc 是「治愈时刻」。两个时间点之间的所有人都被感染了。NVD 的问题是经常搞错感染起始时间——要么把治愈前一天才说成感染，要么把出生算成感染。

### 术语 2：程序切片 (Program Slicing)
- **大白话**：从代码里顺着数据流和控制流，把和某个变量/语句有关的所有语句都切出来。向后切 = 找「谁影响了它」，向前切 = 找「它影响了谁」。
- **在本文中的作用**：Phase 1 的第一步——从补丁代码出发，用程序切片捞出所有和漏洞相关的候选语句（dangerous flow）。但这些语句里混了很多「有关但不核心」的，所以需要 LLM 做第二轮筛选。
- **打个比方**：刑侦里的关系网排查——从一个嫌疑人出发，画出所有和他有联系的人（程序切片），然后让侦探（LLM）判断哪些人真的参与了犯罪。

### 术语 3：AST 展开与归一化 (AST Expansion & Normalization)
- **大白话**：AST 展开 = 把函数调用替你「内联展开」，看到函数内部做了什么。AST 归一化 = 把写法不同但逻辑相同的代码统一格式（比如 for 循环统一转成 while、b+a 统一转成 a+b）。
- **在本文中的作用**：Phase 2 的核心技术——解决「代码重构导致表面看起来不一样但逻辑一样」的问题。传统方法看到代码被封装成函数就认为是新代码，VERCATION 把函数展开后比对，发现逻辑一致就认为不是新引入的。
- **打个比方**：AST 展开 = 「你说了什么不重要，我要看实际做了什么」。AST 归一化 = 把「2+3」和「3+2」都翻译成同一种写法再比较。

### 术语 4：敏感函数加权 (Sensitive Function Weighting)
- **大白话**：给漏洞语句打分时，如果语句里调用了已知的危险函数（如 memcpy、strcpy、free），就给它更高权重（2 倍）。因为这些函数更可能是漏洞的触发点。
- **在本文中的作用**：在 Phase 2 回溯提交时，高权重语句的匹配结果对最终判定影响更大——避免被不重要的语句（如变量定义）的匹配结果带偏。
- **打个比方**：刑侦排查时，有前科的人（敏感函数）优先级高于普通路人。同样是「没找到匹配」，如果危险人物是新增的，基本确认这就是犯罪引入点。

### 术语 5：Few-shot + Chain-of-Thought 提示策略
- **大白话**：Few-shot = 给 LLM 看几个已完成的例子（「这是 CVE 描述，这是危险流，这是应该筛选出的漏洞语句」），让它照着学。CoT = 要求 LLM 先解释漏洞逻辑，再列出漏洞语句（「先分析，再答题」）。
- **在本文中的作用**：Phase 1 中用这两种策略一起让 LLM 从程序切片结果中精准挑出漏洞语句。实验证明 Few-shot+CoT 的 F1（93.1%）比纯 Zero-shot（79.7%）高 13.4%。
- **打个比方**：Few-shot = 考试前给你看两道例题和标准答案。CoT = 要求你先写解题过程再写答案——防止瞎猜。

## C. 核心知识点

### 知识点 1：静态分析 + LLM 的协同效应（不是替代，是互补）
VERCATION 做了一个关键消融实验，证明三者的 F1 分别是：
- 纯静态分析（Joern）：48.3%（召回率 54.4%）
- 纯 LLM（DeepSeek-V3）：76.8%（精确率 74.2%）
- **两者结合：93.1%**

纯静态分析的问题是「切太多」——平均每个 CVE 切出 22.6 条语句，大部分和漏洞逻辑无关。纯 LLM 的问题是给整个函数它会「想太多」——包含太多上下文噪声。**程序切片先做粗筛（22.6→候选集），LLM 再做精筛（→5.66 条最终漏洞语句），两者互补。**

> 📊 **关键数字**：Joern 平均切 22.64 条/CVE → LLM 精筛到 5.66 条/CVE，F1 从 48.3% 跳到 93.1%，提升 92.8%

### 知识点 2：重构提交（refactoring commit）是回溯定位的「隐形杀手」
在 Phase 2 的回溯过程中，32.2%（89/276）的中间提交是重构提交——开发者在 vic 到 vfc 之间重构了代码（函数封装、变量改名、结构优化），表面代码变了但逻辑不变。传统编辑距离方法看到 48% 相似度就认为「代码变了→这是漏洞引入点」，产生大量误报。VERCATION 用 AST 展开+归一化后比对，能识别出这些重构提交，继续往前回溯。

> 📊 **关键数字**：32.2% 的中间提交是重构提交，仅 11% 的漏洞是 vic 和 vfc 之间没有任何中间提交的「简单情况」

### 知识点 3：NVD 版本信息的系统性错误（59.1%！）
VERCATION 在真实世界验证中扫描了 342 个 CVE，发现 202 个（59.1%）的 NVD 受影响版本信息有误：
- **漏报**（134 个，39.2%）：108 个 CVE 只报告了单个版本（实际影响更广）
- **多报**（68 个，19.9%）：56 个 CVE 把所有旧版本都标为漏洞（实际没那么早引入）
- **严重性分布**：中危 49.7%、高危 49.1%、严重 2 个

> 📊 **关键数字**：NVD 准确率仅 Precision 66.8%、Recall 41.8%、F1 51.4%——不到 VERCATION 的一半

### 知识点 4：跨知识截断时间的泛化能力——不是靠「背答案」
论文专门做了 knowledge cutoff 分析：CodeLlama 训练数据截止 2022.09，GPT-4 截止 2023.12，DeepSeek-V3 截止 2024.07。把所有 CVE 按各模型的截止时间分成前后两组，发现性能差距极小（<2%），DeepSeek-V3 在 post-cutoff CVE 上甚至略好（+1.73%）。这说明 LLM 用的是「代码理解能力」而不是「记住具体 CVE 信息」。

> 📊 **关键数字**：DeepSeek-V3 pre-cutoff F1=92.7% vs post-cutoff F1=94.3%，差距仅 1.73%

### 知识点 5：效率和成本的实用化水平
平均每个 CVE 分析时间 28.61 秒，LLM API 成本 $0.0034/CVE（DeepSeek-V3，含 10 次独立运行）。对比：纯动态分析（PoC 执行）每个 CVE 平均需要 3 小时环境搭建+执行——效率差约 742 倍。

> 📊 **关键数字**：28.61 秒/CVE、$0.0034/CVE、vs PoC 动态执行 3 小时/CVE（742 倍差距）

## D. 对我们综述有什么用

| 用到综述哪个章节 | 用来论证什么 |
|:---|:---|
| **§2.1** SCA 工作流 | 用「NVD 59.1% 版本信息有误」和「NVD F1 仅 51.4%」说明漏洞版本数据库的根本性不可靠——这是 SCA 可达性分析的「上游数据质量问题」 |
| **§5** RQ2 核心引用 | 作为方向②「静态+LLM 联合做版本定位」的代表方法，与 VFArche（函数定位）对比：VFArche 定位「漏洞在哪个函数」，VERCATION 定位「漏洞在哪个版本」——两者互补但都止步于可达性判定之前 |
| **§6** RQ3 LLM 消减误报 | 引用「静态分析+LLM 协同」的消融实验（纯静态 48.3%→联合 93.1%）作为 LLM 增强静态分析精度的强证据 |
| **§10.1** 脉络一 | 作为「静态分析与 LLM 从替代转向互补」脉络的核心案例——P1 消融实验完美证明 1+1>2 |
| **§10.2** 5 维矩阵 | D1 ◐（面向 C/C++ 有指针间接调用处理，但非 Java 动态分发）、D2-D5 全部 ○（详见下方） |
| **§11** 未来方向 | Phase 1「程序切片 + LLM 精筛」的架构可直接复用到可达性补判管线——切片做粗筛、LLM 做语义级调用图断边修复。但需要从 C/C++ 迁移到 Java 动态分发场景 |
| **§3** RQ 设置 | 用「vic 定位 ≠ 可达性判定」来区分 RQ2 内部的两个子问题：版本定位 vs 调用路径可达性 |

## E. 5 维矩阵评分

| 维度 | 评分 | 白话理由 |
|:---|:---:|:---|
| D1 动态分发 | ◐ | 面向 C/C++（不涉及 Java 反射/Spring/SPI），但 Joern 切片能部分处理函数指针和间接调用，AST 展开也尝试处理调用目标不明确的情况。方法思想可迁移到 Java 但未验证。 |
| D2 离线小模型 | ○ | 主实验使用 DeepSeek-V3 在线 API（$0.0034/CVE）。虽然也测试了可本地部署的 CodeLlama-13B（F1=87.6%，比 DeepSeek 的 93.1% 低 5.5%），但未将离线部署作为设计目标。 |
| D3 零漏报保守 | ○ | 目标是在精确率和召回率间取得平衡，不以保证零漏报为设计约束。事实上还有少量漏报（recall 94.5%，仍漏了 5.5%）。 |
| D4 动态验证 | ○ | 纯静态+LLM，无运行时执行。验证 ground truth 时用了 PoC，但 PoC 仅用于数据集标注而非方法流程。 |
| D5 证据链 | ○ | 输出是 vic 的 commit ID + 受影响版本范围，不包含调用路径、判定理由或置信度信息。 |

**一句话**：VERCATION 和 VFArche 都是可达性分析的「前置步骤」——VFArche 回答「漏洞函数是哪个」，VERCATION 回答「漏洞版本是哪个区间」。两者距离我们的正文论文目标（「已知漏洞函数，判定在当前系统中是否可达」）都还差一个「调用图可达性判定」环节。但 VERCATION 的「静态分析+LLM 协同」架构是我们正文管线设计的最直接参考模板。
