# LLMxCPG 精读笔记

**论文标题**: LLMxCPG: Context-Aware Vulnerability Detection Through Code Property Graph-Guided Large Language Models  
**作者**: Ahmed Lekssays, Hamza Mouhcine, Khang Tran, Ting Yu, Issa Khalil  
**年份**: 2025 (arXiv:2507.16585)  
**方向**: ④ CPG 引导 LLM 上下文感知漏洞检测

---

## A. 大白话总结（3-5句）

LLMxCPG 是一个"双模型流水线"：先用一个模型（LLMxCPG-Q）学会写 Joern 的 CPGQL 查询语句，自动从代码中提取与漏洞相关的"切片"（把代码砍掉 67–91%，只保留和漏洞沾边的执行路径），再用第二个模型（LLMxCPG-D）只对切片做二分类（有漏洞 / 安全）。核心洞察是——给 LLM 塞整段代码它学到的全是表层模式，但给它精准切片后，它就能学到真正的漏洞特征，结果 F1 比 SOTA 高了 15–40%。弱点也很明显：只针对 C/C++ 内存类漏洞（CWE-119 系），且依赖 Joern 的静态 CPG，对 Java 动态分发（反射/Spring）毫无处理。

---

## B. 术语卡片（3-5个）

1. **CPG（Code Property Graph）**：把 AST、CFG、PDG 合并成一个图结构，用一个图查询语言（CPGQL）做遍历。本文用的 Joern 是开源 CPG 工具。
2. **CPGQL**：Joern 的图查询语言，基于 Scala。本文的核心技术点就是让 LLM 学会写 CPGQL 查询来提取漏洞执行路径。
3. **Backward Slicing**：从"可疑点"（sink/不安全变量）反向追溯所有影响该点的数据流和控制流依赖，生成精简切片。
4. **Interacters**：与执行路径"交互"的变量——即行号与路径节点重叠的标识符。这是 LLMxCPG 切片构造的中间产物，用于补充路径缺失的上下文（如变量定义）。
5. **Threshold Calibration**：LLMxCPG-D 输出两个 logit（Vulnerable/Safe），通过 softmax 转概率后，用数据集特定的阈值 γ 做最终判定。不同数据集阈值差异很大（PrimeVul γ=0.594 vs ReposVul γ=0.193）。

---

## C. 核心知识点（3-5个，留硬数字）

1. **代码缩减比**：函数级数据集 67.84%（PrimeVul）– 78.70%（FormAI）；项目级数据集高达 90.93%（ReposVul）。缩减后 LLM 只分析关键执行路径。
2. **主结果**：FormAI 上 Acc 0.8146 / F1 0.8075；PrimeVul 上 Acc 0.7250 / F1 0.6206（Precision 1.0 但 Recall 仅 0.45—即非常保守，宁可漏报也不误报）；SVEN（未见数据集）上 Acc 0.6020 / F1 0.7048（Recall 0.9534 极高）。
3. **查询生成能力**：LLMxCPG-Q 在 1278 个测试样本上全部生成合法 CPGQL 查询（100%），而 DeepSeek-v3 仅 132 个（10.3%）、Qwen2.5-Coder-32B 仅 19 个（1.5%）。说明专门微调对低资源语言（CPGQL）至关重要。
4. **消融证实切片效果**：去掉 CPG 切片后，FormAI 上 Acc 从 0.8146 降到 0.6762（-17%），PrimeVul 从 0.7250 降到 0.4875（-33%），SVEN 从 0.6020 降到 0.5078（-16%）。
5. **鲁棒性**：在 4 种语义保持变换下，LLMxCPG 表现稳定。FormAI 上 F1 反而微增（0.8075→0.8245 at T3），说明切片机制天然抗语法扰动。

## D. 综述用途

### ① 双轴坐标

- **横轴（补静态分析的哪个短板）**：调用图不完整（CPG 补丁）+ 模式不精确（通过语义切片而非语法匹配）。核心是"用 CPG 给 LLM 提供精确上下文"。
- **纵轴（健全性代价）**：只剪误报（Precision 常为 1.0），但 Recall 不高（0.45–0.80），**没有零漏报保证**。属于"剪误报，但可能引入漏报"象限。
- **部署就绪度评分**：
  - 可编译性要求：必须能运行 Joern 生成 CPG，需要完整编译环境（C/C++）→ **中**
  - 是否依赖云端大模型：本地部署 32B 模型 → **低**（可离线）
  - 可解释性：通过 CPGQL 查询和切片路径提供部分可解释性 → **中**
  - 是否在真实代码库验证：是（ReposVul、PKCO-25、SVEN）→ **高**

### ② 硬数字

- F1 提升 15–40% vs SOTA（函数级）
- 代码缩减 67.84–90.93%
- Joern-scan（纯规则基线）在 50 个样本中检测出**0 个**漏洞——说明传统静态查询在真实 wrapper 场景下完全失效
- 76% 的生成查询语义正确（安全专家评审，Fleiss Kappa=0.6429）

### ③ 方法谱系位置

传统 DL（VulDeePecker / SySeVR / LineVul）→ 图+DL（ReGVD / Vul-LMGNN）→ **LLM+CPG 混合架构（本文）**。LLMxCPG 的核心创新不是"用 LLM 分析代码"，而是**用 LLM 生成图查询来驱动静态分析工具**，让静态分析的结果更精准地喂给 LLM。这与"纯 LLM 端到端"的做法形成对比。

### ④ Gap 贡献

- **直接贡献**：展示了 LLM + CPG 混合路线的可行性和 SOTA 性能，为"静态找候选 + LLM 判语义"范式提供了有力证据。
- **暴露的 gap**：仅 C/C++、仅内存类 CWE、无 Java 动态分发处理、无零漏报约束、切片依赖 Joern（需要完整编译）。这些 gap 直接支撑了我们综述的议程 1（Java 可达性验证）和议程 3（健全性评测框架）。

## E. 5 维矩阵（D1–D5 ○/◐/●，各 1-2 句理由）

| 维度 | 评分 | 理由 |
|------|------|------|
| **D1: 方法新颖度** | ● | 用 LLM 生成 CPGQL 查询来自动指导静态分析工具做切片，这种"LLM 控制静态分析工具"的思路在漏洞检测领域是首次。但基座技术（CPG/Joern/切片）都是现成的，融合方式有新意但非颠覆性。 |
| **D2: 实验严谨性** | ● | 评测非常全面：4 个数据集（含 2025 年 post-cutoff CVE）、4 种鲁棒性变换、人类专家评审查询语义正确率、消融实验、基模型对比。但缺漏报率/FN 的显式报告，且只跑 C 语言内存类漏洞。 |
| **D3: 落地可行性** | ◐ | 32B 模型可用单卡 A100 部署，Joern 开源，可离线运行。但需要完整编译环境（对 Java/Maven 项目更复杂），且切片流水线依赖 Joern server 集群。企业内网部署可行但有一定工程成本。 |
| **D4: 综述引用价值** | ● | 高度相关。① 是"LLM + CPG"混合架构的典型案例；② 有丰富的对比数字（F1 提升 15–40%）；③ 切片思路可直接类比到我的综述的"上下文压缩"主题；④ gap 暴露清晰。 |
| **D5: 可扩展性** | ◐ | 框架设计上与语言无关（Joern 支持多语言），但实验只覆盖 C/C++ 内存类 CWE。扩展到 Java/Spring 需要解决两个问题：Joern 能否正确解析 Java 动态分发（反射/DI），以及 CPGQL 查询能否表达更多 CWE 模式。论文坦言 race condition 和 design errors 无法处理。 |
