# LLM4FPM 精读笔记

> Utilizing Precise and Complete Code Context to Guide LLM in Automatic False Positive Mitigation
> arXiv:2411.03079v2 | 2025-05 | Chen et al. (USTC)

---

## A. 大白话总结

LLM4FPM 解决一个核心问题：给 LLM 喂什么样的代码上下文，它才能准确判断 SAST 告警是不是误报？

现有方法（如 LLM4SA）直接把整个函数体扔给 LLM，一方面有很多无关代码混淆 LLM，另一方面又缺少跨文件的全局变量/外部函数定义等关键信息。LLM4FPM 做了两件事解决这个问题：

1. **eCPG-Slicer**：在 Joern 的 CPG 基础上增加了三种缺失的依赖边（调用依赖、结构依赖、变量依赖），做行级精确切片，只保留和告警相关的代码行。
2. **FARF 算法**：用文件引用图 + 强连通分量（Tarjan 算法）快速找出告警依赖的所有跨文件代码，避免全程序分析的高昂开销。

在 Juliet 数据集上 F1 > 99%（远超标线 LLM4SA），D2A 数据集标签准确率从 53% 提升到 86%，8 个真实项目上消除 >85% 误报。使用开源 qwen2.5-32b 模型，每个告警 4.7 秒，单次运行可省约 $2758。

## B. 术语卡片

| 术语 | 解释 |
|------|------|
| **eCPG (extended Code Property Graph)** | 在 Joern 的标准 CPG 基础上增加调用依赖(F)、结构依赖(S)、变量依赖(V) 三类边，提升切片精度 |
| **FARF (Find All Referenced Files)** | 基于 FRG + Tarjan SCC 的跨文件依赖分析算法，线性时间找出告警依赖的全部文件 |
| **CPG** | 代码属性图，融合 AST、CFG、PDG 的统一图表示 |
| **程序切片 (Program Slicing)** | 从程序中提取与某个特定点（切片准则）相关的所有语句的技术 |
| **Self-Consistency** | LLM 生成 5 次回答，多数投票定结论，减少随机性 |
| **Expert Prompting + CoT + Few-Shot** | 组合提示策略：专家角色 + 思维链 + 少量示例 + 自一致性 |

## D. 综述用途

### 双轴坐标
- **横轴（补静态分析短板）**: 模式不精确（代码上下文精度 + 完整性不足导致的误报）
- **纵轴（健全性代价）**: 只剪误报（不保证零漏报），但 F1 > 99% 意味着漏报极少
- **就绪度评分**:
  - 可编译性要求: 不需要编译（基于 Joern 的 CPG，字节码/源码级）— ○
  - 是否依赖云端大模型: 否，完全本地部署 qwen2.5-32b — ○
  - 可解释性: CoT 推理 + 自一致性投票有一定可解释性 — ◐
  - 真实代码验证: 8 个 20k+ star 真实项目 — ●

### 可引硬数字
- Juliet F1 > 99%，超越 LLM4SA 15-20%
- D2A 标签准确率从 53% → 86%（重要：说明了 SAST 数据集本身标签质量多差）
- 每个告警平均 912 input tokens，4.7 秒（低成本实践的标杆数据）
- 消融结果：去掉切片后 F1 降 1.7pp + 40 例超窗口；去掉跨文件分析后 F1 降 4.5pp

### 在方法演进谱系的位置
- LLM4SA (Wen 2024) → **LLM4FPM (Chen 2025)**: 从"喂整个函数"到"精确行级切片 + 跨文件上下文"
- CPG 用于切片: Yamaguchi → Zheng (VulSPG) → **LLM4FPM (eCPG-Slicer)**
- 与"ZeroFalse"(2510.02534) 和"保守分析"(2506.16899) 方向不同：LLM4FPM 关注上下文精度而不是健全性保证
- 与 LineVul 系列关系：LineVul 是行级漏洞定位，LLM4FPM 是行级切片做误报判定

### 暴露的局限 / gap 贡献
1. **仅 C/C++**：Joern 虽支持多语言但实验只在 C/C++
2. **Joern 解析瓶颈**：函数指针、宏展开、#IFDEF 等导致切片不完整
3. **无健全性保证**：不保证零新增漏报（虽然 F1 很高，但论文未讨论 FN 率）
4. **切片后仍可能很长**：真实项目切片可能几百上千行（作者第 V 节承认）
5. **Joern 的 CPG 导出慢**：Juliet 全量导出 CPG 需近 1 小时（第 II-C 节）
6. **不处理 Java 动态分发**：反射/Spring DI/SPI 等 Joern 不支持
7. **qwen2.5-32B 在 2025 年不算最新**——需关注更强小模型的结果差异
