# SAVANT 精读笔记

> arXiv: 2506.17798v2 | 方向 ② SCA 可达性与漏洞函数/版本定位 | ◆ 第二轮新增关键论文 | 2025

---

## A. 大白话总结

SAVANT 是一个利用大语言模型（LLM）来检测 Java 项目因使用第三方依赖库而引入的漏洞是否真正可被利用的工具。它的核心思路是：先对项目的源代码做智能切块（基于 AST 语法树）并转换为向量嵌入，然后用漏洞补丁中自带的"漏洞证明测试用例"（PoV Test）作为检索种子，通过 LLM 迭代式地搜索和扩展相关代码上下文，最终让 LLM 判断这个 API 调用在具体的应用中是否真的能被触发（比如应用是否做了足够的输入校验）。在 55 个真实 Java 开源项目上的测试结果是：精确率 83.8%、召回率 73.8%、F1 值 78.5%，大幅超过 Eclipse Steady（F1 35.9%）和 VAScanner（F1 38.6%）等传统 SCA 工具。但 SAVANT 跳过了传统静态调用图的构建，完全依赖 LLM 的语义理解来做判定，因此在处理 Java 动态分发（反射、依赖注入等）以及保证零漏报方面存在明显局限。

## B. 术语卡片

### 术语 1：PoV Test (Proof-of-Vulnerability Test)
- **大白话**：漏洞补丁里附带的一段测试代码，演示了"怎么操作就能触发这个漏洞"。
- **在本文中的作用**：SAVANT 把 PoV Test 作为检索种子——用它去匹配应用中语义相似的代码片段，从而找到潜在的漏洞调用点。
- **打个比方**：好比药品说明书里写了"过量服用会导致中毒"的具体剂量和条件——你拿着这个说明去检查自家药柜，看有没有人在按这个危险方式用药。

### 术语 2：反射式迭代上下文检索 (Reflection-Based Context Retrieval)
- **大白话**：先找到一个候选的漏洞调用点，然后反复问 LLM"当前上下文够不够判断？还缺什么？"——如果不够，让 LLM 自己推理"缺哪块代码"，然后自动去检索回来补上，循环直到上下文完整。
- **在本文中的作用**：这是 SAVANT 的核心方法创新——用 LLM 自省式的迭代检索来替代传统静态分析的数据流/控制流分析。
- **打个比方**：侦探查案——先找到线索（候选代码），分析后觉得不够，推理出"应该还需要那个房间的监控录像"，然后去调取，再看够不够——不够再推理还需要什么，直到能结案。

### 术语 3：语义引导的可达性分析 (Semantic-Guided Reachability Analysis)
- **大白话**：不是传统程序分析里那种"沿着函数调用图走，看能不能走到目标函数"的图论可达性，而是让 LLM 理解代码语义后判断"在当前的业务场景里，这个漏洞 API 的调用是否真的能造成危害"。
- **在本文中的作用**：SAVANT 用 LLM 替代了调用图分析——不是回答"调用路径是否连通"，而是回答"从语义上看，这个应用场景是否安全"。
- **打个比方**：传统可达性 = 看地图上有没有路（调用图是否连通）；语义引导的可达性 = 看路上的守卫（应用的安全校验）是否够强——即使路是通的，守卫够强也进不去。

## C. 核心知识点

### 知识点 1：两阶段架构——语义预处理 + 反射式 LLM 检测

SAVANT 分为两个阶段：第一阶段做 AST 增强的代码切块和向量嵌入（将源码按 AST 节点大小切分为语义完整的代码块，然后用 embedding 模型转成向量存储），第二阶段做反射式检测（先用 PoV Test 和 API 签名做向量相似度搜索找到候选块，再用 LLM 迭代扩展上下文，最后让 LLM 做漏洞判定）。这种设计的核心理念是不建完整调用图，用 LLM 语义理解代替结构化的可达性分析。

> **关键特征**：两阶段设计中，第一阶段与代码语言无关（AST 解析 + 嵌入），第二阶段的上限完全取决于 LLM 的语义理解能力。

### 知识点 2：在 55 个真实 Java 项目上的性能

在第三方基准数据集（Kang et al., ISSTA 2022, 25 个 CVE, 55 个项目: 42 个有漏洞, 13 个无）上：
- **精确率 83.8%**，召回率 73.8%，准确率 69.0%，F1 78.5%
- 6 个 false positive（保守倾向：宁可多报也不漏报）
- 对 42 个有漏洞项目正确识别了 31 个（73.8% 召回）

> 📊 **关键数字**：P 83.8% / R 73.8% / Acc 69.0% / F1 78.5%

### 知识点 3：大幅超越传统 SCA 基线

与两个代表性 SCA 工具对比（同一机器、同一数据集）：

| 工具 | 精确率 | 召回率 | 准确率 | F1 |
|:---|:---:|:---:|:---:|:---:|
| **SAVANT** | **83.8%** | **73.8%** | **69.1%** | **78.5%** |
| Eclipse Steady | 70.0% | 24.1% | 24.2% | 35.9% |
| VAScanner | 73% | 26.2% | 27.1% | 38.6% |

Eclipse Steady 和 VAScanner 分别有 17 个和 35 个项目因内存溢出或依赖解析失败而未能完成分析。SAVANT 成功分析了全部 55 个项目（不需要编译，只需要源码解析）。

> 📊 **关键数字**：SAVANT 比 Eclipse Steady F1 高出 119%（0.785 vs 0.359），比 VAScanner F1 高出 103%

### 知识点 4：LLM 消融实验——GPT-4o 最佳，开源仍差距明显

测试了 5 个闭源模型 + 1 个开源模型（Llama-3.1-405B），配合两种 embedding 模型（OpenAI text-embedding-3-small 和 Voyage-code-3），6 种代码块大小（500/1000/1500/2000/2500/3000 tokens）：

- **最佳配置**：GPT-4o + OpenAI embeddings + 2500 tokens → F1 0.87
- **最佳开源模型**：Llama-3.1-405B，最佳 F1 0.74（在 1500-2000 tokens 区间），但召回率仅 ~0.60
- **GPT-o1 表现意外偏低**：F1 0.72-0.83，低于 GPT-4o
- **Gemini 2.0 Flash**：稳定 F1 > 0.85，接近 GPT-4o
- **最优代码块大小**：2000-2500 tokens，超过后性能饱和或略降

> 📊 **关键数字**：GPT-4o 最佳 F1 0.87 / Llama-3.1-405B 最佳 F1 0.74 / OpenAI embeddings 比 Voyage 高 2-3% 精确率

### 知识点 5：方法创新——PoV 驱动 + LLM 替代调用图

SAVANT 的两个洞察是全文方法基础：(1) 补丁中的 PoV Test 精确描述了漏洞触发条件，是天然的知识源；(2) LLM 可以理解代码语义。两者的结合使 SAVANT 能绕过传统静态分析的两大瓶颈——调用图构建的完整性问题（Java 动态分发）和数据流分析的扩展性问题（大规模项目）。但"绕过"也意味着 SAVANT 完全不处理调用图断裂，把判断完全托付给 LLM 的语义理解能力。

> **关键特征**：SAVANT 是知识库中唯一直接组合「Java SCA + LLM + 可达性判定」的工作，但其"可达性"概念已偏离传统图论可达性，转为"语义级可触发判断"。

## D. 综述用途

### ① taxonomy 归类 + 双轴坐标

- **taxonomy 子方向**：② SCA 可达性与漏洞函数/版本定位
- **横轴（LLM 补静态分析哪个短板）**：版本与可达性鸿沟 + 模式不精确（SAVANT 将版本匹配的误报（库有 CVE → 项目有漏洞）推进到语义级判定（项目上下文是否让 CVE 可触发），同时用 LLM 识别应用的安全校验模式来判定误报）
- **纵轴（对健全性 soundness 的代价）**：只剪误报（可能引入漏报）— SAVANT 用 LLM 剪掉了因"应用有安全校验"而实际不可触发的告警（FP），但未提供零漏报保证，论文报告了 73.8% 的召回率（即有 26.2% 的漏报）
- **双轴坐标总结**：横跨「版本与可达性鸿沟 × 模式不精确」两格 × 纵轴「只剪误报（无零漏报保证）」

### ③ 在方法演进谱系的位置（与 VFArche/VERCATION 的关系）

```
VFArche（2025, 方向②） → VERCATION（2024, 方向②） → SAVANT（2025, 方向②）
  定位漏洞函数                  定位漏洞版本区间            判断漏洞在应用中是否可触发
  规则+LLM 分类器              切片+LLM+AST 回溯          语义预处理+LLM 反射式分析
  C/C++ 为主                   Java + C/C++               Java 专属

三者构成「函数定位 → 版本定位 → 可达性判定」的 SCA 精细化演进链条。
```

- **与 VFArche 的关系**：VFArche 回答"漏洞函数是哪个"，SAVANT 回答"这个漏洞在应用里是否可触发"——上下游互补。但 SAVANT 不依赖 VFArche 的输出（直接用 PoV Test 识别候选调用点），两者在方法上独立。
- **与 VERCATION 的关系**：VERCATION 做版本识别（vic→vfc 区间），SAVANT 做 API 使用语义分析——同属 SCA 精细化方向，但 VERCATION 保留静态调用图分析，SAVANT 完全抛弃了调用图。
- **关键区别**：VFArche 和 VERCATION 都保留传统程序分析能力（V0Finder 的函数哈希、静态切片等），只是用 LLM 增强特定子任务。SAVANT 则更激进——用 LLM 语义理解替代整个结构化的可达性分析管线。

### ④ 暴露的局限 / 对 gap 段的贡献

SAVANT 暴露了以下关键局限（直接对应综述的研究空白）：

1. **不处理 Java 动态分发**：SAVANT 依赖 AST 解析 + embedding 搜索来定位 API 调用，不构建调用图——因此完全无法处理反射、Spring DI、SPI、反序列化导致的调用图断裂。它的"可达性"是语义级的（代码逻辑上是否安全），而不是图论级的（调用路径是否连通）。

2. **无零漏报保障**：虽然聚合策略偏保守（任一 query 判有→判有漏洞），但没有 fail-safe 的零漏报约束设计，召回率仅 73.8%（即有 26.2% 的漏报）。

3. **高度依赖云端大模型**：最佳性能需 GPT-4o/Gemini 2.0 Flash，测试的"开源模型"Llama-3.1-405B（405B 参数）F1 仅 0.64-0.74，且 405B 依然是云端级别的模型规模。

4. **无动态执行验证**：所有评估基于第三方基准的人工标注真值，没有用动态执行来验证"漏洞确实可触发"或"确实不可触发"。

5. **无结构化的可审计证据链**：LLM 输出的是"分析理由"（文本形式的推理过程），不是结构化的可追溯证据（调用路径+判定依据+置信度）。

**对 gap 段的直接贡献**：SAVANT 的存在反而更加凸显了**议程 1** 的核心空白——"LLM 做 SCA 可达性判定"可行，但必须回到调用图框架下，且必须引入健全性约束。SAVANT 证明了 LLM 语义理解的潜力（78.5% F1），也暴露了纯 LLM 路线的天花板（26.2% FN，不处理动态分发，无法离线部署）。

## F. 双轴定位 + 议程映射（综述核心分析工具）

| 项 | 内容 |
|:---|:---|
| **横轴**（LLM 补静态分析哪个短板） | **版本与可达性鸿沟 + 模式不精确** — SAVANT 从版本匹配推进到语义级 API 使用分析，横跨两个区间：先判断 API 是否被调用（版本→调用），再用 LLM 分析语义判断应用的安全校验是否足以抵御漏洞（误报消减） |
| **纵轴**（对健全性 soundness 的代价） | **只剪误报（可能引入漏报）** — 裁剪了版本匹配的误报（应用有校验 → 不可触发），但无零漏报约束，论文报告 26.2% 的漏报率 |
| **部署就绪度** | **可编译性要求**：低（仅需源码做 AST 解析，不需要完整编译）／ **云端大模型依赖**：高（最佳性能需 GPT-4o/Gemini；开源 Llama-3.1-405B 仍有较大差距，且 405B 不适用于离线部署）／ **可解释性**：中（LLM 输出文本推理过程，非结构化）／ **真实代码库验证**：高（55 个真实 Java 开源项目，含 25 个 CVE） |
| **方法演进谱系位置** | VFArche（函数定位）→ VERCATION（版本定位）→ **SAVANT（语义级可达性判定）**。SAVANT 是三重中唯一直接用 LLM 替代调用图做可达性判定的工作，代价是放弃了调用图结构信息 |
| **对应研究议程** | **议程 1（带健全性保证的 SCA 可达性验证）** —— SAVANT 印证了"LLM 做 SCA 可达性判定"的可行性，是当前最接近议程 1 的现有工作。但它暴露了两个关键空白：(1) 完全放弃调用图导致无法处理 Java 动态分发（反射/Spring 等）造成的图断裂——议程 1 的解决路线是"静态调用图做主判 + LLM 辅助补判断边"，而非完全替代；(2) 无零漏报约束导致 26.2% FN——议程 1 要求 fail-safe 设计。 |

**一句话（双轴坐标 → 议程空白）**：本文落在【横轴=版本与可达性鸿沟 + 模式不精确 × 纵轴=只剪误报(可能引入漏报)】、部署就绪度【可编译低/云依赖高/可解释中/真实验证高】；它印证了**议程 1** 指出的"用 LLM 增强 SCA 可达性判定"的可行性（目前最接近议程 1 的工作），但暴露了该议程最核心的两个空白——(1) 放弃调用图就无法处理 Java 动态分发导致的断边，(2) 纯 LLM 语义判定无健全性保证，必须有 fail-safe 兜底。
