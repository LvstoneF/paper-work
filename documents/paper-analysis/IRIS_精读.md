# IRIS 精读笔记

> arXiv: 2405.17238 | ICLR 2025 | 方向 ④ LLM + 调用图/代码属性图 | ★ 必引必区分

---

## A. 大白话总结

**这篇论文解决的核心问题**：传统的静态污点分析工具（如 CodeQL）有两个致命弱点——第一，需要人工标注哪些 API 是"污点入口"（source）哪些是"危险终点"（sink），这既累又容易漏；第二，即使标对了，也会报一大堆误报，开发者得一个个去排查。IRIS 的思路是让大语言模型（LLM）来做这两件事：先用 LLM 自动推断 source/sink 规范（替代人工），再用 LLM 对 CodeQL 检测出的每条数据流路径做"上下文分析"——判断这条路径在实际代码中是否真的可利用（过滤误报）。在 120 个真实 Java 漏洞上测试，IRIS+GPT-4 检测出 55 个，比 CodeQL 的 27 个翻了一倍（+103.7%），同时误报率还降低了 5 个百分点。他们还用 IRIS 在 30 个最新版 Java 项目里发现了 4 个前人工具没发现的 0-day 漏洞。开源小模型（DeepSeekCoder 7B）也能检测出 52 个，只比 GPT-4 少 3 个，说明这套框架本身比模型大小更重要。

## B. 术语卡片

### 术语 1：Taint Specification（污点规范）
- **大白话**：告诉静态分析工具"哪些 API 是数据入口（source）、哪些是危险操作（sink）"的规则表。
- **在本文中的作用**：IRIS 的核心创新——让 LLM 替代人工来写这个规则表。传统做法需要安全工程师逐条翻阅 API 文档手动标注，IRIS 利用 LLM 的预训练知识自动完成。
- **打个比方**：机场安检需要知道"哪些物品是危险品"的清单。传统做法是安检员自己查手册编清单（累且容易漏），IRIS 相当于让一个见过全世界机场安检流程的老安检员来帮忙列清单。

### 术语 2：Contextual Analysis（上下文分析）
- **大白话**：LLM 把 CodeQL 检测出的疑似漏洞路径看一遍，结合周围的代码上下文判断"这到底是不是真漏洞"。
- **在本文中的作用**：IRIS 的第二道防线——自动过滤静态分析产生的误报。做法是把 CWE 描述、source/sink 附近的代码片段、中间的调用步骤一起喂给 LLM，让 LLM 给出"是/否漏洞"的判断及理由。
- **打个比方**：CodeQL 像一个过于谨慎的保安——只要有人靠近大门就报警（大量误报）。IRIS 的上下文分析相当于再派一个有经验的警官去看监控录像，判断这人到底是在正常进门还是在撬锁。

### 术语 3：CWE-Bench-Java
- **大白话**：IRIS 团队自己构建的 Java 漏洞测试集——120 个可编译的真实 Java 项目，每个含一个手动验证过的漏洞。
- **在本文中的作用**：现有数据集要么不支持编译（静态分析需要编译）、要么是 C/C++、要么没手动验证。CWE-Bench-Java 是第一个同时满足"真实项目、可编译、手动验证、含 CVE 元数据"四个条件的 Java 漏洞数据集。
- **打个比方**：以前测试厨师的菜谱测试集要么是"理论上可行的菜"（不可编译），要么是西餐菜谱（非 Java 语言），CWE-Bench-Java 是第一套"中餐菜谱且每道菜都实际做出来验证过的"。

### 术语 4：Neuro-Symbolic Approach（神经符号方法）
- **大白话**：把神经网络（LLM）的语义理解能力和传统符号分析（静态分析/CodeQL）的精确推理能力结合起来。
- **在本文中的作用**：IRIS 的总体设计哲学——让 LLM 做它擅长的事（理解 API 语义、判断代码是否可疑），让 CodeQL 做它擅长的事（在大型代码库里精确追踪数据流路径）。两者互补，而不是互相替代。
- **打个比方**：一个侦探组合——LLM 是经验丰富的老刑警（直觉好、见过各种犯罪模式），CodeQL 是痕迹鉴定科（技术精确、不会遗漏指纹）。老刑警说"这几个地方可能有问题"，痕迹科去精确追踪证据链。

## C. 核心知识点

### 知识点 1：IRIS 的两阶段 LLM 增强
IRIS 在传统静态污点分析流程中插入了两个 LLM 组件：
- **阶段 1 — 污点规范推断**：LLM 对项目中使用的外部 API 和内部函数参数进行分类，标注为 source 或 sink。外部 API 用 few-shot（3-4 示例）提示，内部 API 用 zero-shot 提示并附上 JavaDoc/README。
- **阶段 2 — 上下文分析过滤**：对 CodeQL 检测出的每条数据流路径，LLM 根据 CWE 描述和路径周围代码片段判断是否为真漏洞，输出 JSON 格式的判定结果（含解释+是否漏洞+哪个节点是误报）。

> **关键发现**：两个组件缺一不可。消融实验显示，单独 LLM source + CodeQL sink 只检测出 36 个漏洞，LLM sink + CodeQL source 只检测出 24 个，而两者都用能检测出 55 个（Table 3）。

### 知识点 2：性能数据（对比 CodeQL）
| 指标 | CodeQL | IRIS+GPT-4 | IRIS+DSC 7B |
|:---|:---:|:---:|:---:|
| 检测数 (/120) | 27 | **55** (+28, +103.7%) | 52 (+25) |
| 检测率 | 22.50% | **45.83%** (+23.33pp) | 43.33% |
| Avg FDR (误报率) | 90.03% | **84.82%** (-5.21pp) | 95.40% |
| Avg F1 | 0.076 | **0.177** (+0.101) | 0.062 |

> **注意**：报告中的 FDR 是上限（可能把 IRIS 新发现的真漏洞也算作误报）。手动抽查 50 个警报，27/50 确实暴露攻击面，估计真实 FDR 约 46%。

### 知识点 3：按 CWE 类别的检测差异
| CWE | #漏洞 | CodeQL | IRIS+GPT-4 | △ |
|:---|:---:|:---:|:---:|:---:|
| CWE-22 路径穿越 | 55 | 22 | **31** | +9 |
| CWE-78 命令注入 | 13 | 1 | **3** | +2 |
| CWE-79 跨站脚本 | 31 | 4 | **13** | +9 |
| CWE-94 代码注入 | 21 | 0 | **8** | +8 |

> **关键发现**：CWE-78（OS 命令注入）对所有方法都最难。原因：涉及 gadget chains 和文件写入等副作用，纯静态分析无法追踪。CWE-94 提升最显著（0→8），因为 CodeQL 完全缺乏相关规范。

### 知识点 4：LLM 推断规范的质量
- **Sink 召回率**（以 CodeQL 现有规范为基准）：GPT-4 最高 87.11%
- **Source 召回率**（以 CodeQL 现有规范为基准）：GPT-4 相对较低，但 CodeQL 本身的 source 规范是"过度近似"的（为了不漏报故意多标）
- **手动评估精度**（960 个随机样本）：GPT-4 超过 70%，其他 LLM 精度较低但 recall 较高（过度近似）
- **结论**：过度近似（低精度）在 IRIS 框架下反而有帮助——扩大规范覆盖范围，后续上下文分析可以再过滤误报

### 知识点 5：数据集 CWE-Bench-Java 的特点
- 120 个漏洞，均为手动验证过的真实 Java 项目
- 平均 SLOC（源码行数）：300K；最大：7M
- 覆盖 4 种 CWE：CWE-22(55)、CWE-78(13)、CWE-79(31)、CWE-94(21)
- 每个项目：可编译（Maven）、有 CVE/CWE 元数据、有 fix commit、有脚本自动拉取构建
- 从 1065 个 CVE 逐步筛选得到（手动验证是瓶颈：1065→430→265→149→120）

## D. 综述用途

### ① Taxonomy 归类 + 双轴坐标

**主方向**：④ LLM + 调用图/代码属性图
**次方向**：③ LLM 消减静态分析/SAST 误报（上下文分析组件）

**横轴位置**：**污点规范缺失（source-sink）+ 模式不精确（误报）**
- IRIS 用 LLM 填补的两个空白正好对应这两个领域：LLM 推断污点规范解决了"规范缺失"问题；LLM 上下文分析解决了"模式不精确导致的误报"问题。严格来说也触及"调用图不完整"——但 IRIS 不修复调用图，而是通过扩充规范让现有图能发现更多路径。

**纵轴位置**：**只剪误报（可能引入漏报）**
- IRIS 的上下文分析专门剪误报（平均 FDR 从 90.03% 降至 84.82%），但它**没有也不可能保证零漏报**。IRIS 明确承认 LLM 漏标 source/sink 会直接导致漏洞被漏检。作者在 Error Analysis（§C.7）中列出了两种 LLM 导致的漏报模式：(1) LLM 漏标 taint-propagator → 数据流边缺失；(2) LLM 漏标 source/sink → 分析无锚点。说明 IRIS 对健全性的代价是"可能引入漏报"。

### ② 可引硬数字

| 数据点 | 数值 | 来源上下文 |
|:---|:---|:---|

### ③ 在方法演进谱系的位置

**谱系位置**：在"自动推断污点规范"这个研究方向上，IRIS 是第一个系统性使用 LLM 的方案。

**与前人工作的关系**：
- **MERLIN (Livshits 2009) / Seldon (Chibotaru 2019)**：这两个也用自动方法推断污点规范，但靠的是概率建模和线性优化——需要分析第三方库的源代码，成本高且难以扩展。IRIS 用 LLM 替代了这种需要库源码的分析，只需要 API 名称和签名就能推断。
- **InspectJS (Dutta 2022)**：也做 CodeQL + ML 规范增强（针对 JavaScript），但依赖种子规范的质量，且需要对每个第三方库做昂贵分析。IRIS 不做库级分析，更 scalable。
- **Steenhoek 2024 / Khare 2023**：这些工作证明"LLM 直接做漏洞检测效果差"，IRIS 是"既然 LLM 直接做不行，那让它做它擅长的子任务（规范推断+上下文判断）"这一思路的典型代表。
- **与 LLM4SA / LLMDFA 的关系**：IRIS 是第一作者 Li Ziyang 和 Naik 团队的延续。和 LLMDFA（Wang 2024, NeurIPS）同期——LLMDFA 让 LLM 直接做数据流分析（替代静态分析），IRIS 让 LLM 做规范推断+误报过滤（增强静态分析）。

**对后续工作的影响**：IRIS 提出的"静态分析做主干 + LLM 做辅助任务"的神经符号范式，成为后续很多工作（如保守分析 2506.16899、LLM4FPM 等）的参考架构。

### ④ 暴露的局限 / 对 gap 段的贡献

**IRIS 暴露的空白（可直接用于本综述的 gap 论证）**：

1. **仅限 Java，且需完整编译**（→ 对应议程 4 跨语言）：
   - 需要 Maven 构建成功才能生成 CodeQL 数据库，编译失败的项目直接无法分析
   - Java 特有（第三方库 API 的 LLM 知识丰富度与语言流行度正相关）

2. **依赖云端大模型才能达到最佳效果**（→ 对应议程 2 离线/小模型）：
   - GPT-4 效果最好，小模型（L3 8B、G2 27B）的 FDR 高达 95.55% 和 91.23%，甚至比 CodeQL 还差
   - 虽然 DSC 7B 检测数接近 GPT-4，但 FDR 很高（95.40%），意味着误报几乎不可用

3. **无零漏报保证**（→ 对应议程 1/3 健全性约束）：
   - 明确承认 LLM "missing source or sink specifications" 会导致漏报（§C.7）
   - 没有任何机制保证 LLM 不会漏标关键 API
   - 没有对漏报率的系统分析（只报检测数，不报漏了多少、为什么漏）

4. **不处理 Java 动态分发**（→ 对应议程 4）：
   - CodeQL 的数据流图基于静态分析，对反射/Spring DI/SPI/反序列化造成的动态分发边束手无策
   - IRIS 没有尝试解决这个根本性的调用图断裂问题

5. **可解释性有限**（→ 对应议程 5 证据链）：
   - LLM 虽然输出文本解释，但这些解释不可验证（无法保证与代码实际行为一致）
   - 没有结构化的证据链（如精确的调用路径、污点传播步骤的源码标注）

6. **CWE-78（命令注入）是所有方法的瓶颈**：
   - 涉及 gadget chains 和副作用（文件写入），纯静态方法力不从心
   - IRIS 自己也承认"动态分析是未来方向"

## E. 5 维矩阵评分

| 维度 | 评分 | 理由 |
|:---|:---:|:---|
| D1 问题重要性 | ● | 静态分析手动规范标注是工业界公认的痛点（CodeQL 有 360+ 模块的手动规范仍在漏）。IRIS 针对的是实际部署中最关键的瓶颈——不是理论问题，而是工程痛点。 |
| D2 方法新颖性 | ● | 首次系统性用 LLM 替代人工做污点规范推断 + 上下文误报过滤的神经符号方案。虽然单个组件（LLM 分类、CodeQL）都是现成的，但组合方式和"LLM 做规范推断让 CodeQL 能发现更多路径"的设计思路在当时是领先的。 |
| D3 健全性保证 | ◐ | 有明确的误报消减分析和真实 FDR 手动评估，但没有零漏报约束。作者坦诚列出了 LLM 导致的漏报模式，但没做系统漏报率分析。缺失"引入多少漏报"的量化——这是本综述双轴分析要暴露的典型盲区。 |
| D4 实验充分性 | ● | 120 个真实 Java 项目、7 个 LLM（闭源+开源、7B→70B）、4 种 CWE 的全面对比。包含消融实验、超参数分析、规范质量评估（召回+精度）、手动验证、runtime 分析、新漏洞发现。CWE-Bench-Java 数据集本身就是一个贡献。 |
| D5 部署就绪度 | ◐ | 需要完整编译 Java 项目（编译失败则无法分析），依赖 CodeQL 作为静态分析后端，GPT-4 效果最好（云端大模型依赖）。小模型虽然检测数接近但 FDR 很高。部分项目分析需数小时（如 keycloak 3h）。虽然开源了代码和数据集，但离"离线一键扫描"还有距离。 |

**一句话**：IRIS 是"LLM 增强静态分析"范式的里程碑工作——实验极其扎实、代码数据全开源，但缺乏健全性约束、不处理动态分发、依赖云端大模型的特性，恰恰为本综述的研究议程提供了反证：**这些没解决的问题，正是我们要推的 agenda 的起点**。
