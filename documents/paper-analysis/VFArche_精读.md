# VFArche 精读笔记

> arXiv: 2506.18050 | 方向 ② SCA 可达性 | ★ 必引必区分 | 2025

---

## A. 大白话总结

这篇论文解决了一个很实际的问题：**漏洞数据库（NVD）只告诉你哪个组件有漏洞，但不告诉你漏洞具体在哪个函数里**。做 SCA 可达性分析时，你必须知道「漏洞函数是哪个」才能判断它是否被你的代码调用。传统做法是直接去补丁（patch）里找被修改的函数——但这篇论文发现 **超过 1/4 的漏洞函数根本不在补丁修改的函数范围内**。他们做了一个工具叫 VFArche，分三步：先用 CWE 知识库扩充 CVE 描述（解决描述太简略的问题），再根据有没有补丁分别用规则追踪和 AI 分类器筛选候选函数，最后用 GPT-4 对候选函数做两两 PK 排序。在 600 个 Java 漏洞上测试，有补丁时 96% 的漏洞函数能排进 top-10，没补丁时也能达到 84%。应用到真实 SCA 工具（Snyk 等）上，误报从 ~90% 降到了 ~10%。

## B. 术语卡片

### 术语 1：漏洞函数 (Vulnerable Function, VF)
- **大白话**：真正包含漏洞利用代码的那个函数。不是整个组件都有问题，而是组件里某个/某几个函数有问题。
- **在本文中的作用**：整篇论文的目标就是找到这个函数。没找到 VF 就无法做可达性分析。
- **打个比方**：一栋楼里有一个房间放了危险品，VF 就是那个房间的具体门牌号。SCA 工具只知道「这栋楼里有危险品」，但不知道是哪个房间——所以整栋楼的人都要疏散（全报为漏洞），造成大量误报。

### 术语 2：补丁 (Patch)
- **大白话**：开发者修复漏洞时提交的代码变更（git commit）。包含删了什么代码（-行）、加了什么代码（+行）。
- **在本文中的作用**：补丁是定位 VF 最重要的线索——修漏洞时肯定会动到相关代码。但论文核心发现是：被修的不等于有漏洞的（修 A 处的配置来堵住 B 处的漏洞）。
- **打个比方**：水管漏水，你换了水龙头（补丁修改的函数 = 水龙头），但真正的漏洞是你家的水管接头老化（VF = 水管接头）。只盯着水龙头看，就找不到真正的漏水原因。

### 术语 3：Swiss-system Tournament
- **大白话**：一种比赛排位制度——强对强、弱对弱，不需要每个人都和所有人比一场。用远少于全量比较的次数就能排出合理的名次。
- **在本文中的作用**：Phase 3 用 Swiss-system 把 100 个候选函数用 ~400 次 LLM 查询筛到 top-20（vs 全量两两比较需要 4,950 次），大幅降低 API 开销。
- **打个比方**：不是让 100 个人每人都和另外 99 人打一场，而是像国际象棋比赛——赢了的人继续和赢的人比，输的人和输的人比，8 轮就够排出前 20 名。

### 术语 4：知识迁移 (Knowledge Transfer)
- **大白话**：把有标签数据上学到的能力，迁移到没标签数据的场景。这里指：用「有补丁」场景下 Tracker 的输出训练分类器，然后让分类器在「没补丁」场景下也能筛选候选函数。
- **在本文中的作用**：PATCH-ABSENT 模式的核心——Traker 只能在有补丁时工作，但训练出来的 CodeBERT 分类器可以脱离补丁工作。
- **打个比方**：你有地图时学会了怎么找餐厅（Tracker），没地图时凭经验也大概知道「商业街附近可能有餐厅」（Classifier）。

## C. 核心知识点

### 知识点 1：VF 不在补丁中的四种模式（26.67%）
论文手动标注了 600 个 Java CVE，发现 160 个（26.67%）的 VF 不在补丁修改的函数中。四种情况：
- **方法替换**（17.83%）：老方法被换掉，老的那个才是 VF。例如 XStream 被换成 XStreamSupport。
- **类替换**（7.83%）：整个类被换成另一个。
- **追加参数**（13.33%）：方法调用参数变了，多态导致实际执行了不同方法。
- **配置变更**（61%）：修的是配置文件/字段，但引用这些配置的方法也是 VF——只是没在补丁里直接修改。

> 📊 **关键数字**：26.67% VF 不在补丁里 → 仅依赖补丁可达性分析会漏掉 1/4 的漏洞函数

### 知识点 2：CVE 描述的信息量极低
在 600 个 CVE 中：
- 仅 8.5% 提到了相关文件或函数名
- 仅 4% 明确指出了 VF
- 14% 包含的函数名是**误导性的**（提到了但实际不是 VF）

> 📊 **关键数字**：CVE 描述几乎不能直接用于 VF 定位——必须有额外的增强手段（这就是 Phase 1 的必要性）

### 知识点 3：三阶段方法的核心逻辑
- **Phase 1 - 描述增强**：用 CWE 知识库扩充 CVE 描述。CWE 描述更详细（会写漏洞根因），通过检索+加权把相关信息补进去。
- **Phase 2 - 候选筛选**：有补丁时用 Tracker（规则引擎，基于调用图+数据流从补丁出发追踪），没补丁时用训练好的 CodeBERT 分类器（知识迁移）。效果：把全仓库 ~15,000 个方法缩减到 ~100 个。
- **Phase 3 - 无监督排序**：先用 Swiss-system 把 100→20，再用 GPT-4 对 top-20 做全量两两比较。全程不需要标注 VF 数据（无监督）。

> 📊 **关键数字**：候选缩减 ~150 倍（14,910 → 99），Swiss-system 减少 ~92% LLM 查询（4,950 → 400）

### 知识点 4：实验结果
| 指标 | PATCH-PRESENT | PATCH-ABSENT |
|------|:---:|:---:|
| MRR | **0.78** | **0.58** |
| Recall@10 | **96.3%** | **84.0%** |
| ME@10 | 2.03 | 3.47 |

- PATCH-ABSENT 模式比最佳 baseline（VulF, MRR=0.31）高 **1.9 倍**
- 消融实验证明 Ph-3 Ranking 贡献最大：去掉后 MRR 从 0.58 暴跌到 0.10
- 真实场景验证：50 个最新 CVE，**43/50 成功定位** VF
- SCA 工具误报率：Snyk/OWASP DC/Dependabot 从 88-96% **降至 10-12%**

### 知识点 5：局限
- 依赖 GPT-4.1 在线 API（无法离线部署）
- Tracker 规则是 Java 专用的（其他语言需重写）
- VF ground truth 依赖人工标注（有主观性）
- 仓库映射（CVE ↔ 代码仓库）可能出错

## E. 5 维矩阵评分

| 维度 | 评分 | 白话理由 |
|:---|:---:|:---|
| D1 动态分发 | ○ | 处理 Java 但不涉及反射/Spring/SPI/反序列化的调用图断裂问题。Tracker 追踪的是标准调用图+数据流。 |
| D2 离线小模型 | ○ | Phase 3 依赖 GPT-4.1 在线 API，未验证离线部署可行性。Phase 2 的 CodeBERT 倒是可离线，但只在筛选阶段。 |
| D3 零漏报保守 | ○ | Tracker 的设计是「宁可多选不漏」（牺牲精确换召回），但整个框架不以保证零漏报为目标。 |
| D4 动态验证 | ○ | CodeBERT + GPT-4 纯静态，没有运行时插桩或 PoC 执行。 |
| D5 证据链 | ○ | 输出是排序后的函数列表，不是审计证据链（无调用路径、无判定理由、无置信度）。 |

**一句话**：VFArche 做的是「漏洞函数在哪」——这是可达性分析的前置步骤。我们的正文论文做的是「已知漏洞函数，判定它在当前系统中是否可达」——这是下游步骤。两者互补但问题域不同。
