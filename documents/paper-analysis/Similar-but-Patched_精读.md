# Similar-but-Patched (FVF) 精读笔记 (v2)

> arXiv:2412.20740 | Tan et al. | Zhejiang Univ. + Huawei | 2024

---

## A. 大白话总结（3-5句）

这篇论文系统性地研究了代码克隆检测中的一个核心难题：**Similar-but-Patched (SBP)** —— 某段代码看起来和已知漏洞几乎一样，但其实已经被修复了（函数体相似但不再有漏洞）。作者发现现有的四种克隆检测工具误报率高达 76.2%，主要就是分不清漏洞代码和它修复后的版本。他们的方案 FVF 核心思路很朴素：如果检测器报警说"这段代码和漏洞 A 很像"，就去查这个函数的 git 历史里是否包含漏洞 A 的修复操作（patch log）；如果修复操作存在，说明这段代码已经被补过了，是假警。FVF 在 4 个检测器 + 9 个 Linux/Redis 版本上验证零假阳，把整体误报率从 76.2% 降到 26.6%。他们还用 FVF 扫描了 1081 个真实项目，构建了 6827 个 SBP 函数的公开数据集，展示了 SBP 现象的普遍性（40% 的项目包含至少一个 SBP）。

## B. 术语卡片（3-5个）

- **SBP (Similar-but-Patched)**: 与被检测漏洞的代码高度相似但已被修复的代码片段，克隆检测的主要假警来源。
- **Patch Log / 修复日志**: 漏洞修复函数上所有代码变更（diff）按时间顺序拼接而成的序列。
- **Function Change Log / 函数变更日志**: 目标函数在版本控制历史中所有代码变更的序列。
- **Patch Reversion / 补丁回滚**: 漏洞修复 commit 后来被撤销（revert），导致代码恢复到与原始漏洞相同的样子，但实际上不再有漏洞。
- **BLEU-2**: FVF 用来做 patch log 匹配的 2-gram 相似度评分，对标识符重命名有一定鲁棒性。

## C. 核心知识点（3-5个，只留硬数字和结论）

1. **SBP 假警的严重程度**: 四种主流克隆检测工具（ReDeBug、VUDDY、MVP、Hash-based）在 Linux/Redis 上的整体误报率 FAR = 76.2%。
2. **FVF 消减效果**: 将整体 FAR 从 76.2% 降至 26.6%，改善率 49.6%；**零假阳**（即没有误把真漏洞当 SBP 过滤掉）。
3. **SBP 三类成因**: Patch Reversion（52.9%）、Minor Difference（36.1%）、Customized Patch（10.9%）。
4. **SBP 的普遍性**: 1081 个真实 OSS 项目中 40%（430 个）包含至少一个 SBP；31% 的漏洞有 SBP 变体。
5. **DL 检测器同样失败**: LineVul、VulBERTa、Devign、IVDetect 在 SBP 数据集上误报率均 >62%，Token-based 模型几乎把所有 SBP 判为漏洞。

## D. 综述用途

### ① 双轴坐标

- **横轴（补静态分析的短板）**: 模式不精确（误报） —— SBP 正是"模式匹配过于粗糙"导致误报的典型案例。
- **纵轴（健全性代价）**: 只剪误报（可能引入漏报） —— FVF 的保守策略（全部修复操作都匹配上才判为 SBP）保证了零假阳，但仍有 125 个漏网假警。

### ② 可引硬数字

- 克隆检测工具整体 FAR = 76.2%，FVF 降至 26.6%（IR = 49.6%）。
- FVF 零假阳，125 个漏网假警（分属"缺触发条件"72% + "琐碎克隆"28%）。
- SBP 数据集：6,827 SBP 函数 / 3,945 漏洞函数 / 2,834 CVE / 1,081 项目。
- DL 检测器在 SBP 上 FAR > 62%。

### ③ 方法演进谱系

```
克隆检测: ReDeBug(2012) → VUDDY(2017) → MVP(2020)
                                      ↓
FVF: 在克隆检测之上叠加"修复操作历史匹配"后处理层
      └── 核心创新: 从函数签名 → 函数变更历史
```

与"静态找候选 + LLM 判语义"范式不同，FVF 用的是纯版本控制历史匹配，不依赖 LLM。

### ④ Gap 贡献

- **暴露 gap**: SBP 现象被严重低估 —— 几乎所有漏洞检测和 SCA 工具都被此问题干扰而未系统应对。
- **对双轴框架的意义**: 提供了一种**零假阳**的误报消减策略参考（保守的变更历史匹配），但代价是漏掉 125 个需上下文分析才能发现的假警——直接体现"只剪误报的健全性代价"。
- **与综述的关联**: SBP 本质上也是 SCA 版本比对中的误报来源（"版本已修复但代码仍像漏洞"），与 Vercation 的版本纠错形成互补。
