# TOSSS 精读笔记

> **标题**: TOSSS: a CVE-based Software Security Benchmark for Large Language Models
> **作者**: Marc Damie, Murat Bilgehan Ertan, Domenico Essoussi, Angela Makhanu, Gaetan Peter, Roos Wensveen
> **年份**: 2026 (arXiv:2603.10969v2, Mar 2026)
> **方向**: 方向⑥ — 评测/基准
> **输出日期**: 2026-06-23

---

## A. 大白话总结

这篇论文做了一个**全新的安全基准**,思路和所有人都不一样——

不是让 LLM 生成代码然后检测是否安全,而是**给 LLM 两个实现相同功能的代码片段(一个有漏洞、一个安全),让 LLM 选哪个更安全**。

这么做的好处是:基准构建可以**自动化**。他们从 CVE 数据库挖掘漏洞修复前后的代码对,通过 MegaVul 流水线自动提取。这样新发现的漏洞可以随时加进来,不用依赖静态分析工具的规则更新。

14 个模型测下来,分数从 0.48 到 0.89 不等。最好的模型(GLM-5、GPT-5.4、Claude Opus 4.6)能稳定选对安全的代码,但专精代码生成的模型(如 Codestral、Qwen3 Coder)反而表现靠后。最差的是 Mistral Large 2512(0.48),和瞎猜差不多。

给提示加上"请选择最安全的实现"后,大部分模型有提升,但 Codestral 反而变差了——说明安全和代码生成能力可能是两回事。

## B. 术语卡片

| 术语 | 解释 |
|---|---|
| **TOSSS** | Two-Option Secure Snippet Selection — 二选一安全代码选择基准 |
| **Hintless Setting** | 不给任何安全提示,直接让模型选"你更喜欢哪个版本" |
| **w/Hint Setting** | 明确提示"请选择最安全的实现" |
| **MegaVul** | CVE 挖掘流水线和数据集,自动提取漏洞修复前后的函数级代码对 |
| **Code Selection Task** | 代码选择任务 — 不同于生成/补全,模型只需在 A/B 中选一个 |
| **可扩展性(Extensibility)** | 这是 TOSSS 的核心卖点:新漏洞通过 CVE 数据库自动加入,不需要人工干预 |

## C. 核心知识点（硬数字）

### 基准构成
- **任务类型**: 二选一代码选择(A/B 选安全版本)
- **数据来源**: MegaVul 数据集(CVE 挖掘)
- **测试规模**: 各模型 500 C/C++ + 500 Java = 1000 题
- **分数含义**: 1=全部选安全代码,0.5=随机猜,0=全部选漏洞代码

### 14 个模型 TOSSS 分数(C/C++ hintless 排序)

| 排名 | 模型 | C/C++ Hintless | Java Hintless | C/C++ w/Hint |
|---|---|---|---|---|
| 1 | **GLM-5** | **0.878** | 0.836 | 0.880 |
| 2 | **GPT-5.4** | 0.866 | 0.846 | 0.868 |
| 3 | **Claude Opus 4.6** | 0.864 | 0.834 | 0.884 |
| 4 | Kimi K2.5 | 0.858 | 0.814 | 0.890 |
| 5 | MiniMax M2.5 | 0.838 | 0.792 | 0.860 |
| 6 | Gemini 3 Flash | 0.836 | 0.802 | 0.866 |
| 7 | Claude Sonnet 4.6 | 0.808 | 0.782 | 0.840 |
| 8 | **Claude 3.5 Sonnet** | 0.752 | 0.738 | 0.798 |
| 9 | LLaMA 3 70B | 0.719 | 0.732 | 0.721 |
| 10 | Gemini 3.1 Flash Lite | 0.710 | 0.742 | 0.778 |
| 11 | **Codestral 2508** | 0.680 | 0.646 | 0.608 |
| 12 | DeepSeek V3.2 | 0.652 | 0.656 | 0.660 |
| 13 | **Qwen3 Coder Next** | 0.644 | 0.658 | 0.678 |
| 14 | **Mistral Large 2512** | **0.480** | 0.488 | 0.548 |
| | **平均** | **0.756** | 0.740 | **0.777** |

### 关键发现
- **模型间差距巨大**: 最优 0.878 vs 最差 0.480(接近随机)
- **Hint 增益有限**: 平均提升 +0.021(C/C++)和 +0.029(Java);GPT-5.4 只提升 +0.002
- **Codestral 反常**: 加 hint 反而下降 -0.072(C/C++)、-0.066(Java)
- **代码模型不占优**: Codestral(0.680)和 Qwen3 Coder(0.644)排在倒数 — "专精代码生成不代表安全能力强"
- **Anthropic 系列递增**: Claude 3.5 Sonnet(0.752) → Sonnet 4.6(0.808) → Opus 4.6(0.864),版本越新越强
- **C/C++ vs Java 接近**: 平均 0.756 vs 0.740,差距不显著

## D. 综述用途（双轴坐标 + 谱系 + gap）

### 双轴坐标
- **横轴(补静态分析短板)**: N/A — 这是 LLM 能力基准,不是检测方法
- **纵轴(健全性代价)**: N/A
- **部署就绪度**: 高(作为评测工具) — 基于 CVE,可扩展,不需要依赖特定模型,可以跑任意 LLM;低分结果说明 LLM 安全选码能力不足

### 关键硬数字(可引用)
- "TOSSS scores range from 0.48 to 0.89 across 14 models" — 量化 LLM 安全选码能力的差距
- "Coding-specialized models (Codestral 2508: 0.680, Qwen3 Coder Next: 0.644) rank among the bottom four" — 安全和代码生成能力脱钩
- "Adding explicit security hint improves scores by +0.021 on average, but Codestral decreases by -0.072" — 安全提示对不同模型效果截然不同
- "Score of 0.5 indicates random performance" — 部分模型(如 Mistral Large 2512: 0.480)和随机差不多

### 方法演进谱系位置
- 之前所有基准:CyberSecEval, CodeLMSec, SALLM, RealSec-bench 等 — 全部基于**代码生成+静态分析检测**
- TOSSS 定位: **首个基于代码选择的 LLM 安全基准**
- 最大区别: 不依赖静态分析工具的规则(可扩展性强),直接使用 CVE 数据库的 ground truth
- 与 RealSec-bench 互补: RealSec-bench 测"LLM 能不能生成安全的代码",TOSSS 测"LLM 能不能认出安全的代码"

### 暴露的局限(对 gap 段的贡献)
1. **代码选择≠代码生成**: 作者自己也承认,选对安全代码不代表能写出安全代码 — 但"选不对就一定写不对"是一个必要条件约束
2. **CVE 数据的局限性**: 只覆盖有修复 commit 的漏洞,不覆盖没有修复的 0-day
3. **函数级粒度**: 每个任务只给函数片段,不涉及仓库级或跨文件上下文
4. **仅 C/C++ 和 Java**: 和 MegaVul 的覆盖一致
5. **无过程间复杂度控制**: 不像 RealSec-bench 明确报告跨过程跳数,对复杂度的控制不如 RealSec-bench

### 对本综述的独特价值
- 提供"模型间安全能力差距"的量化数据:这对综述中"小模型 vs 大模型可行性分析"有直接参考意义
- 验证了"安全是独立能力维度":代码生成强≠安全强,支持 CLAUDE.md 议程 2(离线/小模型可行性)的"能力差距"论据
- 与 RealSec-bench 配合:一个测"选安全代码",一个测"写安全代码",结论一致 — LLM 安全能力不足
- "Hint 会导致模型变差"的发现与 RealSec-bench 中"安全指导降低 Claude 功能正确率"形成跨论文一致观测

## E. 5维矩阵

| 维度 | 评级 | 说明 |
|---|---|---|
| **方法新颖度** | ● | 思路独特——从代码生成改为代码选择,基于 CVE 自动构建,可扩展性极强,与所有现有基准不同 |
| **实验严谨性** | ◐ | 14 个模型,500+500 测试,规模可观;但缺少跨过程复杂度控制和人工验证环节(依赖 MegaVul 的 ground truth) |
| **对综述的贡献** | ● | 极高——为"安全是独立能力维度"提供关键数据;可扩展基准的思路可引用;14 模型分数表直接可用 |

| **部署就绪度** | ● | 作为评测工具本身很就绪——基于 CVE 自动化构建、可扩展、可跨模型;但评测结论(LLM 安全选码能力差)不乐观 |
