# 代码审计术语词典

> 给渗透测试背景的同学准备的速查手册。每个术语用大白话 + 日常比喻解释，不谈形式化定义。

---

## 一、程序分析基础

### 调用图 (Call Graph)
- **大白话**：函数之间谁调用了谁的关系图。A 函数里调用了 B，就有一条 A→B 的连线。
- **比喻**：公司组织架构图——CEO 调用总监，总监调用经理。你要找「谁向 CEO 汇报」就是顺着图往下走。
- **为什么重要**：做可达性分析的前提——图都没建全，后面的判断全是错的。

### 可达性 (Reachability)
- **大白话**：从程序入口点（比如 main 函数或 Web 接口）出发，沿着调用图走，能不能走到某个函数。能走到=可达，走不到=不可达。
- **比喻**：从你家大门出发，能不能走到某个房间。如果那个房间在另一栋楼（没有走廊连接），就是不可达——即使那个房间里有危险品，也威胁不到你。
- **为什么重要**：SCA 工具报的漏洞 90%+ 都是不可达的（组件有 CVE 但你的代码根本调不到它），这就是误报的主要来源。

### 数据流 (Data Flow)
- **大白话**：数据在代码中怎么传递、被谁修改、最终到了哪里。
- **比喻**：快递包裹从发件人 → 中转站 → 收件人的流转路径。每个经手人都可能在包裹上做手脚。
- **为什么重要**：污点分析（taint analysis）的基础——追踪用户输入从哪进来（source）到哪出去（sink）。

### 控制流 (Control Flow)
- **大白话**：代码执行的分支（if/else）和循环（for/while），决定程序走哪条路。
- **比喻**：地铁线路图——if/else 就是分岔路口，循环就是环线。
- **与数据流的区别**：数据流关心「快递里装了什么、谁碰过」，控制流关心「快递车走了哪条路」。

### 调用图断裂 (Call Graph Breakage)
- **大白话**：静态分析工具画调用图时，某些调用关系画不出来，导致图上有缺口。
- **比喻**：建筑图纸上标注"此处通向外墙"，但到底通到哪不知道——因为那个接口是施工时现场决定的。
- **为什么重要**：Java 的动态分发（反射、Spring DI、SPI、反序列化）就是调用图断裂的四大元凶。这是我们整篇论文要解决的核心问题。

## 二、安全测试工具类型

### SAST (Static Application Security Testing)
- **大白话**：不运行程序，只读源码/编译产物（字节码、IR）来找漏洞的工具。
- **比喻**：安检看 X 光片——不动手、不拆包，只扫描。
- **代表工具**：CodeQL、Semgrep、SonarQube、Fortify
- **优缺点**：覆盖全（不依赖运行环境），但误报多（很多在静态分析里看起来危险的路径，运行时根本走不到）。

### DAST (Dynamic Application Security Testing)
- **大白话**：运行程序，从外部发送攻击载荷，看程序有没有反应。
- **比喻**：红队——直接上去打，看能不能打进去。
- **代表工具**：Burp Suite、OWASP ZAP、sqlmap
- **优缺点**：误报少（真打进去了才是真的），但覆盖不全（只能测暴露在外的接口）。

### SCA (Software Composition Analysis)
- **大白话**：检查你项目里用的第三方开源组件（jar/npm/pip 包），看它们有没有已知漏洞（CVE）。
- **比喻**：安检——查你包里有没有已知的危险品（对照违禁品清单）。
- **代表工具**：Snyk、OWASP Dependency-Check、Dependabot、Black Duck
- **为什么误报多**：SCA 只管「你包里有个危险品」，不管「你实际会不会用它」。90%+ 的告警是——组件确实有 CVE，但你的代码根本调不到有问题的那段。

### IAST (Interactive Application Security Testing)
- **大白话**：在程序运行时插桩（埋监控点），实际执行时顺便检测漏洞。介于 SAST 和 DAST 之间。
- **比喻**：安检时让你边走路边扫描，看实际走了哪条路、碰了什么东西。
- **为什么相关**：IAST 的动态执行信息可以用来验证「静态分析说可达的路径，运行时真的走了吗」。

## 三、漏洞与标准

### CVE (Common Vulnerabilities and Exposures)
- **大白话**：每个公开漏洞的唯一编号，全球通用。
- **比喻**：确诊病例号。CVE-2017-5638 = 「患者#2017-5638，确诊为 Struts2 远程代码执行」。
- **编号格式**：CVE-YYYY-NNNNN（年份-序号）

### CWE (Common Weakness Enumeration)
- **大白话**：漏洞类型分类编号。CVE 是「你得了什么病」，CWE 是「这病属于哪一类」。
- **比喻**：疾病分类编码。CWE-502（反序列化漏洞）= 呼吸道感染。CWE-89（SQL 注入）= 消化系统疾病。
- **为什么重要**：VFArche 用 CWE 描述来扩充 CVE 描述——因为 CWE 描述写得更详细。

### CVSS (Common Vulnerability Scoring System)
- **大白话**：漏洞危险程度打分，0-10 分。
- **比喻**：病情危重程度评分。

### 误报 (False Positive, FP)
- **大白话**：工具报了「这里有漏洞」，实际上没有。
- **比喻**：健康人被误诊为癌症。
- **原因**：静态分析保守——看到"可能危险"的路径就报，不管实际能不能走到。

### 漏报 (False Negative, FN)
- **大白话**：真有漏洞但工具没报出来。
- **比喻**：真有病但体检没查出来。
- **为什么比误报更危险**：误报浪费人工，漏报直接导致安全事故。

### 零漏报保守策略 (Fail-Safe / Conservative)
- **大白话**：宁愿多报几个假的，也绝不漏掉一个真的。
- **比喻**：机场安检宁可让十个良民多等五分钟，也绝不放一个危险品过去。
- **为什么重要**：等保审计场景下，漏报的代价远大于误报。

## 四、Java 动态分发的四大元凶

### 动态分发 (Dynamic Dispatch) 概述
- **大白话**：Java 程序里，代码写的是 `obj.doWork()`，但编译时不知道 obj 到底是哪个类的实例——运行时才能确定。所以静态分析工具画调用图时，画不出 `doWork()` 到底通向哪里。
- **比喻**：你发了个快递到「XX 小区物业代收」，快递单上写的是物业地址，但最终是哪个业主拿到——只有派送时才知道。
- **四大类型**：反射、Spring IoC/DI、SPI、反序列化

### 反射 (Reflection)
- **大白话**：代码运行时，根据字符串（比如类名"com.example.MyClass"）动态创建对象、调用方法。不通过正常的 `new` 或 `import`。
- **比喻**：不用钥匙开门（import），而是知道门牌号（类名字符串）就能遥控操作。
- **代码例子**：`Class.forName("com.example.Evil").getMethod("exec").invoke(null)` ——编译时根本不知道 Evil 类存在。
- **安全影响**：很多漏洞利用（反序列化 RCE、命令注入）依赖反射绕过静态检测。

### IoC / DI (Spring 依赖注入)
- **大白话**：对象不由你手动 `new` 出来，而是 Spring 框架在运行时根据配置文件或注解自动创建和注入。你写个 `@Autowired`，Spring 就帮你把对象塞进去。
- **比喻**：去餐厅不用自己买菜、切菜——服务员（Spring）根据菜单（注解/配置文件）把菜端到你面前。你只说了「我要一盘宫保鸡丁」，根本没见厨师——就像代码里只写了接口类型，不知道具体哪个实现类。
- **安全影响**：静态分析只能看到接口，看不到 Spring 实际注入的实现类——调用图在这里必然断裂。

### SPI (Service Provider Interface)
- **大白话**：Java 内置的插件机制——你在 `META-INF/services` 目录下写个配置，告诉 JVM「遇到这个接口时，加载这些实现类」。JDBC 驱动加载就是靠 SPI。
- **比喻**：酒店前台（JVM）有一本服务目录（META-INF/services），客人说「帮我叫个车」，前台就翻目录找到合作出租车公司的电话——客人根本不知道是哪家公司的车。
- **安全影响**：类似反射——实现类被延迟绑定，静态分析无法预判。

### 反序列化 (Deserialization)
- **大白话**：把网络传来的字节流/字符串还原成内存中的 Java 对象。Java 原生的 `ObjectInputStream.readObject()` 就是反序列化入口。
- **比喻**：拆快递——包装（字节流）拆开后，里面的东西（对象）就出现在你家里了。但问题是你不知道包裹里是不是炸弹——攻击者可以构造恶意序列化数据。
- **为什么与调用图相关**：反序列化可能触发任意类的 `readObject()` / `finalize()` 等方法——这些类在编译时的调用图中根本不存在。
- **著名案例**：2015 年 Apache Commons Collections 反序列化 RCE，几乎影响了所有 Java 企业系统。

## 五、程序分析进阶

### CodeQL
- **大白话**：GitHub 出的工具，把代码编译成数据库，然后你可以用类似 SQL 的查询语句找漏洞模式。
- **比喻**：把整栋楼的图纸全部数字化——然后能查询"找出所有朝南且没有防火门的窗户"。
- **与 Semgrep 的区别**：CodeQL 需要编译（能建调用图），Semgrep 只需源码（只能做模式匹配，建不了调用图）。

### CPG (Code Property Graph)
- **大白话**：把代码的三种图——语法树（AST，代码结构）、控制流图（CFG，执行路径）、数据流图（DFG，数据流转）——融合成一张大图。这样分析工具可以同时利用三种信息。
- **比喻**：把一栋楼的电路图 + 水管图 + 结构图叠在一起——你既能看"这个插座在哪面墙上（结构）"，又能看"它连到哪个电闸（控制流）"，还能看"打开后电流怎么走（数据流）"。

### 污点分析 (Taint Analysis)
- **大白话**：追踪「不可信数据」从入口（source）到危险出口（sink）的完整路径。
- **比喻**：疾控中心追踪传染病——source=第一个感染者（用户输入），sink=被传染的人（执行危险操作的危险函数），taint path=传播链。如果传播链上的每个人都有口罩（过滤/校验），那就安全。
- **source 例子**：`request.getParameter("userInput")`——用户通过 HTTP 参数传入的数据
- **sink 例子**：`Runtime.exec(input)`——直接执行用户输入，这就是命令注入
- **为什么难**：如果中间有反射或动态代理，传播链就断了——这是 SemTaint 那篇论文要解决的问题。

### 程序切片 (Program Slicing)
- **大白话**：从一个语句出发，找出所有「可能影响这个语句」或「被这个语句影响」的代码行。
- **比喻**：从一个漏水点出发，找出所有可能通向这里的管道，以及从这里流出去的所有管道。
- **Backward slice**：往前追溯——这个变量是哪里来的？（从 sink 回溯到 source）
- **Forward slice**：往后追踪——这个变量会影响哪些语句？（从 source 追踪到 sink）

### 指针分析 / Points-to Analysis
- **大白话**：在 Java 里就是分析「这个变量到底指向哪个对象」。因为 Java 里一个接口类型变量可能指向多种实现。
- **比喻**：「给张三打电话」——但张三可能用三个手机号，在不同的时间打会通到不同的手机上。指针分析就是尽量缩小「可能是哪个号」的范围。
- **为什么难**：反射和动态代理让 pointer analysis 几乎不可能精确。

## 六、AI/LLM 相关

### In-Context Learning (ICL)
- **大白话**：不给模型更新参数，而是在提问时附上几个例子，让模型照着例子回答。比起「请总结这篇论文」（zero-shot），「这是几个总结的例子，请按同样格式总结这篇论文」（few-shot / ICL）效果好得多。
- **比喻**：不是重新培训员工（微调），而是给他看几个模板样例，让他按模板干活。

### RAG (Retrieval-Augmented Generation)
- **大白话**：让 LLM 回答问题前，先从外部知识库检索相关资料，把检索结果附在 prompt 里。这样 LLM 就有了「超出训练数据」的知识。
- **比喻**：考试时允许你翻书查资料再作答，而不是仅凭记忆。

### Agent
- **大白话**：让 LLM 像人一样——观察环境 → 做决策 → 执行工具调用 → 观察结果 → 再做决策，循环直到完成任务。不只是一问一答。
- **比喻**：不是给客服发"怎么退货"然后等他回复（一问一答），而是给他一个完整的权限——"帮我退货"，他自己查订单、生成退货单、联系物流、确认退款。
- **安全审计场景的例子**：CVE-Genie 让 Agent 读 CVE 描述 → 自动找漏洞代码 → 生成 PoC → 运行验证。

## 七、评测指标

### 精确率 (Precision)
- **大白话**：工具报了 100 个漏洞，其中 90 个是真的 → Precision = 90%。
- **关心的是**：报出来的东西有多靠谱？

### 召回率 (Recall)
- **大白话**：实际有 100 个漏洞，工具找出了 80 个 → Recall = 80%。
- **关心的是**：漏掉了多少？

### F1 Score
- **大白话**：Precision 和 Recall 的调和平均数——两者都重要时用这个一个数字衡量。
- **为什么不用平均**：如果 P=100%, R=0%（只报了一个，是真的，但漏了 99 个），普通平均=50%——看起来还行但实际很差。F1 会更真实地靠近 0%。

### MRR (Mean Reciprocal Rank)
- **大白话**：第一个正确答案排在第几位。排得越靠前越好。
- **公式直觉**：排第 1 得 1 分，排第 2 得 0.5 分，排第 10 得 0.1 分——排名越靠后分数越低。
- **比喻**：谷歌搜索结果——你真正想点的链接排在第几位。

### Manual Efforts@K (ME@K)
- **大白话**：人工需要检查多少候选项才能找到正确答案。ME@10=3 意味着平均翻 3 个候选项就找到了。
- **比喻**：安检抽查——平均要查几个箱子才找到违禁品。数字越小越好。

## 八、其他常见缩写

| 缩写 | 全称 | 大白话 |
|------|------|--------|
| OSS | Open-Source Software | 开源软件 |
| NVD | National Vulnerability Database | 美国国家漏洞数据库（全球最权威的 CVE 来源） |
| PRF | Pseudo Relevance Feedback | 用第一轮搜索结果来扩充搜索词——搜一次之后拿 top 结果的关键词补到搜索词里再搜 |
| BM25 | Best Match 25 | 一种比 TF-IDF 更先进的文本相关性算法，信息检索标配 |
| AST | Abstract Syntax Tree | 代码的树状结构表示——编译器/解析器把代码拆成语法单元后的结果 |
| IR | Intermediate Representation | 中间表示——编译过程中的半成品，比源码结构化、比机器码可读 |
| CG | Call Graph | 调用图——就是 Call Graph |
| PDG | Program Dependence Graph | 程序依赖图——数据依赖+控制依赖的图 |
| diff / patch | — | Git commit 中的代码变更——加了什么行（+）、删了什么行（-） |
| hunk | — | diff 中的一段连续变更，同一个文件可能有多个 hunk |
