# 综述论文收集：4 人并行任务（v2，可直接发给队员）

> **发给谁**：同学 A / B / C / D，每人领一个卡片往下做。
> **做什么**：每人搜 5–8 篇高质量论文，4 人合计把文献池从 29 篇补到 ~50 篇。
> **怎么交**：按下面第五节表格格式填，核实一条交一条。
> **铁律**：没有可解析 arXiv ID 或 DOI 的论文不收。核实不了的放「待核实区」。宁可少报，绝不编造。

---

## 一、项目背景（让队员知道在干什么）

我们团队在写一篇综述，主题是：

> **LLM + 开源组件/SCA 安全、代码审计、可达性验证、误报消减**

时间范围 **2022–2026**，主体是 2024–2026。

目前已经有 29 篇已核实论文（见附录），分布在 6 个子方向。你的任务是**找还未在清单里的新论文**，把文献池补到 ~50 篇。

## 四、真实性铁律（违反的直接打回）

1. **每篇必须有 arXiv ID 或 DOI**——没有就不收，不用纠结
2. **必须亲自打开核实**：arxiv.org/abs/XXXX.XXXXX 或 doi.org/XX.XXX/XXXX 必须能打开、显示的就是这篇论文
3. **摘要用自己的话写**（2–3 句即可），禁止复制搜索引擎 snippet
4. **拿不准的放「待核实区」**，标注原因（如「arXiv 上只有摘要没有全文」「作者名对不上」）
5. **宁可少报，绝不杜撰**——综述的信誉比篇数重要

| 会议/期刊 | 如适用 | ICSE / USENIX / NeurIPS / arXiv |
| 所属方向 | ①②③④⑤⑥ | ② |

| 核实状态 | 已核实 / 待核实 | 已核实 |
| 核实链接 | 你点开过的 URL | https://arxiv.org/abs/2506.18050 |
| 备注 | 特殊说明 | 与 Vercation 方法可对比写一段 |

## 六、搜索策略（按优先级）

**渠道优先级**（从高到低）：

1. **arXiv** → cs.SE、cs.CR、cs.AI 分类，关键词搜索，按时间倒序
2. **dblp** → 按作者/关键词追踪，适合补会议论文
3. **Google Scholar** → 从已核实论文的「被引」和「引用」反向找
4. **Semantic Scholar** → 推荐相似论文
5. **顶会 proceedings** → ICSE / FSE / ASE / ISSTA / USENIX Security / NDSS / CCS / S&P 的 2024–2026 程序册

**通用搜索式**（把 `KEYWORD` 换成你方向的关键词）：

```
(LLM OR "large language model") AND (KEYWORD)
("false positive" OR "false alarm") AND (LLM OR "large language model")
("call graph" OR CPG OR "code property graph") AND (LLM OR "large language model")
```

**必做动作**：从附录排除清单中挑 2–3 篇与你方向最相关的，去 Google Scholar 看谁引用了它、它引用了谁——这是最高效的找法。

## 七、收录原则

**优先收录：**
- 有清晰方法创新 + 有实验数据
- 与 Java/JVM 生态相关（或方法可迁移到 Java）
- 涉及静态分析 + LLM 结合
- 来自顶会/顶刊（ICSE / FSE / ASE / ISSTA / USENIX / NDSS / CCS / S&P / TSE / TOSEM）
- 2024–2026 年

**排除（除非特别相关）：**
- 纯 C/C++ 且方法明确无法迁移到 Java
- 只做漏洞检测、不涉及可达性/误报分析
- 无实证评估的 position paper / short paper（<5 页）
- 非英文论文

---

## 八、协作规则

1. **开工前**：每人建自己的 sheet（在共享文档里），A/B/C/D 各一个 tab
2. **搜到疑似别人方向的**：瞄一眼别人的 tab，已收录就跳过
3. **每填一篇就更新状态**：其他人实时可见，避免撞车
4. **Day 4 晚**：D 做交叉检查——扫一遍 A/B/C 的 tab，看有没有遗漏
5. **Day 5**：全员自查所有链接能打开，状态改为「已核实」
6. **Day 6**：集中过一遍，合并去重
