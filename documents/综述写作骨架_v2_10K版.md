## §4 双轴分类体系 ★ 核心原创框架(≈ 900 字)

### 4.1 现有分类法的局限(1 短段,≈ 100 字)

### 4.2 横轴:LLM 补偿静态分析的 5 个短板(表 1 + 1 段释义,≈ 250 字)
| 坐标 | 短板 | 代表论文 |
|:---:|---|---|
| **A** | 调用图不完整(可达性) | SemTaint, IRIS, Boosting |
| **B** | 模式不精确(误报) | 保守分析, Sifting, LLM4PFA |
| **C** | 污点规范缺失 | IRIS, AdaTaint |
| **D** | 版本-可达性鸿沟 | VFArche, VERCATION, SAVANT |

**释义要点**:横轴不互斥(IRIS 同覆 B+C);一篇论文可在多坐标贡献。

### 4.3 纵轴:健全性代价的 4 个层级 ★ 本综述首创(表 2 + 1 段释义,≈ 350 字)
| 坐标 | 层级 | 代表论文 | 出现频率 |

## §7 研究议程 ★ 核心交付物(≈ 1500 字)

### 7.0 议程的星型耦合(图 4 + 1 段释义,≈ 150 字)

### 7.1 议程 1:带健全性保证的 SCA 可达性验证(核心,≈ 400 字)
- **问题**:无人将"调用图可达性判定"和"零漏报约束"组合——这是 SCA 误报消减从版本维度推进到可达性维度的关键
- **反向引用**:§6.3 (A,4) 空白;§6.4① D1●+D3● 从未同篇
- **技术路线**:静态调用图主判 + LLM 仅在动态分发断边选择性补判 + 保守 fail-safe;命名为**"健全性脚手架"哲学**(静态调用图作 soundness 地板,LLM 是检修工不是大楼)
- **关键挑战**:Java 反射/Spring/SPI/反序列化断边;LLM 判定不稳定需保守兜底
- **预期影响**:把 SCA 误报消减从"版本维度"推进到"可达性维度",首次引入健全性约束

### 7.2 议程 2:离线/小模型可行性(部署条件,≈ 200 字)
- **问题**:43/48 篇精读依赖云端大模型,等保/气隙不可部署
- **存在性证明**:Bridging <4B Java 93.57% + SecureFixAgent ≤8B Python + Boosting Qwen3-14B C 三点汇聚 → **"≤ 14B 在特定任务上足够好"**
- **路线**:多任务能力基准 + 上下文压缩(CPG 切片/SARIF 轨迹)+ "模型规模-任务精度-部署约束"权衡曲线
- **影响**:为等保/气隙场景给出模型选型指南

### 7.3 议程 3:统一健全性评测框架(测度前提,≈ 200 字)
- **问题**:46/48 篇精读仅报 P/R/F1,几乎无人报 FN——双轴分析暴露的最大盲区
- **路线**:(i) 标准化指标必含 FN 率 + 按断图机制分层;(ii) 建立含动态执行真值的 Java 基准(现有几乎全 C/C++);(iii) 强制"消减率 + 引入率"同报
- **影响**:让该领域实验结果可比较、安全性可审计

### 7.4 议程 4:跨语言/跨框架动态分发建模(推广路径,≈ 200 字)
- **问题**:反射/DI/SPI 是 Java 特有,但"运行时动态分发让静态分析断裂"是跨语言共性(Python monkey-patching、JS 动态 require、Go interface)
- **路线**:抽象"动态分发边"统一表示 + LLM 跨语言迁移(SemTaint JS → Java)
- **影响**:把议程 1 从 Java 推广到多语言

### 7.5 议程 5:可解释证据链与合规对接(合规出口,≈ 200 字)

- **路线**:结构化证据链(完整调用路径 + 每个断边的判定依据 + 可验证性标签);研究证据链与动态执行结果的一致率

- **影响**:打通"LLM 辅助安全分析 → 等保合规交付物"的最后一公里
